CAP Theorem &
Consistency Models

The trade-off that governs every database in your stack.

The Precise Definition

CAP theorem (Brewer, 2000) states that a distributed data store can guarantee at most two of these three properties simultaneously:

PropertyWhat it meansFailure scenario
Consistency Every read receives the most recent write or an error — never stale data Node A and Node B always agree on the current value
Availability Every request receives a (non-error) response — system stays up even if some nodes are down No timeouts, no errors returned to the client
Partition Tolerance The system continues operating even if network messages between nodes are dropped or delayed Nodes are split — they can't talk to each other — but system keeps running
Critical nuance Network partitions always happen in real distributed systems — hardware fails, packets drop. So P is not optional. The real choice is: when a partition occurs, do you sacrifice C (serve potentially stale data) or A (refuse to respond)? Every distributed database makes this choice.

Plain English — The applyGOAT Scenario

Imagine applyGOAT runs on 3 servers — India, Europe, US — all storing user data. Now the cable between India and Europe is cut. They can't talk. That's a network partition.

The three properties, no jargon

Consistency — "Give me the truth or give me nothing."
A user updates their resume on the India server. One second later they query Europe. Consistency means Europe either returns the updated resume — or returns an error. It never silently returns the old one.

Availability — "Always respond, no matter what."
Even while India and Europe are disconnected, the Europe server keeps responding to every request. No errors, no timeouts. But it might serve yesterday's data — it doesn't know it's stale.

Partition Tolerance — "Keep working even when servers can't talk."
This one is not optional. Network cables get cut in real life. If you can't tolerate partitions, your entire system collapses the moment one connection fails.

The real choice Since P is mandatory, the actual decision is: when India and Europe are disconnected and a user queries Europe — what do you do?
DecisionWhat you sacrificeWhat you getCalled
Refuse to respond until India confirms the data Availability — users get an error Consistency — data is always accurate CP
Respond with whatever Europe has, even if stale Consistency — data may be old Availability — users always get a response AP

PostgreSQL (applyGOAT user accounts): "I'd rather return an error than lie to you." → CP
Pinecone (resume vector search): "I'll search what I have, even if it's 2 seconds behind." → AP

Your Stack, Mapped to CAP

This is where most engineers stay vague in interviews. Know your own systems precisely.

SystemCAP choiceWhyYour project
PostgreSQL CP Uses WAL and synchronous replication. Under partition, replicas refuse reads to avoid returning stale data. You get consistency or an error — never stale. applyGOAT user accounts, multi-tenant data
MongoDB Tunable (default AP) Default write concern w:1 is AP — writes ack'd before replication. Set w:"majority" + readConcern: "majority" for CP behaviour. Most teams forget to do this. Document storage in your projects
Pinecone AP Eventual consistency — upserted vectors become queryable within seconds, not instantly. Designed for high availability of vector search at the cost of immediate consistency. RAG retrieval in AI Sales Agent, applyGOAT
Redis (if used) AP (default) Master-replica async replication. If master dies, you may lose recent writes. Sentinel/Cluster modes improve this but don't give full CP. Caching layer (if applicable)
Cross-project check — AI Sales Agent The AI Sales Agent (LangGraph + RAG, insurance) makes the same CAP call twice, for different reasons. Conversation and session state persisted to PostgreSQL is CP — a dropped or duplicated agent turn is worse than a slow one; the graph must know exactly which node it's on. The RAG retrieval step against the vector store is AP — if the index lags a few seconds behind the latest document ingest, the agent still answers, it just cites slightly older material. Same theorem, same two systems doing the same two jobs as in applyGOAT, different domain — this is the pattern to name explicitly in an interview: "my transactional state is CP, my retrieval layer is AP, and that split is deliberate."

Consistency Is Not Binary

CAP's "C" is all-or-nothing, but real systems have a spectrum of consistency models. Know these — interviewers probe here.

ModelGuaranteeExample
Linearisability
(strong consistency)		 Operations appear instantaneous and in real-time order. Every reader sees the latest write immediately after it completes. PostgreSQL with synchronous commit
Sequential consistency All operations appear in some consistent order (same for all nodes), but not necessarily real-time order. Some distributed lock implementations
Causal consistency Operations that are causally related are seen in order. Unrelated ops can be seen in any order. MongoDB with sessions enabled
Eventual consistency If no new updates are made, all replicas will eventually converge to the same value. No timing guarantee. Pinecone, DynamoDB default, DNS
applyGOAT application Your multi-tenant SaaS has a real trade-off here. User auth and billing data (PostgreSQL → linearisable) needs strong consistency — a user must never see a stale subscription state. Resume embeddings in Pinecone (eventual) are fine being slightly stale; a 2-second delay before a new resume chunk is searchable is acceptable.

Beyond CAP: PACELC

CAP only describes behaviour during a partition. PACELC (Daniel Abadi, 2012) extends it to describe normal operation too:

  • During a Partition → choose Availability or Consistency
  • Else (normal operation) → choose Latency or Consistency

PostgreSQL is PC/EC — consistent during partitions, and consistent (low latency sacrificed for consistency) in normal operation. Pinecone is PA/EL — available during partitions, low latency in normal operation (consistency sacrificed).

Dropping PACELC in an interview when the interviewer only expected CAP = instant seniority signal.

Interview Playbook

Click any question to reveal a model answer and follow-up response.

// LIKELY INTERVIEW QUESTIONS — click to reveal answers
Model answer
"CAP says a distributed data store can guarantee at most two of: Consistency (every read returns the latest write or an error), Availability (every request gets a non-error response), and Partition Tolerance (the system works even if nodes can't communicate). Since network partitions are unavoidable in real deployments — cables fail, packets drop — the practical trade-off is always between C and A when a failure happens. CP systems like PostgreSQL return an error rather than stale data. AP systems like Pinecone stay available but may serve slightly old results. The right choice depends on business tolerance — financial data needs CP, search and recommendations can tolerate AP."
Follow-up answer
"Never — not in a real distributed system. Sacrificing partition tolerance means your system stops functioning the moment any two nodes can't communicate — a hardware failure, a network hiccup, anything. The only scenario where you 'don't need' P is a single-node system with no replication, but then it's not a distributed system. CAP's real decision has always been CP vs AP, not CA vs CP vs AP."
Model answer
"For payments I'd choose CP — PostgreSQL with synchronous replication. The business cannot tolerate inconsistency: two nodes disagreeing on a balance could allow double-spending or overdrafts. I'd accept that during a partition, payment writes fail gracefully rather than risk inconsistent state. In the UX this means a clear 'payment temporarily unavailable' message — far better than charging someone twice. For read-heavy non-critical paths like transaction history display, I'd use a read replica with slightly relaxed consistency since historical records don't need real-time accuracy."
Follow-up answer
"PostgreSQL uses linearisability — writes don't succeed until confirmed by the replica, and reads always reflect the latest committed write. If two nodes disagree on a balance, the primary refuses writes rather than risk divergence. Beyond CAP, I'd also use SELECT FOR UPDATE or advisory locks for balance updates — CAP handles inter-node consistency, but you still need application-level locking to serialise concurrent requests hitting the same node."
Model answer
"Not by default — MongoDB is AP out of the box. With default write concern w:1, the primary acknowledges a write before replication completes. If the primary fails immediately after, that write is lost on the replica. Reads with readConcern: 'local' can return data not yet replicated to majority. So in default configuration, MongoDB sacrifices consistency for availability. It can behave as CP — but you have to explicitly configure it."
Follow-up answer
"You need w: 'majority' — the write only succeeds when a majority of replica set members confirm it — combined with readConcern: 'majority' so reads only return data acknowledged by the majority. The latency cost: every write now waits for a network round-trip to replica confirmation. Same datacenter: ~1–5ms extra. Cross-region: 50–200ms. You also lose some write throughput since the primary must coordinate synchronously. This is why most teams run MongoDB in AP mode and reach for PostgreSQL when they genuinely need CP guarantees."
Gap to watch for Many candidates say "I'd choose CP for everything important" without explaining the cost. Strong answer: "strong consistency requires synchronous replication, which adds write latency and means writes fail during a partition rather than accepting stale state — the right choice depends on what the business can tolerate."
