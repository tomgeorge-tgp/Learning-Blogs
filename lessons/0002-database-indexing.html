The Problem Without Indexes

applyGOAT has 20,000 users. Each user may have dozens of job applications. That's potentially millions of rows in your applications table.

Without an index, a query like:

SELECT * FROM applications WHERE user_id = 'abc-123';

…forces PostgreSQL to read every row in the table, comparing each one. That's a full table scan — O(n). At 1M rows, that's 1M comparisons per query.

The rule A full table scan is fine for small tables. At production scale — 100K+ rows — unindexed queries on filtered columns will kill your database.

B-Tree Index — The Default

PostgreSQL and MongoDB both default to B-Tree (Balanced Tree) indexes. This is what CREATE INDEX gives you unless you specify otherwise.

Start with the real table

Say applyGOAT's applications table has these 9 rows. The id column is an auto-increment integer — the primary key, which PostgreSQL automatically indexes with a B-tree.

id (PK)user_idjob_titlestatusscore
3u-101Backend Engapplied78
7u-204ML Engineerinterviewing91
12u-101DevOps Engapplied65
19u-307Frontend Engrejected55
25u-204Backend Engoffered95
31u-101SREapplied72
44u-502Backend Engapplied80
58u-307Data Enginterviewing88
63u-502ML Engineerapplied70

The rows are stored in the heap (the actual table file) in insertion order — not sorted by id. The B-tree index is a separate data structure that keeps the id values sorted and points back to the heap row location (called a heap pointer / ctid).

What the B-tree index looks like on disk

PostgreSQL organises the B-tree into pages (fixed-size blocks, default 8KB). Each page is a node in the tree. Internal nodes hold keys + child pointers. Leaf nodes hold keys + heap pointers to the actual table rows.

CREATE INDEX idx_applications_id ON applications (id);

-- The resulting B-tree (simplified to show structure):

                     ┌─────────────────┐
          ROOT NODE  │    [25]          │   ← one key, two children
                     └────────┬────────┘
                    ┌─────────┴──────────┐
                    ▼                    ▼
           ┌─────────────┐     ┌──────────────┐
 INTERNAL  │  [12, 19]   │     │   [44, 58]   │
  NODE     └──┬───┬───┬──┘     └──┬────┬────┬─┘
              │   │   │           │    │    │
     ┌────────┘ ┌─┘ ┌─┘     ┌───┘  ┌──┘  ┌─┘
     ▼          ▼   ▼       ▼      ▼     ▼
  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
  │ 3    │ │ 12   │ │ 19   │ │ 25   │ │ 44   │ │ 58   │
  │ 7    │ │      │ │      │ │ 31   │ │      │ │ 63   │
  │      │ │      │ │      │ │      │ │      │ │      │
  │→heap │ │→heap │ │→heap │ │→heap │ │→heap │ │→heap │
  └──────┘ └──────┘ └──────┘ └──────┘ └──────┘ └──────┘
  LEAF      LEAF     LEAF     LEAF     LEAF     LEAF

  Leaf nodes are linked: [3,7] → [12] → [19] → [25,31] → [44] → [58,63]
                         ←─────────────────────────────────────────────→
                                  sorted order, linked list

Three things to notice:

  • Root → Internal → Leaf: Every lookup starts at the root and descends. The tree has fixed depth — always the same number of hops regardless of which key you look up.
  • Internal nodes are routing guides: They don't point to heap rows. They tell you which child branch contains your key.
  • Leaf nodes are linked: All leaf nodes form a sorted linked list left-to-right. This is what makes range scans fast.

Step-by-step lookup: WHERE id = 31

You run: SELECT * FROM applications WHERE id = 31;

Step 1 — Start at ROOT [25]
  Is 31 > 25?  YES → go right child

Step 2 — Arrive at INTERNAL NODE [44, 58]
  Is 31 < 44?  YES → go left child (the branch before 44)

Step 3 — Arrive at LEAF NODE [25, 31]
  Scan the leaf: find key 31
  Read heap pointer: (page 4, slot 2)  ← ctid

Step 4 — Fetch from heap
  Go directly to page 4, slot 2 in the table file
  Read the full row: {id:31, user_id:'u-101', job_title:'SRE', ...}

Total: 3 node reads + 1 heap fetch = 4 I/Os
Without index: scan all 9 rows = 9 comparisons (and at 1M rows, 1M comparisons)
Why O(log n)? Each node visit eliminates half (or more) of the remaining candidates by following the correct branch. At 9 rows the tree has 3 levels. At 1 million rows it has ~20 levels. At 1 billion rows it has ~30. The number of levels grows logarithmically with row count — that's O(log n).

What the heap pointer (ctid) actually is

Every leaf node entry stores a ctid — a physical location in the heap file, in the format (page_number, slot_number). After the B-tree lookup finds the matching key, PostgreSQL uses the ctid to jump directly to that physical page. No scanning — one direct I/O.

-- You can see ctids in PostgreSQL
SELECT ctid, id, user_id, status
FROM   applications
ORDER BY id;

-- Output:
--  ctid   | id | user_id | status
-- --------+----+---------+--------------
-- (0,1)   |  3 | u-101   | applied
-- (0,2)   |  7 | u-204   | interviewing
-- (1,1)   | 12 | u-101   | applied
-- (1,2)   | 19 | u-307   | rejected
-- (2,1)   | 25 | u-204   | offered
-- ...
-- ctid (1,1) means: page 1, slot 1 of the table's heap file

Range scan: WHERE id BETWEEN 12 AND 44

This is where the leaf node linked list matters.

Step 1 — B-tree lookup: find the leaf containing id = 12
  Root [25] → Internal [12, 19] → Leaf [12]   (3 node reads)

Step 2 — Scan forward through linked leaf nodes
  Leaf [12]  → read id=12  ✓ (in range)
  Leaf [19]  → read id=19  ✓
  Leaf [25, 31] → read id=25 ✓, id=31 ✓
  Leaf [44]  → read id=44  ✓
  Leaf [58, 63] → id=58 > 44 → STOP

Step 3 — Fetch the 5 matching rows from the heap via their ctids

Total: 3 node reads to find start + 5 sequential leaf reads + 5 heap fetches
Compare to full table scan: read all 9 rows regardless

At production scale with millions of rows and a date range covering 10,000 rows, the difference is: ~20 tree reads to find the start + 10,000 leaf reads + 10,000 heap fetches vs. scanning millions of rows. Orders of magnitude faster.

What happens when a new row is inserted

Insert id = 22. PostgreSQL must update the B-tree to keep it sorted and balanced.

Before insert:
  Leaf [19] — has space → INSERT id=22 here

After insert:
  Leaf [19, 22] — sorted within the leaf

If the leaf is FULL (no space left):
  PostgreSQL SPLITS the leaf into two:
    Leaf A: [19]
    Leaf B: [22]
  And promotes the split key (22) up to the parent internal node.
  If the parent is also full, it splits too — cascading up to the root.
  If the root splits, a new root is created and the tree gains one level.

This is the write overhead of indexes:
  Every INSERT touches the B-tree (find the right leaf + insert + possibly split)
  Every UPDATE on the indexed column: mark old entry dead + insert new entry
  Every DELETE: mark the leaf entry as dead (vacuumed later)
Write overhead is real If you have 8 indexes on applications, every INSERT must update all 8 B-trees. Every UPDATE of an indexed column marks the old entry dead and inserts a new one in all affected B-trees. This is why bulk import operations often drop indexes first, import the data, then rebuild the indexes — it's dramatically faster than maintaining them row by row.

Why B-Tree supports range queries

Because leaf nodes are in sorted order and linked, the engine finds the start of the range via the tree (O(log n)) then scans forward through leaf nodes — never going back up the tree. This is why you can index on created_at and efficiently run:

SELECT * FROM applications
WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31';
Note — UUID or string indexes: same tree, different ordering

The B-tree mechanism is identical — sort the values, build the tree, navigate root → internal → leaf. The only difference is what "sorted" means.

Strings use lexicographic (dictionary) order — byte-by-byte, left to right. So a B-tree on job_title sorts like this:

'Backend Eng'   → comes first  (B < D < F < M < S)
'DevOps Eng'
'Frontend Eng'
'ML Engineer'
'SRE'

The tree looks and navigates exactly the same as the integer example — just with string comparisons instead of number comparisons at each node.

UUIDs work the same way — sorted lexicographically on their string representation. A B-tree on user_id (UUID) would sort:

'2f8a1c3d-...'   ← leaf 1
'5e3b9f21-...'   ← leaf 2
'8c7d4a0e-...'   ← leaf 3
'a1f2e3b4-...'   ← leaf 4

The problem with UUID v4 (random UUIDs) — and this is the one real difference that matters:

With integers (id = 1, 2, 3, 4...), new rows always have the largest id. Every insert goes into the rightmost leaf — the one that's already in memory and being written to. Efficient, sequential writes.

With UUID v4, each new id is random — it lands in a random position in the sorted tree. The target leaf is almost certainly not in the buffer pool, requiring a disk read. Then the insert may split that leaf, requiring a disk write to two pages. At high write volume this causes index fragmentation: constant random I/O and page splits across the whole tree.

UUID v4 (random) inserts:
  Insert 'f7c3...' → goes into the middle of the tree, page split
  Insert '2a9b...' → goes near the start, different split
  Insert 'c4e1...' → somewhere else entirely
  → Every insert is a random read + write across all index pages

UUID v7 (time-ordered) inserts:
  Insert '018f3a1b-...' (Jan 1) → rightmost leaf
  Insert '018f3b4c-...' (Jan 2) → rightmost leaf (same or next)
  Insert '018f3c77-...' (Jan 3) → rightmost leaf
  → Behaves like an auto-increment integer — sequential, fast

In applyGOAT: if you use PostgreSQL's gen_random_uuid() (UUID v4) as a primary key, your primary key index will fragment under write load. Switch to UUID v7 (uuid_generate_v7() via the pg_uuidv7 extension, or gen_random_uuid() with a manual time prefix) or use a BIGSERIAL for high-write tables. UUID v7 gives you the global uniqueness of a UUID and the sequential write performance of an integer.

Hash Index — Fast Equality, Nothing Else

A hash index applies a hash function to the indexed value and stores the result in a hash table. Lookup is O(1) — constant time, regardless of table size.

Index typeLookupRange querySortBest for
B-Tree O(log n) ✓ ✓ Most queries — equality, range, sort
Hash O(1) ✗ ✗ Exact equality only (e.g. session token lookup)
Why hash can't range query Hashing destroys ordering. hash('apple') and hash('apricot') have no relationship — they could be anywhere in the hash table. There's no "start scanning from here" concept.

Composite Indexes — Column Order Matters

A composite index covers multiple columns. In applyGOAT's multi-tenant setup, you almost certainly need these.

CREATE INDEX idx_apps_tenant_status_date
ON applications(tenant_id, status, created_at DESC);

The left-prefix rule

This index can be used for queries that filter on columns from left to right, without skipping:

Query filterUses index?Why
WHERE tenant_id = ? ✓ Yes Uses first column — left prefix
WHERE tenant_id = ? AND status = ? ✓ Yes Uses first two columns in order
WHERE tenant_id = ? AND status = ? ORDER BY created_at DESC ✓ Yes (fully covered) All three columns used — no sort needed
WHERE status = ? ✗ No Skips first column — index can't be used
WHERE tenant_id = ? AND created_at > ? ~ Partial Uses tenant_id to find the right tenant's rows, but then must scan all of them to filter by created_at — because status sits between them in the index and scrambles the date order

Why WHERE tenant_id = ? AND created_at > ? is only partial — step by step

Think of this index like a physical filing cabinet:

  • Drawer = tenant_id
  • Folders inside that drawer = status (applied, interviewing, offered, rejected)
  • Papers inside each folder = rows sorted by created_at newest-first

This is what the index leaf nodes actually contain for goat-001:

Position │ tenant_id  │ status        │ created_at
─────────┼────────────┼───────────────┼────────────
1        │ goat-001   │ applied       │ 2024-06-15  ← row A
2        │ goat-001   │ applied       │ 2024-05-10  ← row B
3        │ goat-001   │ applied       │ 2024-03-01  ← row C
4        │ goat-001   │ interviewing  │ 2024-06-20  ← row D
5        │ goat-001   │ interviewing  │ 2024-04-05  ← row E
6        │ goat-001   │ offered       │ 2024-06-18  ← row F
7        │ goat-001   │ offered       │ 2024-01-12  ← row G
8        │ goat-001   │ rejected      │ 2024-05-30  ← row H
9        │ goat-001   │ rejected      │ 2024-02-20  ← row I

Now you run:

WHERE tenant_id = 'goat-001' AND created_at > '2024-05-01'

Step 1 — tenant_id filter ✓ works perfectly.
PostgreSQL jumps straight to position 1, knows all goat-001 rows end at position 9. Fast, no scan needed.

Step 2 — now find created_at > 2024-05-01 within those 9 rows.
Look at just the dates column:

Position │ created_at   │ After May 1?
─────────┼──────────────┼──────────────
1        │ 2024-06-15   │  ✓ YES
2        │ 2024-05-10   │  ✓ YES
3        │ 2024-03-01   │  ✗ NO
4        │ 2024-06-20   │  ✓ YES
5        │ 2024-04-05   │  ✗ NO
6        │ 2024-06-18   │  ✓ YES
7        │ 2024-01-12   │  ✗ NO
8        │ 2024-05-30   │  ✓ YES
9        │ 2024-02-20   │  ✗ NO

The dates go: YES → YES → NO → YES → NO → YES → NO → YES → NO. They jump up and down with no pattern.

Why? Because the dates are only sorted inside each status folder, not across all folders:

applied:      Jun → May → Mar   (sorted within this folder only)
interviewing: Jun → Apr         (resets — sorted within this folder only)
offered:      Jun → Jan         (resets again)
rejected:     May → Feb         (resets again)

The rows that match created_at > '2024-05-01' (rows A, B, D, F, H) are scattered across four different folders — not sitting together in one place. There is no single point in the index where PostgreSQL can say "everything from here onwards is after May 1."

So PostgreSQL is forced to:

  1. Use the index to jump to all goat-001 rows — 9 rows
  2. Check created_at > '2024-05-01' on each of those 9 rows one by one

That is what "partial" means. The index did half the work (filtered by tenant), then had to resort to a row-by-row check for the date.

What the index would need to look like for the date filter to be fully fast

If the index was (tenant_id, created_at DESC) — no status in the middle — the same 9 rows would be stored like this:

Position │ tenant_id  │ created_at
─────────┼────────────┼────────────
1        │ goat-001   │ 2024-06-20   ← row D
2        │ goat-001   │ 2024-06-18   ← row F
3        │ goat-001   │ 2024-06-15   ← row A
4        │ goat-001   │ 2024-05-30   ← row H
5        │ goat-001   │ 2024-05-10   ← row B
         │            │ ── 2024-05-01 cutoff ──────────── STOP HERE
6        │ goat-001   │ 2024-04-05   ← row E
7        │ goat-001   │ 2024-03-01   ← row C
8        │ goat-001   │ 2024-02-20   ← row I
9        │ goat-001   │ 2024-01-12   ← row G

Now dates are in one clean sorted sequence with nothing interrupting them. PostgreSQL jumps to the 2024-05-01 cutoff and takes everything above it — rows 1 to 5. Zero row-by-row checking. Fully fast.

The fix — and the trade-off Create a second index with created_at directly after tenant_id:

CREATE INDEX idx_apps_tenant_date ON applications(tenant_id, created_at DESC);

Now WHERE tenant_id = ? AND created_at > ? is fully fast. But this new index no longer helps queries that filter by status. For two different query patterns you often need two separate indexes. That is a normal, expected part of index design — you match each index to the query it serves.
applyGOAT real example Your multi-tenant queries always filter by tenant_id first. Put it as the first column in every composite index. This isolates tenant data and the query planner can use the index for all downstream filters.

EXPLAIN ANALYZE — before and after the composite index

Here's what the plan shape actually looks like on a 2M-row applications table (applyGOAT scale) for:

EXPLAIN ANALYZE
SELECT * FROM applications
WHERE tenant_id = 'goat-001' AND status = 'interviewing'
ORDER BY created_at DESC
LIMIT 20;

Before — no index on (tenant_id, status):

  ->  Sort  (cost=45123.08..45748.30 rows=250120 width=812) (actual time=812.439..812.443 rows=20 loops=1)
        Sort Key: created_at DESC
        Sort Method: top-N heapsort  Memory: 41kB
        ->  Seq Scan on applications  (cost=0.00..38210.00 rows=250120 width=812) (actual time=0.031..780.112 rows=250311 loops=1)
              Filter: ((tenant_id = 'goat-001'::text) AND (status = 'interviewing'::text))
              Rows Removed by Filter: 1749689
Planning Time: 0.412 ms
Execution Time: 812.501 ms

After — CREATE INDEX idx_apps_tenant_status_date ON applications(tenant_id, status, created_at DESC);

Limit  (cost=0.56..8.91 rows=20 width=812) (actual time=0.045..0.198 rows=20 loops=1)
  ->  Index Scan using idx_apps_tenant_status_date on applications  (cost=0.56..2104.33 rows=5030 width=812) (actual time=0.043..0.191 rows=20 loops=1)

Two things changed in the plan shape: the Seq Scan (cost starting at 0.00, scanning all 2M rows and discarding 1.75M of them) became an Index Scan that goes straight to the ~5,030 matching rows. And the separate Sort node disappeared entirely — because the index already stores rows in created_at DESC order within each (tenant_id, status) group, PostgreSQL doesn't need to sort after fetching. That's a ~3,600× drop in execution time from one index.

Cross-project check — applyGOAT This is not a hypothetical query — it's the shape of applyGOAT's recruiter dashboard: "show the 20 most recent applications for this tenant in status X." Every tenant shares the same applications table, so without tenant_id leading the index, one tenant's slow unindexed query burns CPU that every other tenant is also waiting on — the classic multi-tenant noisy-neighbor problem. The composite index doesn't just make this query fast, it isolates tenants from each other's query cost.

When Indexes Make Things Worse

Indexes are not free. Every index on a table has a cost:

  • Write overhead — every INSERT, UPDATE, DELETE must update all indexes on that table. 10 indexes = 10x index maintenance per write.
  • Storage — indexes can be as large as the table itself.
  • Index bloat — UPDATEs don't modify indexes in place; old entries are marked dead and space is reclaimed lazily (VACUUM in PostgreSQL). Heavy update workloads bloat indexes over time.
  • Query planner confusion — too many indexes and the planner may choose a suboptimal one, or spend time evaluating options.
Rule of thumb Index columns you filter, join, or sort on in frequent queries. Don't index every column. And always check with EXPLAIN ANALYZE — the query planner often surprises you.

HNSW — How Pinecone Indexes Vectors

Pinecone doesn't use B-Trees. Vectors (embeddings from your resume/job description) have hundreds of dimensions. B-Trees don't work in high-dimensional space. Instead, Pinecone uses HNSW — Hierarchical Navigable Small World.

The core idea

HNSW builds a multi-layer graph of vectors:

  • Top layers: sparse, long-range connections — a "highway" for fast coarse navigation
  • Bottom layer: dense, short-range connections — precise local search

To find the nearest neighbours to a query vector, the algorithm enters at the top (sparse) layer, greedily moves toward the closest known node, then descends to lower layers to refine the search.

Layer 2 (sparse):  A ———————— F ———————— K
                              |
Layer 1:           B — C — D — E — F — G — H
                                   |
Layer 0 (dense):  ... all vectors with local edges ...

Why approximate, not exact?

Exact nearest-neighbor search in 1536 dimensions (OpenAI embedding size) requires comparing the query to every vector — O(n). HNSW trades a small accuracy loss (~1-5%) for O(log n) approximate search. For RAG retrieval, "95% accurate nearest neighbor" is indistinguishable from "exact" in practice — the model handles semantic fuzziness anyway.

applyGOAT / AI Sales Agent When your RAG pipeline queries Pinecone for relevant resume chunks or insurance policy documents, HNSW is what makes that retrieval fast even at hundreds of thousands of vectors. The index is built at upsert time (slow) and queried at runtime (fast). This is the latency-vs-build-time trade-off baked into every vector database.

Model answer
"A B-Tree index stores values in sorted order in a balanced tree. Lookups are O(log n) and it supports equality, range queries, and sorting — it's the default for a reason. A hash index hashes the value and stores it in a hash table — O(1) lookups for exact equality, but it can't do range queries or sorting. In practice I default to B-Tree. The only time I'd consider a hash index is for exact-match lookups on high-cardinality columns — session tokens, API keys — where I know I'll never need a range query."
Follow-up answer
"Hashing destroys ordering. hash('2024-01-01') and hash('2024-01-02') produce unrelated values — they could be anywhere in the hash table. There's no concept of 'scan from this point forward.' B-Trees maintain sorted order across leaf nodes, so a range query just finds the start point and walks forward — that's impossible with a hash structure."
Model answer
"I'd create a composite index on (tenant_id, status, created_at DESC). This fully covers the query — the planner can use the index to filter by tenant and status, then return rows already sorted by date with no separate sort step. It also benefits from the left-prefix rule: queries filtering on just tenant_id or tenant_id + status can also use this index. In a multi-tenant app like applyGOAT, tenant_id must always be the first column — it isolates each tenant's data in the index and makes all downstream filters within that tenant fast."
Follow-up answer
"Low cardinality on status (only 3 values like 'open', 'closed', 'expired') makes it less selective as a filter — the index will still help because it narrows by tenant first. But if most queries only care about status = 'open', you could create a partial index: CREATE INDEX ... ON applications(tenant_id, created_at DESC) WHERE status = 'open'. This index is smaller (only open rows), faster to build, and more cache-friendly. The trade-off: it only helps queries that filter on that exact status."
Model answer
"Every index is a separate data structure that must be kept in sync with the table. On every INSERT, UPDATE, or DELETE, PostgreSQL must update every index on that table — not just the row. Ten indexes means ten index maintenance operations per write. Beyond that, UPDATE on an indexed column doesn't modify in place — it marks the old index entry as dead and creates a new one. Under heavy write load this causes index bloat, which further slows writes and scans. The fix is to audit which indexes are actually used in production and drop the ones that aren't pulling their weight."
Follow-up answer
"In PostgreSQL, pg_stat_user_indexes shows how many times each index has been scanned. An index with idx_scan = 0 over a meaningful time window is almost certainly useless. I'd also run EXPLAIN ANALYZE on the critical queries to see which index the planner chooses and whether it matches what I expected. For indexes the planner ignores, I'd check if column statistics are stale (ANALYZE the table) or if the query shape doesn't match the index's left-prefix."
Model answer
"Pinecone uses HNSW — Hierarchical Navigable Small World. B-Trees don't work for high-dimensional vectors because there's no meaningful 'sorted order' across hundreds of dimensions — the curse of dimensionality makes tree-based indexing break down. HNSW builds a multi-layer graph: upper layers have sparse, long-range connections for fast coarse navigation, the bottom layer has dense local connections for precision. A query enters at the top, greedily navigates to the closest node, and descends layers to refine. Search is O(log n) — a massive improvement over the O(n) brute-force scan you'd need without an index."
Follow-up answer
"Approximate means HNSW might miss the mathematically closest vector by a tiny margin — typically 1–5% recall loss depending on the ef parameter (search breadth). For RAG this is almost never a problem. The retrieval is feeding context to an LLM that can reason over semantically similar but not identical text. If the top-3 retrieved chunks are 'very similar' rather than 'exactly the nearest 3', the generation quality doesn't meaningfully degrade. Where you'd care about recall is high-stakes deduplication or fraud detection — not document retrieval for generation."
Gap most candidates have Candidates know indexes exist and make things faster. Senior candidates know why — the tree structure, sorted leaf nodes, left-prefix rule, write overhead, and when to use partial indexes. That's the depth interviewers are probing for.
