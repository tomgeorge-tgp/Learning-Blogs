& Query Anatomy

The shape of every SQL query, and why the order you write it is not the order it runs.

The Six Clauses of a SELECT

Every SQL query is built from these six clauses. Most engineers know them by rote. The ones who perform in interviews know exactly what each one does, when the database evaluates it, and what it has access to at that moment.

SELECT   job_status, COUNT(*) AS total, AVG(score) AS avg_score
FROM     applications
JOIN     users ON applications.user_id = users.id
WHERE    tenant_id = 'goat-001'  AND created_at > '2024-01-01'
GROUP BY job_status
HAVING   COUNT(*) > 10
ORDER BY avg_score DESC
LIMIT    5;

The Execution Order — Most Interviewers Are Testing This

You write SQL in a human-readable order but the database executes it in a completely different order. This is not a trivia question — it has concrete consequences for what aliases and filters are valid where.

1
FROM
Identifies the base table(s). If there's a JOIN, it also happens here — the engine builds the initial working set of rows.
Columns and rows first exist here
2
WHERE
Filters the rows from step 1. Operates row-by-row before any grouping occurs.
Cannot use SELECT aliases — they don't exist yet
3
GROUP BY
Collapses matching rows into groups. After this point, you work with groups, not individual rows.
Non-aggregated SELECT columns must appear here
4
HAVING
Filters groups (not rows). Like WHERE but operates after grouping — can use aggregate functions.
WHERE filters rows; HAVING filters groups
5
SELECT
Computes the output columns. Aliases are defined here for the first time.
Aliases available from this step onward
6
ORDER BY
Sorts the output. The one place you can reference SELECT aliases by name.
Can use SELECT aliases (they now exist)
7
LIMIT / OFFSET
Takes a slice of the sorted output. Applied last.
Pagination happens after full sort
Classic mistake this catches WHERE total > 10 fails — total is a SELECT alias and doesn't exist at step 2. You need HAVING COUNT(*) > 10 instead. Understanding execution order explains exactly why.

JOINs — The Full Picture

All JOINs combine rows from two tables. The differences are in what happens when there's no match.

INNER JOIN
Only rows where the condition matches in both tables.
SELECT a.id, u.name
FROM applications a
INNER JOIN users u ON a.user_id = u.id

Applications with no user record disappear. Users with no application disappear.

LEFT JOIN
All rows from left table, matching rows from right. NULLs if no match on right.
SELECT u.name, a.id
FROM users u
LEFT JOIN applications a ON a.user_id = u.id

All users returned. Users with no applications get NULL for application columns.

RIGHT JOIN
All rows from right table, matching from left. Rarely used — swap tables and use LEFT JOIN instead.
-- Equivalent to swapping tables in a LEFT JOIN
SELECT u.name, a.id
FROM applications a
RIGHT JOIN users u ON a.user_id = u.id
FULL OUTER JOIN
All rows from both tables. NULLs where no match exists on either side.
SELECT u.name, a.id
FROM users u
FULL OUTER JOIN applications a ON a.user_id = u.id

Use for reconciliation queries — find rows in either table that don't have a match.

Self JOIN — query a table against itself

-- Find users referred by another user
SELECT r.name AS referrer, u.name AS referred
FROM   users u
JOIN   users r ON u.referred_by = r.id;

CROSS JOIN — cartesian product

-- Every combination of plan and feature (e.g. pricing matrix)
SELECT p.name, f.feature_name
FROM   plans p
CROSS JOIN features f;

Returns M × N rows. Use deliberately — accidental cross joins are a common performance disaster.

Finding unmatched rows — the anti-join pattern

-- Users who have never submitted an application
SELECT u.id, u.name
FROM   users u
LEFT JOIN applications a ON a.user_id = u.id
WHERE  a.id IS NULL;  -- NULL means no match was found on the right side
applyGOAT use case "Which users in tenant X have not completed their profile?" LEFT JOIN users to profiles, filter WHERE profiles.id IS NULL. This is cleaner and faster than a NOT IN subquery in almost all cases.

Aggregation

Aggregation collapses multiple rows into one summary value. The core functions:

FunctionWhat it doesNULL handling
COUNT(*)Counts all rows including NULLsCounts NULL rows
COUNT(col)Counts rows where col is NOT NULLIgnores NULLs
COUNT(DISTINCT col)Counts unique non-null valuesIgnores NULLs
SUM(col)Sum of valuesIgnores NULLs
AVG(col)Mean of valuesIgnores NULLs — can skew averages
MIN(col) / MAX(col)Smallest / largest valueIgnores NULLs
 
-- Application stats per status for a tenant
SELECT
  status,
  COUNT(*)                        AS total,
  COUNT(DISTINCT user_id)         AS unique_users,
  AVG(EXTRACT(EPOCH FROM (updated_at - created_at))/3600) AS avg_hours_open
FROM   applications
WHERE  tenant_id = 'goat-001'
GROUP BY status
HAVING COUNT(*) > 5
ORDER BY total DESC;

Subqueries

Non-correlated subquery — runs once

-- Users who applied to the same job as the most popular job
SELECT DISTINCT user_id
FROM   applications
WHERE  job_id = (
  SELECT job_id
  FROM   applications
  GROUP BY job_id
  ORDER BY COUNT(*) DESC
  LIMIT 1
);

The inner query runs once, returns a single value, and the outer query uses it. Independent of the outer query.

Correlated subquery — runs once per row

-- Get each user's most recent application
SELECT u.id, u.name,
  (SELECT MAX(created_at)
   FROM applications a
   WHERE a.user_id = u.id) AS last_applied
FROM users u;

The inner query references u.id from the outer query. It re-executes for every row. Performance trap at scale — O(n × m). Use a JOIN or window function instead at production scale.

CTEs — Common Table Expressions

CTEs give a name to a subquery, making it readable and reusable within the same statement. The WITH clause.

-- Active users in the last 30 days, then their application counts
WITH active_users AS (
  SELECT DISTINCT user_id
  FROM   sessions
  WHERE  created_at > NOW() - INTERVAL '30 days'
    AND  tenant_id = 'goat-001'
),
user_app_counts AS (
  SELECT user_id, COUNT(*) AS app_count
  FROM   applications
  WHERE  user_id IN (SELECT user_id FROM active_users)
  GROUP BY user_id
)
SELECT u.name, uac.app_count
FROM   users u
JOIN   user_app_counts uac ON u.id = uac.user_id
ORDER BY uac.app_count DESC;
PostgreSQL 12 gotcha — the CTE optimization fence Before PostgreSQL 12, CTEs were always materialised — the inner query ran completely before the outer query could filter its results. This meant filters in the outer query couldn't push down into the CTE, sometimes causing full table scans. PostgreSQL 12+ inlines CTEs by default unless you write WITH ... AS MATERIALIZED. Know this — interviewers at companies running older Postgres will probe it.

Recursive CTEs

-- Walk an organisation hierarchy
WITH RECURSIVE org_tree AS (
  -- Anchor: start from CEO
  SELECT id, name, manager_id, 0 AS level
  FROM   employees
  WHERE  manager_id IS NULL

  UNION ALL

  -- Recursive: join each employee to their manager
  SELECT e.id, e.name, e.manager_id, ot.level + 1
  FROM   employees e
  JOIN   org_tree ot ON e.manager_id = ot.id
)
SELECT level, name FROM org_tree ORDER BY level, name;

Pattern: anchor query (base case) + UNION ALL + recursive query that references the CTE itself. Essential for trees and graphs stored in relational tables.

Window Functions — The Senior Differentiator

Window functions compute a value across a set of rows related to the current row — without collapsing them into groups like GROUP BY does. Every row is preserved in the output. This is the feature that separates junior SQL from senior SQL.

SELECT
  user_id,
  created_at,
  status,
  ROW_NUMBER()   OVER (PARTITION BY user_id ORDER BY created_at) AS application_number,
  RANK()         OVER (PARTITION BY user_id ORDER BY score DESC) AS score_rank,
  LAG(status, 1) OVER (PARTITION BY user_id ORDER BY created_at) AS prev_status,
  SUM(1)         OVER (PARTITION BY user_id)                     AS total_for_user
FROM applications
WHERE tenant_id = 'goat-001';
FunctionWhat it computesWhen to use
ROW_NUMBER() Sequential 1, 2, 3... per partition. No ties — always unique. Get the Nth row per group. Deduplication.
RANK() Rank with gaps: tied rows get same rank, next rank skips. 1,1,3,4. Leaderboards where ties leave gaps.
DENSE_RANK() Rank without gaps: 1,1,2,3. Leaderboards where tied rows shouldn't skip the next rank.
LAG(col, n) Value from N rows behind the current row in the window. Compare current row to previous row (e.g. detect status changes).
LEAD(col, n) Value from N rows ahead. Look-ahead comparisons.
SUM / COUNT / AVG OVER () Running or partition-level aggregation without collapsing rows. Running totals, per-group totals alongside detail rows.
NTILE(n) Divides rows into n buckets (quartiles, deciles). Score distribution bucketing.

Most common use: get the latest record per group

-- Latest application per user (without a subquery per user)
WITH ranked AS (
  SELECT *,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at DESC) AS rn
  FROM applications
  WHERE tenant_id = 'goat-001'
)
SELECT * FROM ranked WHERE rn = 1;

This is O(n log n) — one pass over the table. The correlated subquery equivalent is O(n²). At 1M rows the difference is enormous.

Set Operations

-- UNION: combine results, remove duplicates
SELECT user_id FROM active_users
UNION
SELECT user_id FROM premium_users;

-- UNION ALL: combine results, keep duplicates (faster — no dedup step)
SELECT user_id FROM active_users
UNION ALL
SELECT user_id FROM premium_users;

-- INTERSECT: rows in both result sets
SELECT user_id FROM active_users
INTERSECT
SELECT user_id FROM premium_users;

-- EXCEPT: rows in first set but not second
SELECT user_id FROM active_users
EXCEPT
SELECT user_id FROM premium_users;
UNION vs UNION ALL UNION ALL is always faster — no deduplication step. Use UNION only when you specifically need to remove duplicates. Many engineers default to UNION out of habit and pay an unnecessary sort cost.

Interview Playbook

Click any question to reveal a model answer.

// LIKELY INTERVIEW QUESTIONS — click to reveal answers
Model answer
"SQL executes in logical order: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY. The WHERE clause runs at step 2, before SELECT at step 5. Aliases are defined in the SELECT clause, so they simply don't exist when WHERE evaluates. This isn't a syntax limitation — it's a consequence of the evaluation order. The same reason you can't use a SELECT alias in GROUP BY or HAVING in most databases."
Follow-up answer
"ORDER BY — because it runs after SELECT (step 6 vs 5). That's the one place where SELECT aliases are valid. Some databases like MySQL and BigQuery also allow GROUP BY to reference SELECT aliases as an extension, but that's non-standard and can produce confusing results. PostgreSQL doesn't allow it in WHERE or HAVING, which is the correct behaviour."
Model answer
SELECT u.id, u.name FROM users u LEFT JOIN applications a ON a.user_id = u.id WHERE a.id IS NULL;

"LEFT JOIN brings back all users. When there's no matching application, the application columns are NULL. Filtering WHERE a.id IS NULL keeps only users with no applications — the anti-join pattern."
Follow-up answer
"LEFT JOIN + IS NULL and NOT EXISTS are equivalent in performance on modern PostgreSQL — the planner converts both into an anti-join internally. NOT IN is dangerous: if the subquery returns even one NULL value, the entire NOT IN condition returns NULL (due to SQL's three-valued logic), silently returning zero rows. This is a real production bug. I avoid NOT IN when the subquery can produce NULLs. My default is NOT EXISTS — it's semantically explicit and NULL-safe."
Model answer
"Correlated subquery approach (slow):
SELECT *, (SELECT MAX(created_at) FROM applications WHERE user_id = u.id) FROM users u
This runs the inner query once per user — O(n × m), catastrophic at scale.

Window function approach (fast):
WITH ranked AS (SELECT *, ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at DESC) AS rn FROM applications) SELECT * FROM ranked WHERE rn = 1
One pass over the table, O(n log n) for the sort within each partition. At 1M applications, this is the difference between seconds and milliseconds."
Follow-up answer
"PARTITION BY is like GROUP BY for the window — it says 'restart the computation for each unique value of this column.' OVER (PARTITION BY user_id ORDER BY created_at DESC) means: within each user's applications, number them starting from the most recent. The key difference from GROUP BY is that rows are NOT collapsed — every application row is in the output, just with an extra column.

ROW_NUMBER vs RANK: if two applications have the same created_at timestamp, ROW_NUMBER assigns them 1 and 2 arbitrarily (non-deterministic for ties). RANK would assign both 1 and the next row gets 3 (gap). DENSE_RANK assigns both 1 and the next gets 2 (no gap). For 'get the latest single row per user' I use ROW_NUMBER — ties are rare and I just need one row."
Model answer
"Subquery: use when the logic is simple and only referenced once. Inline and readable for one-shot use. CTE: use when a subquery is referenced more than once in the same query, or when the query has multiple logical steps that benefit from named, readable stages. Temp table: use when you need to materialise a large intermediate result, index it, and use it across multiple subsequent queries in a session — ANALYZE can even be run on it. CTEs are query-scoped; temp tables are session-scoped."
Follow-up answer
"Before PostgreSQL 12, CTEs were always materialised — the planner treated the CTE as a 'black box' and couldn't push predicates from the outer query into it. If you filtered the CTE result outside with WHERE, that filter couldn't reduce work inside the CTE. This sometimes caused full table scans where index scans were possible. PostgreSQL 12 changed the default to inline CTEs (treating them like subqueries), allowing the planner to optimise across CTE boundaries. You can still force materialisation with WITH x AS MATERIALIZED (...) when you specifically want to prevent the CTE being evaluated multiple times."
Gap to watch Most candidates can write basic GROUP BY queries. The senior differentiator is window functions — specifically knowing when a correlated subquery can be replaced by a window function and why it matters for performance. If asked "how would you get the top N per group," the answer is ROW_NUMBER() OVER (PARTITION BY ...), not a subquery.
