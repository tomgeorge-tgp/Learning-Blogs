Database Internals

Query Execution
& the Query Planner

What happens between typing a SQL query and receiving a result — and how to read the plan that decides it.

The Query Lifecycle

A SQL string goes through four distinct stages before any data is touched. Understanding this pipeline is how you diagnose slow queries and know which tool to reach for.

1
Parse
The SQL string is tokenised and checked for syntax. If valid, it becomes a parse tree (abstract syntax tree). Errors here are syntax errors — column names and types aren't validated yet.
2
Analyse / Rewrite
PostgreSQL validates the parse tree against the catalog — checking that tables and columns actually exist, resolving types, and expanding views. Views are rewritten into their underlying queries here. Output: a query tree with resolved references.
3
Plan (the interesting one)
The query planner (also called the query optimiser) generates multiple ways to execute the query, estimates the cost of each, and picks the cheapest. It uses table statistics (row counts, value distributions) and knowledge of available indexes. Output: an execution plan — a tree of plan nodes.
4
Execute
The executor walks the plan tree. Each node pulls rows from its child nodes (a pipeline model called the Volcano / Iterator model). Data flows upward through filter, sort, join, and aggregate nodes until results reach the client.
The key insight You can't influence parsing or execution — those are fixed. Everything that matters for performance happens in the Plan step. Your indexes, table statistics, query shape, and join order all feed the planner's decision. When a query is slow, the question is always: "What did the planner choose and why?"

How the Planner Estimates Costs

The planner doesn't run the query to find the fastest plan. It uses statistics stored in pg_statistic (exposed via pg_stats) to estimate the cost of each plan without executing it.

Key statistics the planner uses

StatisticWhat it tells the plannerSource
n_distinct Number of unique values in a column. Low = low selectivity (e.g. boolean). High = highly selective (e.g. user_id). pg_stats.n_distinct
most_common_vals Most frequent values and their frequencies. Used to estimate how many rows a WHERE filter will return. pg_stats.most_common_vals
histogram_bounds Distribution of values across ranges. Used for range query selectivity estimation. pg_stats.histogram_bounds
reltuples Estimated number of rows in the table. pg_class.reltuples
relpages Number of disk pages the table occupies. Drives sequential scan cost. pg_class.relpages

Statistics are collected by the ANALYZE command (runs automatically via autovacuum). Stale statistics cause the planner to make wrong choices — often choosing a sequential scan over an index scan because it underestimates how many rows the filter will eliminate.

-- Manually refresh statistics on a table after large bulk loads
ANALYZE applications;

-- View the statistics for a specific column
SELECT * FROM pg_stats WHERE tablename = 'applications' AND attname = 'status';

Scan Strategies

The planner's first decision per table: how to access the rows. Three main strategies.

Sequential Scan
Reads every page in the table from start to finish. Best when fetching a large fraction of the table (no index would help). I/O is sequential — fast on SSDs.
Chosen when: no useful index, or filter returns >~5–20% of rows.
Index Scan
Walks the B-tree index to find matching rows, then fetches each row from the heap (the actual table). Each row lookup is a random I/O.
Chosen when: highly selective filter, small fraction of rows returned.
Bitmap Index Scan
Two-pass: first, collects all heap page locations from the index into a bitmap. Second, reads those pages in physical order — converting random I/O into sequential. Used when the index is moderately selective (too many rows for an index scan, too few for a sequential scan).
Chosen when: medium selectivity. Also used to combine multiple indexes (Bitmap AND/OR).
Index-only scan A fourth strategy: when all columns needed by a query are included in the index itself (a covering index), PostgreSQL never touches the heap at all. This is the fastest possible read — memory access to index pages only. More on this in Lesson 06.

Why a small table uses a sequential scan even with an index

The planner compares costs. For a 100-row table, reading all 100 rows sequentially (maybe one page of I/O) is cheaper than walking the B-tree (multiple index page reads) + fetching heap rows. The index adds overhead. The planner knows this and chooses the sequential scan correctly.

Join Strategies

When joining two tables, the planner picks one of three algorithms.

StrategyHow it worksBest forCost
Nested Loop For each row in the outer table, scan the inner table (using an index if available). Like two for-loops. Small outer table + indexed inner table. LIMIT queries. O(n × m) worst case. O(n × log m) with index.
Hash Join Build a hash table from the smaller table. Probe it for each row from the larger table. Large unsorted joins where no index exists. Equi-joins. O(n + m). Requires memory for the hash table.
Merge Join Both tables must be sorted on the join key. Merge them in one pass like merge-sort. Both sides already sorted (by index or ORDER BY). Range joins. O(n log n + m log m) for sort. O(n + m) if pre-sorted.
Hash join memory warning Hash joins build an in-memory hash table from the smaller relation. If it exceeds work_mem (default 4MB in PostgreSQL), it spills to disk — dramatically slower. For large analytical joins, increasing work_mem for the session can change a 30-second query into a 2-second one.
SET work_mem = '256MB'; — set per-session, not globally, to avoid OOM.

Reading EXPLAIN ANALYZE

EXPLAIN shows the plan the planner chose. EXPLAIN ANALYZE actually runs the query and shows real execution times. Always use ANALYZE to diagnose slow queries — estimated costs and actual costs can differ significantly.

EXPLAIN ANALYZE
SELECT a.id, u.name, a.status
FROM   applications a
JOIN   users u ON a.user_id = u.id
WHERE  a.tenant_id = 'goat-001'
  AND  a.created_at > '2024-01-01'
ORDER BY a.created_at DESC
LIMIT 20;
Hash Join (cost=145.20..892.45 rows=20 width=52) (actual time=3.421..18.234 rows=20 loops=1) Hash Cond: (a.user_id = u.id) -> Index Scan using idx_apps_tenant_date on applications a (cost=0.56..712.33 rows=2847 width=44) (actual time=0.082..14.113 rows=2847 loops=1) Index Cond: (tenant_id = 'goat-001') Filter: (created_at > '2024-01-01') Rows Removed by Filter: 153 -> Hash (cost=89.20..89.20 rows=4400 width=16) (actual time=2.891..2.891 rows=4400 loops=1) Buckets: 8192 Batches: 1 Memory Usage: 238kB -> Seq Scan on users u (cost=0.00..89.20 rows=4400 width=16) (actual time=0.012..1.203 rows=4400 loops=1) Planning Time: 0.834 ms Execution Time: 18.451 ms

How to read a plan — bottom up

The plan is a tree. Data flows from the leaves upward. Read it from the most-indented lines outward.

FieldWhat it means
cost=0.56..712.33 Estimated cost: startup cost .. total cost. Units are arbitrary (related to disk page reads). Lower is better.
rows=2847 Estimated number of rows this node will output. Compare to actual rows to spot planner miscalibration.
actual time=0.082..14.113 Actual milliseconds: time to first row .. time to last row. Only with ANALYZE.
loops=1 How many times this node was executed. In nested loops, the inner node may loop many times.
Rows Removed by Filter: 153 Rows read from the index but rejected by a secondary filter. High numbers here mean the index isn't as selective as ideal.
Batches: 1 vs Batches: N Hash join batches. Batches > 1 means the hash table spilled to disk. Performance problem — increase work_mem.

Red flags in an EXPLAIN ANALYZE output

  • Estimated rows ≪ actual rows: Stale statistics. Run ANALYZE on the table. The planner made its choice based on wrong data.
  • Seq Scan on a large table: Either no index exists, the index isn't being used (check column types, function wrapping), or the query is returning too large a fraction of rows.
  • Hash Batches > 1: Hash table spilled to disk. Increase work_mem for the session.
  • loops=N with high actual time per loop: Nested loop with a non-indexed inner scan. Add an index on the join key of the inner table.
  • Sort node with large actual time: External sort (spilled to disk). Increase work_mem or provide the data pre-sorted via an index.

pg_stat_statements — Production Query Monitoring

The pg_stat_statements extension tracks cumulative statistics for every query shape executed on the server. This is the tool for finding slow queries in production without looking at individual EXPLAIN outputs.

-- Enable in postgresql.conf:
-- shared_preload_libraries = 'pg_stat_statements'

-- Find the top 10 slowest queries by total time spent
SELECT
  LEFT(query, 80)          AS query_preview,
  calls,
  total_exec_time::int     AS total_ms,
  mean_exec_time::int      AS avg_ms,
  (total_exec_time / calls)::int AS avg_ms_check,
  rows
FROM   pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;

This shows you where the database is actually spending time across all requests — not just the queries you happen to EXPLAIN. The highest total_exec_time entries are your optimisation targets regardless of per-call latency.

applyGOAT workflow Enable pg_stat_statements on your PostgreSQL instance. After a day of traffic, query it to find the top 10 by total_exec_time. Those are your real slow queries — often a surprise compared to what you expected. Then EXPLAIN ANALYZE the top ones to diagnose.

Interview Playbook

Click to reveal answers.

// LIKELY INTERVIEW QUESTIONS — click to reveal answers
Model answer
"First, identify the query — in production I'd use pg_stat_statements to find the highest total_exec_time queries rather than guessing. Then run EXPLAIN ANALYZE on the specific query to see the actual execution plan. I look at: (1) are there sequential scans on large tables where an index should exist, (2) are estimated rows close to actual rows — if not, statistics are stale and ANALYZE needs to run, (3) are hash joins spilling to disk (Batches > 1), (4) are nested loops executing the inner node many times without an index. Once I know the bottleneck, the fix is usually one of: add an index, rewrite the query shape, update statistics, or increase work_mem for a session."
Follow-up answer
"EXPLAIN shows the plan the planner intends to use — estimated costs, estimated row counts, plan nodes. It does not execute the query. EXPLAIN ANALYZE actually runs the query and augments the plan output with actual execution times, actual row counts, and loop counts. Always use ANALYZE when diagnosing a slow query — the estimated costs can be wildly wrong when statistics are stale, and you need the actual numbers to know what's really happening. The only reason to use plain EXPLAIN is when the query is too expensive to run (e.g. a destructive UPDATE you only want to plan, not execute)."
Model answer
"Several reasons: (1) The query returns too large a fraction of rows — the planner correctly decides a sequential scan is cheaper than many random index reads. (2) The column is wrapped in a function — WHERE LOWER(email) = ? prevents the index on email from being used because the function transforms the value before comparison. (3) Implicit type coercion — if user_id is a UUID and you pass a string, the planner can't use the index without a cast. (4) Stale statistics — the planner underestimates selectivity and thinks the index isn't worth it. Fix: run ANALYZE. (5) The index doesn't match the query's column order (composite index left-prefix rule)."
Follow-up answer
"You can hint the planner by disabling specific scan types for debugging: SET enable_seqscan = off. This forces the planner away from sequential scans so you can verify the index works. But you should never leave this in production — it overrides the planner's cost calculations globally and can make other queries much worse. The right fix is to understand why the planner chose the sequential scan and either fix the query, update statistics, or accept that the planner is correct and the sequential scan genuinely is faster for that query."
Model answer
"Hash join is faster when: joining two large tables without useful indexes on the join key, or when the inner table would need many random reads in a nested loop. It builds a hash table from the smaller relation (O(n)) then probes it for each outer row (O(m)) — total O(n+m). Nested loop is faster when: the outer table is very small and the inner table has an index on the join key — each inner lookup is O(log m) and the number of loops is small. Also, nested loops can return the first row quickly (useful for LIMIT queries), while hash join must build the full hash table before returning any results."
Follow-up answer
"When the hash table exceeds work_mem, PostgreSQL spills it to disk in 'batches' — you'll see 'Batches: N' in EXPLAIN ANALYZE. Each batch requires an extra I/O pass over the data. A hash join that fits in memory (Batches: 1) is fast. The same join with Batches: 8 can be 10–20× slower. The fix is increasing work_mem for that session: SET work_mem = '256MB'. Don't increase it globally unless you have enough RAM — each sort or hash node in a query plan can use up to work_mem, and a complex query with multiple joins can multiply that."
Gap to watch Most candidates say "run EXPLAIN and add an index." Senior answer: explain what the planner does with statistics, how to read actual vs estimated row counts to detect stale stats, what each scan and join strategy means, and when an index won't be used even if it exists. The planner is not a black box — know how it thinks.
