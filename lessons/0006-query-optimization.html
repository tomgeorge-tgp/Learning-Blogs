Query Optimization

The specific patterns that break indexes, create N+1 problems, and kill pagination — and the rewrites that fix them.

Optimization #1 — Never Wrap Indexed Columns in Functions

The single most common reason a perfectly good index is ignored. When you apply a function to an indexed column in a WHERE clause, the planner can no longer use the index — it has to compute the function for every row and compare the result.

Breaks the index
-- Index on email — NOT used
WHERE LOWER(email) = 'tom@example.com'

-- Index on created_at — NOT used
WHERE DATE(created_at) = '2024-01-15'

-- Index on name — NOT used
WHERE SUBSTRING(name, 1, 3) = 'Tom'
Preserves the index
-- Store emails lowercase, or use expression index
WHERE email = LOWER('Tom@example.com')

-- Use a range instead of extracting
WHERE created_at >= '2024-01-15'
  AND created_at < '2024-01-16'

-- Prefix search with LIKE (only left-anchored)
WHERE name LIKE 'Tom%'
Expression indexes — the escape hatch If you must filter on a transformed column, create an index on the expression itself:
CREATE INDEX idx_users_email_lower ON users (LOWER(email));
Now WHERE LOWER(email) = ? uses this index. PostgreSQL matches the expression in the query to the expression in the index.

Optimization #2 — Implicit Type Coercion

If the type of your parameter doesn't match the column type exactly, PostgreSQL applies an implicit cast — which behaves like wrapping the column in a function and breaks the index.

Coercion breaks index
-- user_id is UUID, but passing a string
-- PostgreSQL must cast every stored UUID to text
WHERE user_id = 'abc-123'

-- id is INTEGER, but param is text
WHERE id = '42'
Explicit cast preserves index
-- Cast the parameter, not the column
WHERE user_id = 'abc-123'::uuid

-- Or ensure your ORM passes the right type
-- SQLAlchemy: use uuid.UUID() objects, not strings

In FastAPI with SQLAlchemy or asyncpg, pass UUIDs as uuid.UUID objects, not strings. The driver handles the correct wire type and the planner uses the index.

Optimization #3 — The N+1 Query Problem

The most common ORM-related performance failure. You fetch N records, then issue one query per record to get related data — N+1 total queries instead of 1 or 2.

N+1 (ORM lazy loading)
# SQLAlchemy lazy loading
users = session.query(User).all()  # 1 query
for user in users:
    # Each access fires a new SELECT
    print(user.applications)       # N queries

# FastAPI equivalent
users = await db.fetch_all("SELECT * FROM users")
for user in users:
    apps = await db.fetch_all(
        "SELECT * FROM applications WHERE user_id=$1",
        user['id']  # N separate DB round trips
    )
Eager load or JOIN
# SQLAlchemy eager loading
from sqlalchemy.orm import selectinload
users = session.query(User)\
    .options(selectinload(User.applications))\
    .all()  # 2 queries total: users, then all apps

# Or a single JOIN query
users_with_apps = await db.fetch_all("""
  SELECT u.id, u.name,
         a.id AS app_id, a.status
  FROM   users u
  LEFT JOIN applications a ON a.user_id = u.id
  WHERE  u.tenant_id = $1
""", tenant_id)  # 1 query, all data
applyGOAT exposure Any endpoint that returns a list of users with their applications or cover letters is a classic N+1 risk. In FastAPI with async SQLAlchemy or asyncpg, use selectinload for ORM, or write explicit JOIN queries. Log slow requests in production — N+1 usually looks like many fast queries, not one slow one, which is why it hides in p50 but kills p99 and total DB load.

Optimization #4 — EXISTS vs IN vs JOIN (for filtering)

PatternHow it worksPerformanceNULL safety
WHERE id IN (SELECT ...) Evaluates inner query, creates a list, checks each outer row against the list. OK for small lists. Risk: if any inner value is NULL, can return zero rows silently. Unsafe with NULLs
WHERE EXISTS (SELECT 1 FROM ... WHERE ...) For each outer row, checks if the inner query returns at least one row. Short-circuits on first match. Excellent for semi-joins. NULL-safe. Modern planners convert IN to EXISTS anyway. Safe
INNER JOIN Combines tables, returns only matching rows. One pass with hash or merge join. Best when you also need columns from the joined table. Avoids correlated subquery. Safe 
-- Three equivalent queries: "users who have at least one application"

-- Option A: IN (can have NULL trap)
SELECT id FROM users WHERE id IN (SELECT user_id FROM applications);

-- Option B: EXISTS (recommended for filtering)
SELECT id FROM users u WHERE EXISTS (
  SELECT 1 FROM applications a WHERE a.user_id = u.id
);

-- Option C: JOIN (best when you need application data too)
SELECT DISTINCT u.id
FROM users u
JOIN applications a ON a.user_id = u.id;

For pure filtering (you only need data from the outer table), EXISTS is the safest and most explicit. PostgreSQL's planner typically generates the same plan for EXISTS and IN, but EXISTS is clearer in intent and avoids the NULL pitfall.

Optimization #5 — Pagination: OFFSET is a Trap

The most common pagination pattern is offset-based. It feels correct but is a performance disaster at large page numbers.

OFFSET pagination (gets worse with page number)
-- Page 1: fast
SELECT * FROM applications
WHERE tenant_id = 'goat-001'
ORDER BY created_at DESC
LIMIT 20 OFFSET 0;

-- Page 500: must scan and discard 9,980 rows
SELECT * FROM applications
WHERE tenant_id = 'goat-001'
ORDER BY created_at DESC
LIMIT 20 OFFSET 9980;
-- Cost grows linearly with page number
Cursor-based pagination (constant cost)
-- First page
SELECT * FROM applications
WHERE tenant_id = 'goat-001'
ORDER BY created_at DESC, id DESC
LIMIT 20;
-- Remember last row: {created_at: T, id: X}

-- Next page — filter by the cursor
SELECT * FROM applications
WHERE tenant_id = 'goat-001'
  AND (created_at, id) < ('2024-01-15T10:30:00', 'abc-123')
ORDER BY created_at DESC, id DESC
LIMIT 20;
-- Index can jump directly to that point — O(log n)

Why OFFSET is slow: PostgreSQL must scan and discard all rows before the offset point. OFFSET 9980 = scan 10,000 rows, return 20, throw away 9,980. Cost grows O(offset). At page 500 with 20 rows per page, you're scanning 10,000 rows per request.

Cursor pagination trade-offs: You can't jump to "page 500" directly. You must page forward sequentially. This is fine for "load more" or infinite scroll (most real-world UIs). For admin UIs needing random page access, OFFSET may be acceptable if the dataset is bounded.

Compound cursor key Always include a tiebreaker (like id) in the cursor alongside your sort key (created_at). If multiple rows share the same timestamp, a single-key cursor is ambiguous and you'll skip or duplicate rows. The compound cursor (created_at, id) is always unique.

Optimization #6 — Covering Indexes (Index-Only Scans)

When every column a query needs is included in the index, PostgreSQL never touches the heap (the table itself). This is an index-only scan — the fastest possible read.

-- Query: summary of open applications per tenant
SELECT tenant_id, COUNT(*) AS total
FROM   applications
WHERE  status = 'open'
GROUP BY tenant_id;

-- Standard index on (status): planner uses index to find open rows,
-- then must fetch heap pages for tenant_id values — two I/Os per row

-- Covering index: include tenant_id in the index
CREATE INDEX idx_apps_status_covering
ON applications (status) INCLUDE (tenant_id);
-- Now the planner reads tenant_id directly from the index
-- No heap access needed — index-only scan
INCLUDE vs part of the key INCLUDE (tenant_id) stores tenant_id in the index leaf pages but NOT in the tree nodes — it's not searchable. This keeps the index tree compact and efficient, while still allowing the value to be read without a heap fetch. If you put tenant_id in the key itself, it becomes part of the sort order (potentially changing which queries can use the index).

Optimization #7 — Partial Indexes

A partial index indexes only the rows satisfying a WHERE clause. Smaller, faster, more cache-friendly than a full index.

-- 90% of applications are 'closed'. Most queries are for 'open' ones.
-- A full index on status is large and mostly useless.

-- Partial index: only open applications
CREATE INDEX idx_apps_open
ON applications (tenant_id, created_at DESC)
WHERE status = 'open';

-- This index is tiny (10% of rows), fits in cache easily,
-- and is perfectly targeted for the most common query pattern.

-- The query must include the partial condition to use it:
SELECT * FROM applications
WHERE  tenant_id = 'goat-001'
  AND  status = 'open'   -- required for index use
ORDER BY created_at DESC;

Use partial indexes when a large fraction of rows are rarely queried (soft-deleted rows, completed jobs, archived records) and your hot queries always filter on that condition.

Optimization #8 — SELECT * is Expensive

Selecting all columns prevents index-only scans, sends unnecessary data over the wire, and increases memory pressure on the application server.

Never do this in production
SELECT * FROM applications
WHERE tenant_id = 'goat-001';
-- Fetches all columns including large JSONB blobs,
-- resume_content text, metadata, etc.
-- Cannot use index-only scans
-- Transfers MB of data per request
Select only what you need
SELECT id, status, job_title, created_at
FROM   applications
WHERE  tenant_id = 'goat-001'
ORDER BY created_at DESC
LIMIT 20;
-- Minimal data transfer
-- Allows covering/index-only scans
-- Faster serialization in FastAPI

Optimization #9 — Batching Writes

Individual INSERT statements have overhead: each one parses, plans, and executes as a separate transaction. Batch inserts reduce this dramatically.

1000 individual INSERTs
# Each INSERT is a round trip + transaction
for record in records:
    await db.execute(
        "INSERT INTO events(user_id, type) VALUES($1, $2)",
        record.user_id, record.type
    )
# 1000 round trips, 1000 transactions
Bulk INSERT
# One round trip, one transaction
await db.executemany(
    "INSERT INTO events(user_id, type) VALUES($1, $2)",
    [(r.user_id, r.type) for r in records]
)

# Or COPY for very large datasets (fastest)
await conn.copy_records_to_table(
    'events',
    records=[(r.user_id, r.type) for r in records],
    columns=['user_id', 'type']
)

COPY is PostgreSQL's bulk load mechanism — bypasses most of the per-row overhead including index update batching. For importing thousands of rows, it's 10–100× faster than individual INSERTs.

The Optimization Workflow

This is the sequence for diagnosing any slow query in production. Do it in this order — don't skip steps.

  1. Find the target: pg_stat_statements ORDER BY total_exec_time DESC. Optimize what the data says is expensive, not what feels slow.
  2. Run EXPLAIN ANALYZE: See the actual plan. Identify the slow node (highest actual time).
  3. Check for missing indexes: Seq Scan on a large table with a selective WHERE clause? Add an index on the filter column(s).
  4. Check for index-killing patterns: Function on column? Type mismatch? Rewrite the query.
  5. Check statistics: Estimated rows ≪ actual rows? Run ANALYZE.
  6. Check memory: Hash join with Batches > 1? Sort with disk spill? Increase work_mem for the session.
  7. Verify: Re-run EXPLAIN ANALYZE and confirm actual time improved.

Model answer
"The index is on the raw email column. The query applies LOWER() to the column before comparing, which means PostgreSQL can't use the index — it has to compute LOWER(email) for every row and compare the result. The index stores the original values, not the lowercased ones, so it's useless for this predicate."
Follow-up answer
"Create an expression index: CREATE INDEX idx_users_email_lower ON users (LOWER(email)). PostgreSQL stores the lowercased value in this index. Now the planner matches the LOWER(email) in the query to the LOWER(email) in the index and uses it. The query doesn't need to change at all — you just need the expression index to match the expression in the predicate."
Model answer
"This is offset pagination. The query is: LIMIT 20 OFFSET (page × 20). PostgreSQL must scan and discard all rows before the offset — OFFSET 9980 means reading 10,000 rows to return 20. Cost is O(offset), so it grows linearly with page number. At page 500 you're doing 500× more work than page 1. Switch to keyset/cursor pagination: store the last row's sort key, then filter by it on the next request. The planner can jump directly to that position using the index — constant O(log n) regardless of page number."
Follow-up answer
"Cursor pagination: constant performance, can't jump to arbitrary pages, more complex to implement, cursor must encode the full sort key. Offset pagination: linear performance degradation, can jump to page N directly, trivial to implement. In practice: cursor for infinite scroll / load-more UIs (almost all production apps), offset only for bounded admin UIs where users genuinely need page 47 of 50. One gotcha: cursor must include a tiebreaker column (usually id) alongside the sort key, otherwise tied sort values create ambiguous cursors and you lose or duplicate rows."
Model answer
"N+1: you fetch N parent records in 1 query, then issue 1 additional query per parent to load related data. Total: N+1 queries. In FastAPI it appears whenever you iterate over a list of objects and access a relationship inside the loop — with SQLAlchemy lazy loading, each relationship access fires a separate SELECT. At N=100 users this is 101 database round trips where 2 would suffice. The symptom in production is many fast queries (not one slow one) that together add up to high total DB time and high p99 latency."
Follow-up answer
"SQLAlchemy lazy loading is the default — accessing a relationship on an ORM object fires a SELECT when the attribute is first accessed. Fix options: (1) selectinload — loads all related objects in a second bulk query (2 queries total, regardless of N). (2) joinedload — loads via LEFT JOIN in the original query (1 query, can produce large result sets with many-to-many). (3) Write explicit JOIN queries with asyncpg/databases and skip the ORM for read-heavy endpoints. I use selectinload by default for one-to-many, joinedload for many-to-one (parent fetch), and raw SQL for complex analytical queries."
Optimization priority Fix in this order: (1) function on indexed column — one-line fix, huge gain. (2) N+1 — usually an ORM config issue. (3) Missing index — EXPLAIN ANALYZE will show the seq scan. (4) Offset pagination — architectural change, do it for high-traffic endpoints first. (5) covering/partial indexes — micro-optimization for high-frequency queries.
