Database Design

Database Normalization

Why bad table design causes data corruption by design — and the rules that prevent it.

Why Normalization Exists — The Anomalies

Normalization isn't about aesthetic table design. It's about preventing three categories of bugs that occur naturally when data is stored redundantly. These are called modification anomalies.

Consider this denormalized table for applyGOAT:

applications(
  app_id, user_id, user_email, user_name,
  job_id, company_name, company_industry,
  status, applied_at
)
Update Anomaly
Tom changes his email. You must update it in every application row. Miss one: your database now has two "truths" for Tom's email. Inconsistent forever unless you find all rows.
Insertion Anomaly
You can't store a company in the database unless a user has already applied to one of their jobs. Company data only exists as a side effect of an application row.
Deletion Anomaly
Tom deletes his only application to CompanyX. The record of CompanyX's industry disappears from the database entirely — you've accidentally deleted company metadata.

All three anomalies come from the same root cause: one fact stored in multiple places. Normalization is the process of reorganizing tables so each fact is stored exactly once.

Functional Dependencies — The Foundation

Before the normal forms: you need to understand functional dependency. Column B is functionally dependent on column A if knowing A's value tells you exactly one B value.

user_id  →  user_email     (knowing the user_id tells you their email — 1:1)
user_id  →  user_name      (same)
job_id   →  company_name   (knowing the job tells you the company)
company_name → industry    (knowing the company tells you the industry)

-- But NOT:
user_id  →  status         (one user can have many application statuses)
app_id   →  user_id        (an application belongs to one user — this IS a dependency)

Normal forms are rules about which functional dependencies are allowed to exist within a table, relative to its primary key.

1NF
First Normal Form
Rule: Every column contains atomic (indivisible) values. No repeating groups. Every row is uniquely identifiable by a primary key.

Violation: Storing multiple values in one cell.

-- VIOLATES 1NF: multiple skills in one column
users(user_id, name, skills)
-- row: (1, 'Tom', 'Python,FastAPI,React')
-- Cannot query by individual skill. Can't index. String parsing in app code.
-- 1NF: atomic values, separate table for skills
users(user_id, name)
user_skills(user_id, skill)
-- Can now: WHERE skill = 'Python', INDEX on skill, JOIN cleanly

The red flag in a non-1NF column: comma-separated lists, JSON arrays used as filters, or columns like tag1, tag2, tag3. These force string parsing in application code and prevent efficient querying.

JSONB exception PostgreSQL's JSONB columns technically store non-atomic data, but with GIN indexes they can be queried efficiently. This is a pragmatic exception to 1NF — acceptable when the JSON is opaque metadata that you never filter on individual keys, or when GIN indexing covers your query patterns.
2NF
Second Normal Form
Rule: Must be in 1NF. Every non-key column must depend on the entire primary key — not just part of it.

2NF only applies when the primary key is composite (multiple columns). A partial dependency is when a non-key column depends on only one column of the composite key.

-- PK: (app_id, skill_id) — composite
application_skills(app_id, skill_id, skill_name, applicant_email)
--
-- PARTIAL DEPENDENCY: skill_name depends only on skill_id (not app_id)
-- skill_name is duplicated across every application that uses that skill
-- Violates 2NF
-- 2NF: decompose into separate tables
application_skills(app_id, skill_id)   -- PK: (app_id, skill_id)
skills(skill_id, skill_name)           -- skill_name depends on its full PK
applications(app_id, applicant_email)  -- applicant_email depends on its full PK
3NF
Third Normal Form
Rule: Must be in 2NF. No non-key column should depend on another non-key column (no transitive dependencies).

A transitive dependency: A → B → C, where A is the key but B is not. C depends on the key transitively through B.

-- PK: app_id
applications(app_id, job_id, company_id, company_name, company_industry)
--
-- TRANSITIVE: app_id → company_id → company_name
--             app_id → company_id → company_industry
-- company_name and company_industry depend on company_id, not app_id directly
-- Update a company's name: every application row for that company must change
-- Violates 3NF
-- 3NF: each fact in its own table
applications(app_id, job_id, company_id, status, applied_at)
companies(company_id, company_name, industry)
jobs(job_id, company_id, title, location)
-- company_name stored once in companies — updating it updates everywhere
The 3NF test For each non-key column: "Does this column describe the primary key, the whole primary key, and nothing but the primary key?" If no — it belongs in a separate table. (Mnemonic: "the key, the whole key, and nothing but the key.")
BCNF
Boyce-Codd Normal Form
Rule: For every non-trivial functional dependency X → Y, X must be a superkey (a set of columns that uniquely identifies a row).

BCNF is a stricter version of 3NF. The difference surfaces when a table has multiple overlapping candidate keys.

-- A user can have only one active subscription per plan
-- A subscription_code belongs to exactly one plan
user_subscriptions(user_id, plan_id, subscription_code)
-- Candidate keys: (user_id, plan_id) AND (user_id, subscription_code)
-- But: subscription_code → plan_id (the code determines the plan)
-- subscription_code is NOT a superkey — violates BCNF

In practice, violations of BCNF (beyond 3NF) are rare in typical application schemas. Most designs that reach 3NF automatically satisfy BCNF.

The applyGOAT Schema — Applied

Here's how a typical first-draft applyGOAT schema evolves through normalization:

-- First draft (violates 2NF and 3NF)
applications(
  id, user_id, user_email, user_plan,
  job_id, job_title, company_id, company_name, company_industry,
  status, score, notes, applied_at
)
-- Normalized to 3NF
users(id, email, name, created_at)
subscriptions(id, user_id, plan_id, status, expires_at)
plans(id, name, features, price)

companies(id, name, industry, website)
jobs(id, company_id, title, location, salary_range, posted_at)

applications(
  id, user_id, job_id,
  status, score, notes, applied_at
)
-- user_email: join to users. company_name: join to jobs→companies.
-- No redundancy. One change in companies updates everywhere.

When to Denormalize — Intentionally

Normalization is not the end state. At production scale, denormalization is a deliberate performance trade-off: accept some redundancy to avoid expensive joins on hot read paths.

ScenarioDenormalization approachConsistency strategy
Dashboard shows application count per user on every page load Add application_count column to users table. Update it on INSERT/DELETE via trigger or application logic. Trigger keeps it in sync. Occasional recompute job as safety net.
Search result cards show company name alongside job title Store company_name denormalized in jobs table (it changes rarely). Update jobs.company_name when companies.name changes (application or trigger).
Analytics dashboard over millions of rows Pre-aggregate into a separate summary table, refreshed on schedule. Eventual consistency acceptable — dashboard is slightly behind.
Audit log / event sourcing Snapshot full record state at event time (fully denormalized by design). Events are immutable — no consistency issue.
Denormalization is a paid-off technical debt The moment you denormalize, you own the consistency strategy for that redundancy. If you forget to update the denormalized column, you now have silent data inconsistency. Document every denormalization decision with why it was made and how consistency is maintained.

Interview Playbook

Click to reveal answers.

// LIKELY INTERVIEW QUESTIONS — click to reveal answers
Model answer
"Using a job application table: 1NF requires atomic values and no repeating groups — skills must be a separate table, not a comma-separated string in one column. 2NF requires that every non-key column depends on the entire composite primary key — if the PK is (app_id, skill_id), then skill_name should go in a skills table because it only depends on skill_id, not the pair. 3NF requires no transitive dependencies — company_name shouldn't be in the applications table because it depends on company_id, not app_id. Each fact belongs in its own table."
Follow-up answer
"1NF prevents the need to parse composite values in application code and enables proper indexing. 2NF prevents update anomalies on partial key dependencies — if skill_name is in the application_skills table, changing the skill name requires updating every row that skill appears in. 3NF prevents transitive update anomalies — company_name stored in applications means updating a company's name requires touching every application row. Each normal form collapses a specific category of redundancy that was causing a specific class of anomaly."
Model answer
"Optimize before denormalizing. Check EXPLAIN ANALYZE: are indexes missing on the join columns? Is work_mem causing hash join spills? Is the result cacheable with Redis? Denormalization should be a last resort because it moves a consistency problem into my application. If after optimization the join is still the bottleneck and the data is cacheable — cache at the Redis layer instead of denormalizing the schema. Only denormalize the schema if the data changes too frequently to cache and the join genuinely can't be fast enough with indexes."
Follow-up answer
"Three options: (1) Application-level consistency — when company.name updates, also update applications.company_name in the same transaction. Risks missing it if any code path updates companies without the join. (2) Database trigger — ON UPDATE on companies, trigger propagates to applications. Invisible to application code, always fires, but adds write latency and can surprise future developers. (3) Scheduled reconciliation job — periodically re-sync denormalized columns from source of truth. Accepts eventual consistency. I'd go with option 2 (trigger) documented in migration comments, or option 1 enforced through a service layer that's the only code path that can update companies."
Model answer
"Tenants table: id, name, domain, created_at. Plans: id, name, price, features (JSONB). Subscriptions: id, tenant_id → tenants, plan_id → plans, status, expires_at. Users: id, tenant_id → tenants, email, name, created_at. Companies: id, name, industry, website. Jobs: id, company_id → companies, title, location, salary_range. Applications: id, user_id → users, job_id → jobs, status, score, notes, applied_at. Each entity owns its own facts. User email is in users. Company name is in companies. Application status is in applications. No column in any table depends on a non-key column of the same table."
Follow-up answer
"Three layers: (1) Row-level security (RLS) in PostgreSQL — policies that automatically filter every query by tenant_id based on a session variable. No application code needed for tenant filtering. (2) Application-level: every query is parameterized with tenant_id from the JWT, enforced in a service layer. (3) Schema-per-tenant (like Instagram's approach) — each tenant is a separate PostgreSQL schema. True isolation but harder to query across tenants. For applyGOAT at current scale I'd use RLS + tenant_id on every relevant table with a composite index (tenant_id, ...) as the first column. This gives isolation without schema sprawl."
Gap to watch Candidates often recite the normal form definitions without anchoring them to the anomalies they prevent. Interviewers want to hear: "2NF prevents update anomalies from partial key dependencies" not just "2NF means every column depends on the whole key." The why matters more than the rule.
