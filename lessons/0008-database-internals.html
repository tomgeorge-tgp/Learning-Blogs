Database Internals

Database Internals

MVCC, WAL, VACUUM, isolation levels — how PostgreSQL actually guarantees the ACID properties your application relies on.

ACID — The Contract

Every relational database transaction comes with four guarantees. These are not abstract — each is implemented by specific internal mechanisms.

PropertyGuaranteeImplemented by
Atomicity A transaction either fully commits or fully rolls back. No partial writes. WAL: if the database crashes mid-transaction, the WAL allows rollback on recovery.
Consistency A transaction moves the database from one valid state to another, respecting all constraints. Constraint checking (FK, UNIQUE, CHECK) before commit. Triggers.
Isolation Concurrent transactions don't interfere with each other in unintended ways. MVCC (Multi-Version Concurrency Control). Locks for write-write conflicts.
Durability A committed transaction survives crashes. Data is permanently stored. WAL: the write is flushed to the WAL before the commit is acknowledged.

MVCC — How PostgreSQL Handles Concurrent Reads and Writes

The key design decision in PostgreSQL's concurrency model: readers never block writers and writers never block readers. This is achieved through Multi-Version Concurrency Control (MVCC).

How it works

PostgreSQL never overwrites a row in place. When you UPDATE a row, it:

  1. Marks the old row version as dead (sets its xmax — the transaction ID that expired it)
  2. Inserts a new row version with the updated values (new xmin — the transaction ID that created it)
Time ──────────────────────────────────────────► TXN 100 starts: SELECT * FROM users WHERE id = 1 TXN 101 starts: UPDATE users SET name = 'Thomas' WHERE id = 1 Heap page for users(id=1): Version A: xmin=50, xmax=101, name='Tom' ↑ visible to TXN 100 (started before 101 committed) ↑ NOT visible to new transactions after 101 commits Version B: xmin=101, xmax=∞, name='Thomas' ↑ NOT visible to TXN 100 (started before 101) ↑ visible to all new transactions after 101 commits TXN 100: still reads 'Tom' (consistent snapshot) TXN 101: commits → new transactions read 'Thomas' TXN 100: reads 'Tom' until it commits or rolls back

Each transaction sees a consistent snapshot of the database as it was at its start time (or at the start of each statement, depending on isolation level). Dead row versions accumulate until VACUUM reclaims them.

Why this matters for your app Long-running transactions in applyGOAT (e.g. a data export that takes 5 minutes) hold a snapshot. All the row versions that were visible when it started are kept alive — even if newer transactions have replaced them. This causes table bloat. Keep transactions short. Don't hold transactions open across HTTP requests or AI generation calls.

WAL — Write-Ahead Log (Durability + Replication)

The WAL is the mechanism behind both Atomicity and Durability. The rule is simple: write to the WAL before writing to the actual data files.

Every data change (INSERT, UPDATE, DELETE) produces a WAL record describing the change. This is flushed to disk (fsync) before the commit is acknowledged to the client. If the server crashes after a commit but before the heap files are written, the WAL records allow full recovery on restart.

WAL and replication

Read replicas work by tailing the WAL stream from the primary. The replica applies the same WAL records in order, keeping its state in sync. This is covered in depth in Lesson 03 (Database Scaling).

-- Check current WAL position
SELECT pg_current_wal_lsn();  -- Log Sequence Number

-- Check how far a replica is behind
SELECT
  application_name,
  state,
  sent_lsn,
  write_lsn,
  flush_lsn,
  replay_lsn,
  (sent_lsn - replay_lsn) AS replication_lag_bytes
FROM pg_stat_replication;

Transaction Isolation Levels

Isolation is not binary. PostgreSQL supports four isolation levels, each permitting or preventing specific read phenomena. The higher the isolation, the fewer anomalies — but more lock contention and potential performance cost.

The three read phenomena

PhenomenonWhat happensExample
Dirty Read Read uncommitted data from another transaction that hasn't committed yet (may still roll back). TXN A reads a balance updated by TXN B. TXN B rolls back. TXN A saw data that never existed.
Non-Repeatable Read Read the same row twice in a transaction, get different values because another transaction committed between the reads. TXN A reads user plan = 'free'. TXN B upgrades user to 'premium' and commits. TXN A reads again: 'premium'. Same row, different value in same transaction.
Phantom Read Run the same query twice in a transaction, get different sets of rows because another transaction inserted or deleted rows between the reads. TXN A: SELECT COUNT(*) = 10. TXN B inserts a row and commits. TXN A: SELECT COUNT(*) = 11. New row "appeared."

Isolation levels and the phenomena they prevent

Isolation LevelDirty ReadNon-Repeatable ReadPhantom ReadPostgreSQL default?
READ UNCOMMITTED Possible Possible Possible No (PostgreSQL treats it as READ COMMITTED)
READ COMMITTED Prevented Possible Possible Yes — the default
REPEATABLE READ Prevented Prevented Prevented* No — opt in
SERIALIZABLE Prevented Prevented Prevented No — opt in

* PostgreSQL's MVCC implementation prevents phantom reads at REPEATABLE READ — stronger than the SQL standard requires.

-- Set isolation level for a specific transaction
BEGIN;
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
-- ... your queries
COMMIT;

-- Or per session
SET default_transaction_isolation = 'repeatable read';
READ COMMITTED is PostgreSQL's default — what it means for you At READ COMMITTED, each statement in a transaction sees a fresh snapshot of committed data at the moment that statement starts. This means if TXN B commits between two SELECT statements in your TXN A, you'll see TXN B's changes in the second SELECT. For most applyGOAT operations this is fine. For financial operations (balance checks + deductions), use REPEATABLE READ or SERIALIZABLE plus SELECT FOR UPDATE.

Locks

MVCC handles read-write conflicts without locks. Write-write conflicts still need locking.

Row-level locks

-- Lock a row for update — prevents other transactions from modifying it
BEGIN;
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;
-- Other transactions trying to UPDATE this row will wait here
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;

FOR UPDATE is the standard pattern for "read-then-update" operations where correctness depends on the value you read still being current when you write. Without it, two concurrent transactions could both read the same balance and each subtract from it, resulting in a total subtraction of only one of the amounts.

FOR UPDATE vs FOR SHARE

Lock typeBlocksAllowsUse case
FOR UPDATE Other writers, other FOR UPDATE/SHARE Reads (MVCC) You intend to update the row
FOR SHARE Writers Other FOR SHARE readers You need the row to not change while you validate it (FK checks)
FOR NO KEY UPDATE Writers, FOR UPDATE/SHARE FOR NO KEY UPDATE from others Update a row without affecting FK referencing rows

Advisory locks

-- Application-level locks using PostgreSQL as the coordinator
-- Useful for preventing duplicate processing of the same job

SELECT pg_try_advisory_lock(12345);  -- non-blocking: returns true/false
SELECT pg_advisory_lock(12345);      -- blocking: waits until lock available
SELECT pg_advisory_unlock(12345);    -- release

-- Example: ensure only one worker processes a specific job
async def process_job(job_id: int):
    lock_acquired = await db.fetchval(
        "SELECT pg_try_advisory_lock($1)", job_id
    )
    if not lock_acquired:
        return  # another worker is handling it
    try:
        await do_work(job_id)
    finally:
        await db.execute("SELECT pg_advisory_unlock($1)", job_id)

Deadlocks

A deadlock occurs when TXN A holds a lock TXN B needs, and TXN B holds a lock TXN A needs. Both wait forever. PostgreSQL detects deadlocks automatically and kills one transaction with an error.

-- Classic deadlock pattern
TXN A: UPDATE accounts SET balance = balance - 100 WHERE id = 1;  -- locks row 1
TXN B: UPDATE accounts SET balance = balance - 50  WHERE id = 2;  -- locks row 2
TXN A: UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- waits for B
TXN B: UPDATE accounts SET balance = balance + 50  WHERE id = 1;  -- waits for A → DEADLOCK

-- Prevention: always lock resources in the same order
-- Both transactions should lock row 1 before row 2
Deadlock in FastAPI If your FastAPI endpoints run multiple UPDATEs in a transaction, ensure any endpoint that updates multiple rows always touches them in the same order (e.g. always by ascending user_id). This is the standard deadlock prevention technique for application code.

VACUUM and Table Bloat

MVCC creates dead row versions on every UPDATE and DELETE. These dead tuples take up space and slow down sequential scans (the scanner must check each tuple's visibility). VACUUM reclaims this space.

Autovacuum

PostgreSQL runs autovacuum automatically in the background. For most tables this is sufficient. Problems arise when:

  • Very high write/update rate: autovacuum can't keep up, bloat grows faster than it's reclaimed.
  • Long-running transactions: autovacuum cannot reclaim dead tuples that are still visible to the oldest running transaction.
  • Large bulk updates: updating millions of rows creates millions of dead tuples at once — autovacuum may lag for hours.
-- Check table bloat (dead tuples)
SELECT
  relname,
  n_live_tup,
  n_dead_tup,
  ROUND(n_dead_tup::numeric / NULLIF(n_live_tup + n_dead_tup, 0) * 100, 2) AS dead_pct,
  last_autovacuum,
  last_autoanalyze
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC;

-- Manually vacuum a specific table (without locking)
VACUUM applications;

-- Full vacuum: reclaims disk space, requires exclusive lock — use in maintenance windows only
VACUUM FULL applications;
Transaction ID wraparound — the silent disaster PostgreSQL transaction IDs are 32-bit integers (max ~2.1 billion). If a table goes 2 billion transactions without a VACUUM, its old row versions could appear "newer" than current ones — causing data corruption. PostgreSQL has a forced VACUUM mechanism to prevent this but it can cause downtime if it triggers unexpectedly. Monitor pg_database.datfrozenxid on production instances.

The Buffer Pool (shared_buffers)

PostgreSQL maintains a shared memory area called the buffer pool (shared_buffers, default 128MB). All reads and writes go through this buffer. Hot pages stay in memory; cold pages are evicted when needed.

The latency cliff from Lesson 03 (Database Scaling): when your working set fits in shared_buffers, reads are RAM-speed (~100ns). When it overflows to disk, you pay the NVMe/HDD latency per miss.

-- Recommended: set shared_buffers to 25% of total RAM
-- For a 16GB server: shared_buffers = 4GB

-- Check cache hit ratio (should be > 99% for a well-tuned instance)
SELECT
  SUM(heap_blks_hit) AS cache_hits,
  SUM(heap_blks_read) AS disk_reads,
  ROUND(
    SUM(heap_blks_hit)::numeric /
    NULLIF(SUM(heap_blks_hit) + SUM(heap_blks_read), 0) * 100, 2
  ) AS cache_hit_pct
FROM pg_statio_user_tables;

If cache_hit_pct is below 95%, either add RAM (increase shared_buffers) or investigate which queries are causing excessive disk I/O.

Interview Playbook

Click to reveal answers.

// LIKELY INTERVIEW QUESTIONS — click to reveal answers
Model answer
"MVCC — Multi-Version Concurrency Control. When a row is updated, PostgreSQL doesn't overwrite it in place. Instead it marks the old version as expired and writes a new version. Each transaction holds a snapshot — it sees only the row versions that were committed before the snapshot was taken. A reader reading old version A doesn't block a writer creating new version B, and the writer doesn't block readers from continuing to read A. They operate on different row versions simultaneously."
Follow-up answer
"The downside is dead tuple accumulation. Every UPDATE creates a dead row version that accumulates space. High-write tables bloat over time — the physical table file grows and sequential scans must check each dead tuple's visibility. PostgreSQL deals with this via VACUUM: a background process that scans tables and reclaims dead tuple space. Autovacuum runs automatically, but long-running transactions prevent it from reclaiming tuples that are still visible to the oldest transaction snapshot."
Model answer
"Non-repeatable reads. At READ COMMITTED, each statement in a transaction gets a fresh snapshot. If TXN B commits between two SELECTs in TXN A, the second SELECT sees TXN B's data. This matters when: you SELECT a value to use in a conditional, then UPDATE based on that condition — another transaction may have changed the value between your SELECT and your UPDATE. Example: check user's subscription tier, then grant access. If another transaction downgrades the subscription between the check and the access grant, you let an un-authorized operation through."
Follow-up answer
"SERIALIZABLE for operations where the result of your transaction depends on a condition that another concurrent transaction could change — and getting it wrong would be a data correctness bug, not just a stale read. Example in applyGOAT: a user can only have one active free trial. The check-and-insert must be serializable to prevent two concurrent requests both passing the check and both inserting a trial. In FastAPI: wrap the critical section in a SERIALIZABLE transaction, catch the serialization_failure exception, and retry with exponential backoff. Most frameworks have this built in as a retry decorator."
Model answer
"Use SELECT FOR UPDATE inside a transaction. The flow: BEGIN → SELECT balance FROM credits WHERE user_id = X FOR UPDATE → check if balance >= deduction_amount → UPDATE credits SET balance = balance - deduction_amount WHERE user_id = X → COMMIT. The FOR UPDATE locks the row the moment it's read. Any concurrent transaction attempting to SELECT FOR UPDATE the same row will block until the first transaction commits or rolls back. This prevents two transactions from both seeing sufficient balance and both deducting, leaving the balance negative."
Follow-up answer
"SELECT FOR UPDATE is a database-level lock — durable across database restarts, works across multiple application servers, and is released automatically if the application crashes (transaction rolls back). An application-level mutex (asyncio.Lock, Redis distributed lock) only works within one process or one Redis instance, and if the app crashes holding the lock you may have a stuck lock. For anything involving financial data or correctness guarantees, use the database lock — it's part of the same transaction as your data change. Use application-level mutexes only for coordinating non-database work."
Gap to watch Candidates often know ACID exists but can't explain how each property is implemented. The strong answer ties each property to its mechanism: Atomicity → WAL rollback, Isolation → MVCC snapshots, Durability → WAL fsync before commit. Knowing that MVCC creates dead tuples and that VACUUM reclaims them is a senior-level signal.
