Beyond Relational

NoSQL Databases

MongoDB, Redis, Cassandra — what each is for, how data is modeled, and when to reach for each over PostgreSQL.

Press F to toggle

Why NoSQL Exists

NoSQL databases weren't invented to replace SQL. They were invented to solve specific problems that relational databases handle poorly at scale:

  • Schema flexibility: Documents with varying shapes stored without migrations
  • Horizontal write scale: Relational primaries have one WAL; NoSQL databases like Cassandra distribute writes by design
  • Specific access patterns: Key-value lookups (Redis), graph traversals (Neo4j), time-series (InfluxDB) are awkward in SQL
  • Operational simplicity for specific shapes: Storing and retrieving a JSON blob is simpler in MongoDB than mapping it to a relational schema
The right framing NoSQL is not "better than SQL." PostgreSQL with JSONB handles semi-structured data. PostgreSQL with proper indexing handles most scale scenarios. Choose NoSQL when a specific database's model is a materially better fit for your access pattern — not because it's fashionable or "scales better."

Document Store

MongoDB

The data model

Data is stored as JSON-like documents (BSON internally) in collections. No fixed schema per collection — different documents can have different fields. Documents can nest arrays and sub-documents.

// A MongoDB application document (flexible shape)
{
  "_id": ObjectId("..."),
  "user_id": "abc-123",
  "job": {
    "title": "Backend Engineer",
    "company": "Stripe",
    "location": "Remote"
  },
  "status": "applied",
  "ai_analysis": {
    "match_score": 87,
    "gaps": ["system design", "Kubernetes"],
    "highlights": ["FastAPI", "PostgreSQL", "Docker"]
  },
  "notes": "Strong fit for their infra team",
  "applied_at": ISODate("2024-06-01")
}

Embedding vs Referencing — the critical design decision

In MongoDB, you decide at schema design time whether related data is embedded in the same document or stored separately and referenced by ID. This decision is permanent and shapes your entire query pattern.

ApproachWhen to useAdvantageDisadvantage
Embed
(nested sub-document)		 Data that is always queried together. Data with 1:1 or bounded 1:few relationship. Data that doesn't need to be independently queried. Single read to get everything. Atomic update of parent + embedded data. Document can grow unbounded (1:many with high cardinality). Embedded data can't be queried in isolation efficiently.
Reference
(store an ID, join in app)		 Data queried independently. Data with 1:many (large) or many:many relationships. Data shared across documents. Avoids document bloat. Can query embedded entity independently. Requires multiple queries or $lookup (aggregation pipeline join). No atomic update across documents without a transaction.
Rule of thumb Embed when: "I always read this together" and the nested array is bounded. Reference when: the related data grows unboundedly (e.g. a user's message history), or it's queried independently, or it's shared across many parents.

Aggregation pipeline

// MongoDB's equivalent of SQL GROUP BY + JOIN
db.applications.aggregate([
  // Stage 1: filter
  { $match: { tenant_id: "goat-001", status: "applied" } },

  // Stage 2: join with users (equivalent of LEFT JOIN)
  { $lookup: {
      from: "users",
      localField: "user_id",
      foreignField: "_id",
      as: "user"
  }},

  // Stage 3: unwind the joined array
  { $unwind: "$user" },

  // Stage 4: group and aggregate
  { $group: {
      _id: "$user.plan",
      total_applications: { $sum: 1 },
      avg_score: { $avg: "$ai_analysis.match_score" }
  }},

  // Stage 5: sort
  { $sort: { total_applications: -1 } }
]);

When MongoDB over PostgreSQL?

  • Documents genuinely vary in shape across records (e.g. e-commerce product catalog — a shirt and a laptop have completely different attribute sets)
  • Rapid schema evolution without migrations (prototyping)
  • Data that is naturally document-shaped and always accessed as a unit

When NOT to choose MongoDB: You need strong ACID across multiple documents, complex cross-collection joins are common, or the data is naturally relational. PostgreSQL with JSONB columns handles semi-structured data while retaining full SQL and ACID.

Key-Value Store

Redis

The data model

Redis is an in-memory key-value store with a rich set of data structures. Every key maps to one of: String, List, Hash, Set, Sorted Set, Stream, and more. Operations are atomic.

Data structureUse caseKey commands
String Cache any serialized value. Counters. Feature flags. Session tokens. GET, SET, INCR, EXPIRE
Hash Store an object's fields without serializing the whole thing. Update individual fields. HGET, HSET, HMGET, HGETALL
List Message queues. Recent activity feeds. Task queues. LPUSH, RPOP, LRANGE, BLPOP
Set Unique item tracking. Tags. Union/intersection queries. SADD, SMEMBERS, SINTERSTORE, SUNIONSTORE
Sorted Set (ZSet) Leaderboards. Rate limiting. Time-sorted feeds. ZADD, ZRANGE, ZRANK, ZRANGEBYSCORE
Stream Append-only log. Event sourcing. Pub/sub with persistence. XADD, XREAD, XGROUP 
# applyGOAT: caching AI-generated cover letter
import json

async def get_cover_letter(user_id: str, job_id: str) -> str:
    cache_key = f"cover_letter:{user_id}:{job_id}"

    # Check cache
    cached = await redis.get(cache_key)
    if cached:
        return json.loads(cached)

    # Generate (expensive)
    result = await ai_service.generate_cover_letter(user_id, job_id)

    # Store with TTL
    await redis.setex(cache_key, 3600, json.dumps(result))
    return result

# Rate limiting with Redis sorted set
async def is_rate_limited(user_id: str) -> bool:
    key = f"ratelimit:{user_id}"
    now = time.time()
    window = 60  # 1 minute

    pipe = redis.pipeline()
    pipe.zremrangebyscore(key, 0, now - window)  # remove old entries
    pipe.zadd(key, {str(now): now})              # add current request
    pipe.zcard(key)                               # count in window
    pipe.expire(key, window)
    results = await pipe.execute()
    return results[2] > 100  # 100 requests per minute limit

AOF (append-only file) Log every write command. Replay on restart. Configurable: every second, or every write Slower (replay log)
RDB + AOF Both. AOF provides durability, RDB provides fast restart. Near-zero data loss Moderate

applyGOAT use For caching AI outputs: no persistence (cache-aside, warm on miss). For rate limiting and session state: AOF + every-second fsync (small data, high durability need). For job queues (AI generation tasks): AOF to avoid losing queued jobs on restart.
Wide-Column Store

Cassandra / ScyllaDB

The data model

Cassandra organizes data in tables with a mandatory partition key and optional clustering columns. The partition key determines which node stores the data. Clustering columns determine the sort order within a partition.

-- Cassandra schema for Discord-style message storage
CREATE TABLE messages (
  channel_id   UUID,           -- Partition key: all messages for a channel on one node
  message_id   TIMEUUID,       -- Clustering key: sorted chronologically within partition
  user_id      UUID,
  content      TEXT,
  created_at   TIMESTAMP,
  PRIMARY KEY (channel_id, message_id)
) WITH CLUSTERING ORDER BY (message_id DESC);

-- This table is optimized for ONE query:
-- "Get the last N messages in channel X"
SELECT * FROM messages
WHERE channel_id = ? LIMIT 100;

Design rule: model your tables for your queries

In SQL, normalize first, query with JOINs. In Cassandra, design your table for each specific query pattern. Denormalization is intentional — you often create multiple tables for the same data to support different access patterns.

ConceptWhat it doesDesign constraint
Partition key Routes data to the correct node via consistent hashing. All rows with the same partition key live on the same node(s). Choose a partition key with high cardinality and even distribution. Low-cardinality keys = hot spots. Avoid partition keys that grow unboundedly (a single partition can't exceed ~100MB).
Clustering key Determines sort order within a partition. You can range-scan on clustering columns. You can only efficiently filter on clustering columns in prefix order (same as SQL composite index left-prefix rule).
Tombstones DELETEs don't remove data — they write a tombstone marker. The old data is removed at compaction. Heavy delete workloads create tombstone accumulation, which slows reads (the scanner must check each tombstone's visibility). Avoid delete-heavy patterns; use TTL instead.
Cassandra is not a general-purpose database Cassandra has no JOINs, no subqueries, limited aggregation. You cannot query by non-key columns without allowing full partition scans (ALLOW FILTERING) which are catastrophically slow. Every query pattern needs its own table. It's purpose-built for: high write throughput, large datasets, geographically distributed write availability. If you don't need these, PostgreSQL is the better choice.

Highly variable document shapes ✓ JSONB columns ✓ Native document model Not a fit Fixed schema

Cross-project check — Mini Hermes Agent Mini Hermes's persistent memory layer is a good self-test for this whole decision tree. The agent's long-term memory records are heterogeneous by nature — a "learned fact," a "tool-call outcome," and a "user preference" all have different shapes, and new memory kinds get added as the agent evolves — so a rigid relational schema fights you on every addition. That's the MongoDB branch: flexible document schema, no migration required to introduce a new memory type. Layer a Redis tier on top for the agent's active working-memory window — the last N turns and any short-TTL scratch state a single reasoning loop needs at sub-millisecond latency. Two different access patterns inside one agent, two different NoSQL choices — exactly the "name the constraint, name the fit" discipline this decision tree is testing for.

Interview Playbook

// LIKELY INTERVIEW QUESTIONS — click to reveal answers
Model answer
"The main case is genuinely variable document shapes where a rigid relational schema is a poor fit — for example, a product catalog with hundreds of different product types each having completely different attributes. MongoDB lets you store each product type naturally without nullable columns or a complex EAV schema. A secondary case is when schema is evolving rapidly in early product development and you want to avoid migrations. That said, PostgreSQL with JSONB handles most semi-structured data cases while giving you full SQL and stronger ACID guarantees. I'd reach for MongoDB only when the document model is a materially better fit, not out of habit."
Follow-up answer
"Since MongoDB 4.0, yes — multi-document ACID transactions are supported. But they come with performance overhead and MongoDB's transaction model is more limited than PostgreSQL's. Single-document operations in MongoDB are always atomic by design (because the document is the unit of atomicity). Cross-collection transactions work but add latency. For systems where strong ACID across many operations is the primary requirement, PostgreSQL remains the cleaner choice. MongoDB's transaction support is 'good enough' for many use cases, not a direct replacement for Postgres's transactional model."
Model answer
"Redis Sorted Set (ZSet) — designed exactly for this. Each member is a user_id, score is their score. Updates are O(log n): ZADD leaderboard 1250 'user:123'. Range queries by score or rank are O(log n + k). The data structure maintains sorted order automatically on every update — no re-sorting needed. This handles thousands of score updates per second at sub-millisecond latency. Store in Redis as the primary for leaderboard reads, persist to PostgreSQL asynchronously for durability."
Follow-up answer
"Top 10: ZREVRANGE leaderboard 0 9 WITHSCORES — returns top 10 by score descending, with scores. O(log n + 10). A specific user's rank: ZREVRANK leaderboard 'user:123' — returns 0-based rank from the top. O(log n). Both are very fast even with millions of users in the sorted set. One gotcha: if you need the user's display name, you'll need a second lookup (HGET or a Redis hash for user metadata, or a database join). The sorted set stores IDs, not display names."
Model answer
"Cassandra has no JOINs and can only efficiently query by partition key then clustering columns. If your data is normalized into related tables, you'd need JOINs to assemble a response — impossible in Cassandra. Instead, you design one table per query pattern, denormalizing the data so everything needed for a read lives in the same partition. This means duplicating data across multiple tables. The trade-off: write amplification (update multiple tables) in exchange for fast, predictable reads where the partition key routes you directly to the right nodes in O(1)."
Follow-up answer
"A hot partition: one partition key receiving disproportionately more traffic than others. Example: partition_key = 'status', status = 'active' — all active users hit one partition, one set of nodes. Prevention: choose high-cardinality partition keys (user_id, channel_id, tenant_id — things with many unique values). Avoid low-cardinality keys (status, country, date) as standalone partition keys. If you need to query by status, create a secondary index or a separate table with a composite partition key (status, user_id bucket) that spreads the data. For a 'celebrity problem' (one user_id getting millions of reads), add a random suffix: partition_key = user_id + '_' + RANDOM(0-9) spreads across 10 partitions."
Gap to watch Candidates choose NoSQL "for scale" without specifying what kind of scale (read, write, data volume, distribution) and without naming what they'd lose (JOINs, ACID, flexible querying). Strong answer: name the specific constraint that makes the SQL model a poor fit, name the NoSQL model that fits it, and name what you're giving up.
