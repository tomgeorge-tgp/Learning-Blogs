Vector Databases

Vector Databases

Embeddings, ANN search, HNSW, IVF — the infrastructure behind every RAG pipeline you build.

What is an Embedding?

An embedding is a dense numerical vector that encodes the semantic meaning of content. Texts, images, code, and audio can all be embedded. Things that are semantically similar end up close together in the vector space.

Text: "Software engineer with Python and FastAPI experience" Embedding: [0.021, -0.184, 0.337, ..., 0.092] ← 1536 floats (OpenAI text-embedding-3-small) Text: "Backend developer skilled in Python web frameworks" Embedding: [0.019, -0.179, 0.341, ..., 0.088] ← very close in 1536-dimensional space Text: "Professional chef specializing in Italian cuisine" Embedding: [0.412, 0.093, -0.228, ..., -0.301] ← far away in vector space Similarity measure: cosine similarity cos_sim(engineer, backend_dev) ≈ 0.94 ← very similar cos_sim(engineer, chef) ≈ 0.12 ← unrelated

This is why RAG (Retrieval Augmented Generation) works: you embed a user's query, find the document chunks whose embeddings are closest, and feed those as context to the LLM. The retrieval is semantic, not keyword-based.

Why Not a B-Tree for Vector Search?

B-Trees work because values have a natural total order — integers, strings, dates. You can compare any two values and say "A < B." This is what makes range scans possible: walk left in the tree, walk right, done.

Vectors in 1536 dimensions have no total order. You can't say vector A is "less than" vector B in any meaningful sense. And the curse of dimensionality makes tree-based indexing degenerate:

2D
B-tree works fine. Sorted order clear.
100D
Tree pruning becomes ineffective — most branches must be explored.
1536D
All vectors are approximately equidistant. Tree degenerates to linear scan.

In high-dimensional space, the ratio of the nearest to farthest neighbor approaches 1 — everything is equally "far." The tree's ability to prune branches disappears. Exact nearest neighbor becomes O(n) — a linear scan over every vector. With 10 million 1536-dimensional vectors, that's unusable.

The solution: Approximate Nearest Neighbor (ANN) algorithms. Trade a small accuracy loss (~1–5%) for O(log n) or sub-linear search. For RAG, this trade-off is completely acceptable — semantic retrieval doesn't need mathematical exactness.

HNSW — Hierarchical Navigable Small World

The dominant ANN algorithm used by Pinecone, pgvector, Weaviate, Qdrant, and FAISS. Builds a multi-layer graph of vectors. Navigation is greedy — always move to the neighbor closest to the query.

Layer 2 (very sparse — "highway"): A ─────────────────── F ─────────────── K │ Layer 1 (moderately connected): B ── C ── D ── E ── F ── G ── H ── I ── J ── K Layer 0 (fully connected — "local neighborhood"): All vectors, dense local edges, precise search Query vector Q enters at Layer 2 (sparse, fast navigation) Step 1: Start at random node. Greedy: move to nearest neighbor to Q. Step 2: Descend to Layer 1 at current position. Greedy again. Step 3: Descend to Layer 0. Explore neighborhood to find top-k nearest. Result: O(log n) traversal to approximate nearest neighbors.

Key HNSW parameters

ParameterWhat it controlsHigher value →
M Number of edges per node in the graph. Controls graph connectivity. Better recall, more memory, slower index build. Typical: 16–64.
ef_construction Search breadth during index build — how many candidates considered when inserting a new vector. Better recall at query time, much slower index build. Typical: 100–400.
ef_search Search breadth at query time — how many candidates explored during search. Better recall, higher query latency. Tune per use case.

The recall-latency dial: Increase ef_search to get closer to 100% recall (exact result) at the cost of slower queries. Decrease it for faster queries at ~95% recall. Pinecone exposes this as the top_k and internal beam width.

Where does cosine similarity fit inside HNSW?

HNSW and cosine similarity are two different layers that work together. A common point of confusion: HNSW is the navigation strategy — it decides which vectors to even look at. Cosine similarity is the ruler used at every single comparison step during that navigation.

What it isRole
HNSWMulti-layer graph navigation algorithmDecides which vectors to compare — skips most of the index
Cosine similarityDistance metric: dot product of unit vectorsThe actual measurement at each comparison — "how close is this node to the query?"
Query vector arrives ↓ HNSW: "start at top layer, pick entry point" ↓ At each step: cosine_sim(query, neighbor_A) = 0.87 cosine_sim(query, neighbor_B) = 0.91 ← highest cosine_sim(query, neighbor_C) = 0.74 ↓ "Move to neighbor_B" ← greedy step uses cosine similarity to decide ↓ Descend a layer, repeat with local neighbors ↓ Layer 0: sweep local neighborhood using cosine similarity ↓ Return top-k by cosine similarity score

Without HNSW: you'd compute cosine similarity against every vector in the index — O(n). At 10M vectors that's unusable.
Without cosine similarity: HNSW has no way to know which direction to move.
HNSW reduces the number of cosine comparisons from O(n) to O(log n) by intelligently skipping most of the space.

In Pinecone, you declare which metric to use when creating the index (metric="cosine"). That tells HNSW what ruler to use at each graph step. The top_k results come back already ranked by cosine similarity score (1.0 = identical, 0.0 = orthogonal).

applyGOAT context When your RAG pipeline calls index.query(vector=query_embedding, top_k=5), HNSW navigates the graph in ~dozens of cosine comparisons to find the 5 best chunks — instead of computing cosine similarity against all 500K resume chunks. HNSW is what makes this O(log n) fast. Cosine similarity is what defines "best."

Common misconception — the top layer is NOT a summary

Intuition says: "the top layer must be some kind of summary or centroid of the vectors below it." This is wrong. The top layers contain the same vectors as Layer 0 — just a random sparse subset of them, with longer-range connections.

When a vector is inserted into HNSW, a random number is rolled to determine how many layers it appears in:

Most vectors:  Layer 0 only                (~60% probability)
Some vectors:  Layer 0 + Layer 1           (~25% probability)
Few vectors:   Layer 0 + Layer 1 + Layer 2 (~10% probability)
Very few:      All layers                  (~5% probability)

The same vector, just promoted to higher layers by random chance — like how some people are in a local group chat AND a national committee. They're the same person, not a representative of the group.

Why does a random sparse sample still work?

  • You don't need to start at the semantically "closest summary" — you just need to start somewhere not terrible and refine from there
  • In a well-connected graph, a random sparse sample gives you fast "highway" jumps across the space
  • The multi-layer structure guarantees convergence to the right neighborhood by Layer 0 regardless of where you started

The analogy that actually holds: a skip list. A skip list doesn't summarize lower levels — it randomly promotes some nodes to an express lane so you can skip large sections during search. HNSW is the same idea but in graph form across high-dimensional space.

IVF does use summaries — don't confuse them IVF (below) does use centroids — cluster centers computed by k-means that genuinely summarize groups of nearby vectors. That's the key structural difference between HNSW and IVF. HNSW: random sparse promotion. IVF: trained centroid summaries.

IVF — Inverted File Index

A different ANN algorithm, complementary to HNSW. Used by FAISS (Facebook AI Similarity Search) and pgvector (as IVFFlat).

How it works:

  1. During index build: run k-means clustering. Divide all vectors into N clusters (centroids).
  2. During query: compute distance from the query to each centroid. Search only the closest nprobe clusters.
  3. Within those clusters: linear scan of candidate vectors.
Training (index build): All 1M vectors → k-means clustering → 1000 centroids Query: Distance from Q to all 1000 centroids → pick nearest nprobe=10 centroids Linear scan of ~10,000 vectors (10 clusters × 1000 vectors each) vs. scanning 1,000,000 vectors for exact search → 100× speedup Trade-off: nprobe=1: fast, low recall (might miss the true nearest neighbor) nprobe=100: slower, high recall (searches 10% of all vectors) nprobe=1000: = full scan, 100% recall, no benefit from index

HNSW vs IVF

HNSWIVF
Query speedFaster at same recallFaster with GPU (FAISS+CUDA)
MemoryHigher (graph edges)Lower (centroids + cluster assignments)
Index buildSlow, incremental insert possibleRequires training phase, then fast bulk insert
Dynamic updatesYes — insert/delete without rebuildDifficult — retraining needed for significant data changes

Pinecone — Managed Vector Database

What you're using in applyGOAT and AI Sales Agent. Managed HNSW under the hood.

from pinecone import Pinecone

pc = Pinecone(api_key="...")
index = pc.Index("applygoat-resumes")

# Upsert vectors (embed first, then store)
index.upsert(vectors=[
    {
        "id": "resume:user123:chunk0",
        "values": embedding_vector,           # 1536 floats
        "metadata": {
            "user_id": "user123",
            "tenant_id": "goat-001",
            "chunk_index": 0,
            "text": "Software engineer with 4 years Python experience...",
            "created_at": "2024-06-01"
        }
    }
])

# Query — semantic search
results = index.query(
    vector=query_embedding,
    top_k=5,
    filter={"tenant_id": {"$eq": "goat-001"}},   # metadata filter
    include_metadata=True
)
for match in results.matches:
    print(match.score, match.metadata["text"])

Pinecone architecture concepts

ConceptWhat it isUse in applyGOAT
Namespaces Logical partitions within an index. Queries are scoped to a namespace. Namespace per tenant: namespace="goat-001". Tenant isolation without separate indexes.
Metadata filtering Filter results by key-value metadata attached to vectors. Applied after ANN, not during (in most implementations). filter={"user_id": "abc"} — only return chunks belonging to this user's resume.
Hybrid search Combine dense (embedding) search with sparse (BM25 keyword) search. Pinecone supports this natively. Job description search where exact keyword match AND semantic match both matter.
Eventual consistency Upserted vectors become queryable within seconds, not instantly. After resume upload: wait for embedding + upsert before telling user "search is ready." Or accept the delay and let the user search immediately (new chunks may not appear for a few seconds).

pgvector — Vector Search Inside PostgreSQL

pgvector is a PostgreSQL extension that adds a vector data type and ANN indexes (HNSW and IVFFlat). Lets you do vector search directly in your existing PostgreSQL database without a separate service.

-- Enable extension
CREATE EXTENSION IF NOT EXISTS vector;

-- Add vector column to existing table
ALTER TABLE resumes ADD COLUMN embedding vector(1536);

-- Create HNSW index
CREATE INDEX ON resumes
USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);

-- Semantic search: find 5 most similar resumes to a job description
SELECT id, user_id, LEFT(content, 200)
FROM   resumes
ORDER BY embedding <=> $1   -- <=> = cosine distance operator
LIMIT 5;

-- Hybrid: semantic match + metadata filter in one query
SELECT id, user_id
FROM   resumes
WHERE  tenant_id = 'goat-001'
  AND  created_at > NOW() - INTERVAL '30 days'
ORDER BY embedding <=> $1
LIMIT 10;

applyGOAT recommendation At 20K users, pgvector is more than sufficient and keeps your architecture simpler — one PostgreSQL instance handles both relational data and vector search. Migrate to Pinecone when vector count exceeds ~5M or when query latency under load becomes unacceptable. The migration is straightforward: same embeddings, different client.

Building a RAG Pipeline — The Full Architecture

# Complete RAG pipeline for applyGOAT job matching

## Ingestion (on resume upload)
1. User uploads resume PDF
2. Extract text, chunk by paragraph/section
3. For each chunk: embed via Azure OpenAI text-embedding-3-small
4. Upsert to Pinecone with metadata: {user_id, tenant_id, chunk_index, text}
5. Store resume metadata in PostgreSQL

## Query (on job match request)
1. Receive job description from user
2. Embed job description: query_vector = embed(job_description)
3. Query Pinecone: top_k=10 most similar resume chunks, filter by tenant_id
4. Fetch full context: retrieve matched chunks' text from metadata
5. Construct LLM prompt:
     System: "You are a career advisor..."
     User: f"Job description: {job_desc}\n\nRelevant resume sections:\n{chunks_text}\n\nAssess match and gaps."
6. Stream response via Azure OpenAI
7. Cache result in Redis: key = hash(user_id + job_id), TTL = 1 hour

Chunking strategy matters more than most engineers realize

StrategyChunk sizeProCon
Fixed-size 512 tokens, 50-token overlap Simple, predictable May split semantically coherent sections mid-sentence
Semantic/section Variable, by paragraph/heading Chunks contain coherent ideas. Better embedding quality. Variable sizes, harder to manage
Sentence-level 1–5 sentences Very precise retrieval Too granular — loses context. LLM gets insufficient text per chunk.

Recommendation: section-level chunks for resumes (each section = one chunk), with 100-token overlap to preserve cross-section context. Embed the chunk title + body together for better signal.

// LIKELY INTERVIEW QUESTIONS — click to reveal answers
Model answer
"A vector database stores dense numerical vectors (embeddings) and indexes them for similarity search — finding the k vectors closest to a query vector by some distance metric (cosine similarity, Euclidean distance). Traditional B-tree indexes require a total ordering of values — you compare A and B and one is smaller. Vectors have no such order. Finding the nearest vector requires comparing the query to all stored vectors (O(n)) unless you use a specialized data structure like HNSW or IVF that approximates the result in O(log n) time."
Follow-up answer
"The curse of dimensionality: in high-dimensional space, all vectors become approximately equidistant from each other. The ratio of the distance to the nearest neighbor vs. the farthest neighbor approaches 1 as dimensions increase. This means the 'nearest' neighbor isn't much closer than any random vector — the notion of proximity breaks down. Tree-based indexes exploit the clustering of nearby data points to prune large sections of the search space. When everything is equally far, there's nothing to prune — the tree degenerates to a linear scan. This is why specialized ANN algorithms (HNSW, IVF) are needed for vectors above ~20 dimensions."
Model answer
"HNSW builds a multi-layer graph. Upper layers are sparse with long-range connections — like a highway network for fast coarse navigation. Lower layers are denser with local connections for precision. Search starts at the top layer: greedily move to the neighbor closest to the query. Descend to the next layer at the current position. Repeat until reaching Layer 0 where the local neighborhood is searched exhaustively. The hierarchical structure means you navigate across the full space quickly at the top, then zoom in for precision at the bottom. This gives O(log n) average complexity — similar to how you navigate a city: highway (sparse) → city road (medium) → street (dense)."
Follow-up answer
"Approximate means HNSW may miss the mathematically closest vector by a tiny margin — typically 95–99% recall depending on the ef_search parameter. For a RAG pipeline: not a problem in practice. The LLM receives semantically similar chunks, and semantic retrieval doesn't need mathematical exactness. If the true nearest neighbor has cosine similarity 0.94 and HNSW returns a chunk with 0.92, the LLM gets useful context either way. Where approximate retrieval is problematic: exact deduplication (need the exact nearest), fraud detection (missing a 0.2% closer match has business impact), or medical/legal where missing a specific document creates liability. For conversational RAG: ANN is fine."
Model answer
"At 5M vectors I'd start with pgvector — it handles this scale comfortably on a well-provisioned PostgreSQL node, and you get ACID consistency and SQL JOINs (can join vector results directly with user/tenant tables). Schema: ALTER TABLE resume_chunks ADD COLUMN embedding vector(1536). HNSW index with m=16, ef_construction=128. At query time: embed the job description, query resume_chunks ORDER BY embedding <=> $query LIMIT 10 WHERE tenant_id = $tenant AND user_id = $user. Cache frequent queries in Redis. If traffic grows past ~5M vectors or query latency degrades, migrate to Pinecone — same embeddings, new client. Use namespaces for tenant isolation."
Follow-up answer
"Pinecone is eventually consistent — upserted vectors become queryable within seconds, not instantly. Two strategies: (1) Optimistic UI — tell the user 'your resume is being analyzed' and let them trigger the job match after ~5 seconds. By then, chunks are indexed. (2) Track index state in PostgreSQL — after upsert, store embedding_status = 'indexed' with a timestamp. Disable job matching for that resume until the status is updated (poll or wait for a webhook if Pinecone adds one). For the AI Sales Agent use case where a policy document is uploaded: background task embeds + upserts, marks document as 'ready,' then surfaces it in the UI. Don't let users query unindexed documents."
Gap to watch Most candidates know vector databases exist and that they use them via Pinecone. The senior differentiator: explaining WHY B-trees fail in high dimensions (curse of dimensionality), what HNSW actually does (navigable graph, greedy descent, multi-layer), what the ef_search parameter trades, and when pgvector is a better choice than Pinecone. Know the tool, not just the API.
