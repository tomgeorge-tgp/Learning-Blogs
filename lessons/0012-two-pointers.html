Array & String Patterns

Two Pointers

Two indices moving toward or away from each other — collapsing O(n²) pair-search into O(n).

Press F to toggle

The Precise Definition

Two pointers is a family of algorithms that maintain two indices into one or two sequences and advance them according to a rule that provably eliminates candidates at every step. The brute force for any "find a pair/segment satisfying X" question is a nested loop over all Θ(n²) pairs. Two pointers replaces it with a single pass: each pointer only ever moves forward (or inward), so the total number of pointer movements is bounded by the sequence length — O(n) time, O(1) extra space.

The pattern only earns that speedup when the input has a monotonic property (usually sorted order) that lets one comparison rule out an entire block of pairs at once. Without that property, the pointer-move rule has no proof behind it and the "optimization" is just a wrong answer delivered quickly.

Worked example — Two Sum II, no hashmap

Given a sorted array and a target, find two indices whose values sum to the target. The nested-loop brute force checks all pairs: O(n²). Converging pointers:

function twoSumSorted(arr, target) {
  let left = 0, right = arr.length - 1;
  while (left < right) {                  // strict < : a pair needs two distinct indices
    const sum = arr[left] + arr[right];
    if (sum === target) return [left, right];
    if (sum < target) left++;             // arr[left] can't pair with ANYTHING remaining
    else right--;                          // arr[right] can't pair with ANYTHING remaining
  }
  return null;                             // exhausted: no pair, proven
}

Trace with arr = [2, 7, 11, 15, 19, 24], target = 30:

Why moving a pointer never skips the answer: when sum < target, arr[right] is the largest value still in play. If arr[left] plus the maximum is still short, then arr[left] plus anything smaller is shorter — no pair containing left can ever hit the target. Discarding left discards only proven non-answers. The symmetric argument holds for right when the sum overshoots. Each comparison kills a full row or column of the n×n pair matrix, so ~n comparisons cover all ~n²/2 pairs. That elimination-per-step is the entire O(n²) → O(n) collapse — and it is exactly what evaporates if the array isn't sorted.

Plain English — The applyGOAT Scenario

applyGOAT's matching engine has to answer, per candidate–job pair: which of the job's required skills does this candidate have, and which are gaps? Both lists are normalized to canonical skill IDs and stored sorted. A candidate has a sorted skill list of length n; a job has a sorted requirement list of length m.

The naive version is a nested loop — for each requirement, scan the whole skill list: O(n·m). With ~300 skills against ~40 requirements that's up to 12,000 comparisons for one candidate–job pair; scoring one candidate against 10,000 jobs makes it 120 million. The two-pointer version walks both sorted lists once — O(n+m), at most ~340 comparisons per pair:

function skillGapAnalysis(skills, reqs) {   // both sorted by canonical skill ID
  let i = 0, j = 0;
  const matched = [], gaps = [];
  while (i < skills.length && j < reqs.length) {
    if (skills[i] === reqs[j])      { matched.push(reqs[j]); i++; j++; }
    else if (skills[i] < reqs[j])   { i++; }        // candidate skill no job needs — skip it
    else                            { gaps.push(reqs[j]); j++; }  // requirement the candidate lacks
  }
  while (j < reqs.length) gaps.push(reqs[j++]);      // requirements past the last skill
  return { matched, gaps };
}

Walk it: skills [docker, fastapi, postgres, react] vs requirements [fastapi, kubernetes, postgres].

  • docker < fastapi → the job never asks for docker; advance i. Safe because both lists are sorted: every requirement from j onward is ≥ fastapi > docker, so docker can never match anything ahead.
  • fastapi === fastapi → matched; advance both.
  • postgres > kubernetes → the candidate's remaining skills are all ≥ postgres, so kubernetes can never be matched — it's a proven gap; advance j.
  • postgres === postgres → matched; advance both. react is left over on the skill side — irrelevant to this job.

Result: matched [fastapi, postgres], gaps [kubernetes], in one linear pass. Same-direction variant, two sequences — it's structurally the merge step of merge sort, just classifying elements instead of merging them. The pointer-advance rule is justified by the same sorted-order proof as Two Sum II: whichever head is smaller can never match anything ahead of the other pointer, so discarding it loses nothing.

The Canonical Problems, Mapped

These four cover both variants and every move-rule you'll be asked to justify. Know the elimination argument, not just the code.

ProblemVariantPointer rule & why it's safeComplexity
Two Sum II (sorted array) Opposite-direction sum < target → left++; sum > target → right--. Safe because the discarded index can't pair with any remaining value (sorted order). O(n) time, O(1) space
Container With Most Water Opposite-direction Always move the shorter line inward. Width only shrinks as pointers converge, so keeping the shorter line can never yield a bigger area — every container using it at this width or narrower is dominated. Note: heights are unsorted; the monotonic property is the shrinking width, not sorted values. O(n) time, O(1) space
Valid Palindrome Opposite-direction Compare s[left] vs s[right], converge on match, fail on mismatch (skip non-alphanumerics first). Symmetry is the structure being exploited; same skeleton reverses an array in place via swaps. O(n) time, O(1) space
Merge Two Sorted Arrays Same-direction, two sequences Take the smaller head, advance that pointer. The smaller head is ≤ everything remaining in both lists, so it's provably next in output order. (In-place variant: fill from the back to avoid overwriting unread elements.) O(n+m) time
Cross-project check — eBider Auction settlement has the same shape. To decide which lots clear, you have a sorted list of standing bid amounts and a sorted list of lot reserve prices, and want to pair each reserve with the lowest bid that clears it. Walk both sorted lists with two same-direction pointers: if the current bid ≥ the current reserve, pair them and advance both; if the bid is below the reserve, that bid clears nothing ahead either (reserves only get higher) — advance the bid pointer. O(bids + lots) instead of the O(bids × lots) cross-join, and the elimination argument is identical to applyGOAT's gap analysis: sorted order guarantees a discarded bid could never have cleared any later reserve.

Deeper — In-Place Partitioning (Same-Direction)

The converging variant answers pair questions; the same-direction variant does in-place restructuring under a loop invariant. The interview staple is Move Zeroes: push all zeroes to the end, preserving the relative order of non-zeroes, O(1) extra space.

function moveZeroes(arr) {
  let write = 0;                          // invariant: arr[0 .. write-1] = all non-zeroes, in order
  for (let read = 0; read < arr.length; read++) {
    if (arr[read] !== 0) {
      [arr[write], arr[read]] = [arr[read], arr[write]];
      write++;
    }
  }                                        // invariant at exit ⇒ zeroes occupy arr[write .. n-1]
}

The two pointers here split the array into three regions at all times: [0, write) = finalized non-zeroes, [write, read) = zeroes shuffled behind, [read, n) = unexamined. read classifies one element per step; write marks the boundary of finished output. Same skeleton solves "remove duplicates from sorted array": advance write only when arr[read] !== arr[write-1].

The three-pointer generalization is the Dutch National Flag partition (Dijkstra) — sort an array of 0s/1s/2s in one pass, and the core of quicksort's 3-way partition:

function dutchFlag(arr) {
  let low = 0, mid = 0, high = arr.length - 1;
  // invariant: [0,low) = 0s   [low,mid) = 1s   (mid,high] unexamined   (high,n) = 2s
  while (mid <= high) {                    // <= not < : arr[high] itself is still unexamined
    if (arr[mid] === 0)      { [arr[low], arr[mid]] = [arr[mid], arr[low]]; low++; mid++; }
    else if (arr[mid] === 2) { [arr[mid], arr[high]] = [arr[high], arr[mid]]; high--; }
    else                     { mid++; }    // it's a 1 — already in the middle region
  }
}

Two boundary subtleties interviewers probe: after swapping with high, you must not advance mid — the element that just arrived from the back is unexamined and could be a 0 that still needs to move to the front. After swapping with low you can advance both, because everything below mid has already been classified, so the swapped-in value is guaranteed to be a 1. And the loop must run while mid <= high, not <, or the final unexamined element is silently skipped.

Common mistakes

  • Two pointers on unsorted data with no justification. The move rule is a theorem about sorted order (or another monotone quantity). On unsorted input, "sum too big, move right left" proves nothing — you'll confidently skip the answer. Either sort first (O(n log n), and note that sorting destroys original indices — carry them as pairs if the problem wants indices) or use a hashmap.
  • < vs <= when pointers cross. Pair problems need left < right — at left === right you'd sum an element with itself. Partition problems like Dutch Flag need mid <= high because the element at high is still unclassified. Decide from the invariant, not from habit.
  • Forgetting duplicate-skipping in 3Sum-family problems. After emitting a valid triple, advancing each pointer by exactly one lands on equal values and emits the same triple again. You must skip runs: while (left < right && nums[left] === nums[left+1]) left++; and the mirror for right, plus skipping duplicate anchor values i.
The one-line litmus test Before writing two pointers, say out loud what a single comparison lets you permanently discard. If you can't complete the sentence "if this comparison goes this way, then no candidate involving index X can ever be the answer, because…" — the pattern doesn't apply, and the interviewer will find that out before you do.

Model answer
"Converging pointers: left at 0, right at n−1. If the sum equals the target, done. If it's too small, left++ — because arr[right] is the largest value still available, so if arr[left] falls short even with the maximum, it falls short with everything; no pair containing left can work. Symmetrically, too big means right--. Every comparison permanently eliminates all pairs through one index, so we cover all Θ(n²) pairs in at most n−1 comparisons: O(n) time, O(1) space. Termination at left < right — strict, because a pair needs two distinct indices — and if the loop exhausts, we've proven no pair exists, not just failed to find one."
Follow-up answer
"The hashmap is one pass: for each element, check whether target − x was already seen. Also O(n) time, but O(n) space, and — the key difference — it needs no sorted order. So: input already sorted, or memory-constrained, or the problem wants an ordered pair scan (like 3Sum's inner loop, where two pointers extends naturally and a hashmap gets messy with duplicates) → two pointers. Unsorted input where sorting isn't free, or the answer must be original indices — sorting destroys index positions — → hashmap. If someone says 'sort first, then two pointers' on an unsorted Two Sum, they've just spent O(n log n) to save O(n) space; sometimes right, but they should say the trade-off out loud."
Model answer
"The precondition isn't 'sorted' — it's an elimination proof: some monotonic structure that guarantees each pointer move discards only candidates that provably cannot be the answer. Sorted order is the most common source of that proof, but not the only one. Where it fails: pair-sum on an unsorted array — comparing arr[left] + arr[right] to the target tells you nothing about which direction to move, because neither end bounds the remaining values. Also subarray problems where the predicate isn't monotone — e.g. 'subarray sum equals k' with negative numbers: growing the window no longer monotonically grows the sum, so shrinking/growing decisions prove nothing, and you fall back to a prefix-sum hashmap."
Follow-up answer
"No — it confirms it. The monotone quantity there is the width, which strictly shrinks as the pointers converge. Area is min(height) × width. If I keep the shorter line and move inward, width goes down and the min-height can't exceed that shorter line — so every container still reachable with the shorter line is dominated by the one I already measured. Moving the shorter pointer discards only proven non-answers. That's the general form of the precondition: I don't need sorted values, I need a comparison whose outcome eliminates a whole set of candidates with certainty. Sorted order is just the most common way to manufacture that certainty."
Model answer
"3Sum is: sort, fix an anchor i, then run converging two pointers on the remainder for target −nums[i]. Duplicates leak in at two places, and both need explicit skips. First, the anchor: if nums[i] === nums[i−1], every triple starting at i was already produced by i−1 — continue. Second, inside the pointer loop: after emitting a valid triple, advancing each pointer by one may land on an equal value and re-emit the same triple — so skip runs: while (left < right && nums[left] === nums[left+1]) left++; and the mirror for right, then advance both once more past the run. The classic bug is doing only the anchor skip and assuming the inner loop is safe — it isn't, because sorted order packs equal values adjacent."
Follow-up answer
"O(n log n) for the sort plus n anchors × O(n) pointer scan = O(n²) time, O(1) extra space beyond the sort. Two pointers can't beat that here because the pattern's job is collapsing one pair-dimension from O(n²) to O(n) — with three free variables you still have to fix one of them, and that costs the outer n. There's also an output-size argument: 3Sum can have Θ(n²) distinct triples in the worst case, so any algorithm that lists them all is Ω(n²) regardless. The honest claim in an interview is 'O(n²), and that's essentially tight for enumerating solutions' — not 'two pointers makes everything linear'."
Gap to watch for Weak candidates recite the pointer moves; strong candidates state the invariant and the elimination proof unprompted. "Move left when the sum is small" is memorization. "Moving left discards every pair containing it, because even the largest remaining partner was insufficient" is understanding — and it's what lets you re-derive the rule under pressure on a variant you've never seen.
