Array & String Patterns

Sliding Window

A window that grows and shrinks over a sequence — the pattern behind every "longest/shortest subarray satisfying X" question.

Press F to toggle

Core Definition

Sliding window is two pointers, specialized: both pointers move only forward, and together they define a contiguous window [left..right] over an array or string. You maintain a running aggregate of the window's contents — a sum, a count, a frequency map — and update it incrementally as elements enter on the right and leave on the left. That incremental update is the entire trick: brute force recomputes the aggregate for every candidate subarray (O(n²) candidates × O(k) recompute), while sliding window touches each element exactly twice — once entering, once leaving — for O(n) total.

If you did Lesson 12: converging two pointers walk toward each other and the "window" between them is what you're discarding. Here the pointers chase each other in the same direction and the window between them is what you're measuring.

Note the shrink arrow loops back to the constraint check, not forward to the answer. You evict one element at a time and re-check — the loop decides how far to shrink, you don't compute it in one jump. That distinction is where most buggy implementations die (more in the Common Mistakes section and Playbook Q3).

Plain English: Rate Limiting an API Gateway

The interview framing is arrays and strings, but you've already met this algorithm in infrastructure. A sliding window rate limiter — "allow at most 5 requests per rolling 60 seconds per API key" — is exactly the variable-size window, except the window slides over time instead of over an array index.

Keep a queue of request timestamps per API key. On each incoming request:

const WINDOW_MS = 60_000;   // rolling 60s window
const LIMIT = 5;            // max requests per window
const log = new Map();      // apiKey -> queue of timestamps (oldest first)

function allowRequest(apiKey, now) {
  const q = log.get(apiKey) ?? [];

  // SHRINK from the left: evict timestamps that fell out of the window.
  // Same move as left++ — the constraint is checked against what remains.
  while (q.length && q[0] <= now - WINDOW_MS) q.shift();

  // CHECK the constraint on the current window
  if (q.length >= LIMIT) return false;   // 429 Too Many Requests

  // EXPAND on the right: this request enters the window
  q.push(now);
  log.set(apiKey, q);
  return true;
}

Walk it with LIMIT = 3 and a 60s window. Requests from one key arrive at t = 0s, 10s, 20s, 30s, 65s:

t=0s   evict: nothing older than -60s        queue []        → allow, queue [0]
t=10s  evict: nothing ≤ -50s                 queue [0]       → allow, queue [0,10]
t=20s  evict: nothing ≤ -40s                 queue [0,10]    → allow, queue [0,10,20]
t=30s  evict: nothing ≤ -30s                 queue [0,10,20] → len 3 ≥ LIMIT → REJECT
t=65s  evict: 0 ≤ 5 → shift it out           queue [10,20]   → allow, queue [10,20,65]

Map the pieces back to the array version and it's a one-to-one correspondence: the timestamp queue is the window contents, q.shift() is left++, q.push(now) is right++, and q.length ≥ LIMIT is the constraint check. The only structural difference: time advances on its own, so the "shrink" happens lazily at read time (evict-then-check) rather than in a loop over indices. This is the sliding window log algorithm; the cheaper sliding window counter approximates it with two fixed buckets and a weighted interpolation — the classic memory-vs-precision trade-off when the timestamp queue per key gets too expensive at scale.

Why not a fixed window counter? The naive "reset the counter every minute on the minute" limiter admits a burst of 2× LIMIT straddling the boundary (5 requests at 0:59, 5 more at 1:01). The sliding window exists precisely to kill that edge — the window is anchored to now, not to wall-clock buckets. Same reason the array version tracks [left..right] continuously instead of chopping the input into fixed chunks.

The Canonical Problems, Mapped

Longest substring without repeating characters Variable (maximize) Frequency map (or last-seen-index map) freq[s[right]] > 1 O(n) time, O(min(n, Σ)) space
Longest substring with at most K distinct chars Variable (maximize) Frequency map + distinct count map.size > K O(n) time, O(K) space
Minimum window substring Variable (minimize) Need-map vs have-map + "formed" counter Inverted: shrink while valid to find the smallest valid window, recording best before each shrink O(n + m) time, O(Σ) space

Note the inversion on minimum window substring: when you're maximizing, the window is usually valid and you shrink to restore validity; when you're minimizing, you expand until valid and then shrink while still valid, harvesting the answer on the way. Same machinery, opposite harvest point — say this out loud in an interview and you've demonstrated you understand the pattern rather than having memorized one problem.

How to recognize a sliding-window problem

  • Contiguity is stated or implied: "subarray", "substring", "consecutive", "window", "rolling". If the answer can be non-contiguous (a subsequence), sliding window doesn't apply.
  • You're optimizing under a constraint: "longest / shortest / max / min / count of … such that X". The constraint (no duplicates, ≤ K distinct, sum ≤ S) becomes your shrink condition.
  • Monotonicity holds: expanding the window can only preserve or worsen a violation, and shrinking can only preserve or fix it. This is why "shortest subarray with sum ≥ S" works with all-positive numbers (adding elements only grows the sum) but breaks with negatives — a longer window can have a smaller sum, so eviction decisions stop being safe and you need prefix sums + a monotonic deque instead. Checking this before you code is a senior-signal move.
Cross-project check — AI Sales Agent Your LangGraph agent makes LLM calls per conversation session. Provider rate limits (requests-per-minute, tokens-per-minute) are rolling windows — so a per-session throttle guarding those calls is the timestamp-queue limiter above, keyed by session instead of API key, with tokens-per-minute being the same window but with a weighted aggregate: the running sum of token counts in the window rather than a plain length check. Evict expired entries from the left, sum what remains, compare to the TPM cap. One pattern, two constraints.

Fixed Window First: Max Sum Subarray of Size k

The simpler variant — no shrink logic to reason about, because the shrink is mechanical: every step after warm-up, exactly one element enters and exactly one leaves.

function maxSumOfSizeK(nums, k) {
  let sum = 0;
  for (let i = 0; i < k; i++) sum += nums[i];   // build the first window
  let best = sum;
  for (let i = k; i < nums.length; i++) {
    sum += nums[i] - nums[i - k];   // one enters, one leaves — O(1) per slide
    best = Math.max(best, sum);
  }
  return best;
}
nums = [2, 1, 5, 1, 3, 2], k = 3

warm-up   [2, 1, 5] 1  3  2    sum = 8            best = 8
slide      2 [1, 5, 1] 3  2    sum = 8 + 1 - 2 = 7  best = 8
slide      2  1 [5, 1, 3] 2    sum = 7 + 3 - 1 = 9  best = 9
slide      2  1  5 [1, 3, 2]   sum = 9 + 2 - 5 = 6  best = 9  → return 9

Brute force computes each of the n − k + 1 windows from scratch: O(n·k). The sum += nums[i] − nums[i−k] line is the whole optimization — the overlap between consecutive windows is k − 1 elements, so recomputing it is pure waste. Every sliding-window speedup, fixed or variable, is this one observation: consecutive windows overlap almost entirely, so carry the aggregate instead of rebuilding it.

Full Trace: Longest Substring Without Repeating Characters

The canonical variable-size problem. Constraint: no character appears twice in the window. Aggregate: a frequency map. Shrink condition: the character just admitted has count > 1.

function lengthOfLongestSubstring(s) {
  const freq = new Map();
  let left = 0, best = 0;
  for (let right = 0; right < s.length; right++) {
    const c = s[right];
    freq.set(c, (freq.get(c) ?? 0) + 1);          // EXPAND

    while (freq.get(c) > 1) {                      // constraint violated?
      const l = s[left];
      freq.set(l, freq.get(l) - 1);                // SHRINK one step
      if (freq.get(l) === 0) freq.delete(l);
      left++;
    }                                              // re-check, repeat

    best = Math.max(best, right - left + 1);       // window valid → harvest
  }
  return best;
}

Trace on s = "abcabcbb". Watch right = 6 — it's the step that proves why the shrink is a loop, not a single eviction:

right=1  c='b'  freq {a:1,b:1}      no violation           window [0..1] "ab"   best=2
right=2  c='c'  freq {a:1,b:1,c:1}  no violation           window [0..2] "abc"  best=3

At right = 6 the first eviction removes 'a' — an innocent bystander that isn't the duplicate — and the constraint is still violated, so the loop evicts again. The window can't skip past 'a': it must stay contiguous, so everything left of the old duplicate's position has to go, one step at a time, with the aggregate updated on every eviction. That's the whole reason the shrink is while, not if.

The jump optimization: you can replace the while-loop by storing each character's last-seen index and jumping left = Math.max(left, lastSeen[c] + 1). Fewer operations per step, same O(n). But the Math.max is load-bearing — drop it and stale indices move left backward, silently re-admitting duplicates (dissected in Playbook Q3). The freq-map version has no such trap, which is why it's the safer default under interview pressure and generalizes directly to "at most K distinct" where a jump target doesn't even exist.

Common mistakes

  • Shrinking by the wrong amount. Computing "how far to move left" in one arithmetic jump instead of evicting one element at a time and re-checking. The while-loop version is self-correcting; the jump version is only correct with the Math.max guard against stale indices.
  • Set where you need a frequency map. A set answers "is x present" — fine for the no-repeats problem. But "at most K distinct" and minimum-window-substring need counts: when you evict one 'b' and another 'b' remains in the window, a set deletion wrongly declares 'b' gone. If eviction of one occurrence must not erase the character's presence, you need counts. Decrement, and only delete the key at zero.
  • Forgetting to update the aggregate on shrink. Everyone remembers freq[s[right]]++; the bug magnet is the eviction side — left++ without freq[s[left]]-- (or without sum -= nums[left]). The aggregate must describe exactly the current window at all times, or every subsequent constraint check is garbage.
  • Off-by-one on window length. It's right − left + 1 for an inclusive [left..right] window. Harvest it only when the window is valid — after the shrink loop for maximize-problems, before/during the shrink for minimize-problems.
Cross-project check — SyncPoint "Alert if a workflow step fails more than N times in the last 10 minutes" is the same shape: a per-step queue of failure timestamps, evict-from-the-left on read, compare remaining length to N. Any rolling-metric feature — error-rate windows, per-user activity throttles, burst detection — is a variable-size sliding window over time. Recognizing that saves you from reaching for a cron job that recounts a database table every minute (the O(n·k) brute force of infrastructure).

Interview Playbook

// LIKELY INTERVIEW QUESTIONS — click to reveal answers
Model answer
"Brute force checks every substring — O(n²) substrings, O(n) to verify uniqueness each, so O(n³), or O(n²) with a set. But the constraint is monotonic: if a window has a duplicate, every larger window containing it does too — so I never need to revisit shrunken ground, which means a sliding window works. I keep a window [left..right] and a frequency map of its characters. Each step I advance right and increment freq of the new character. If its count exceeds 1, the constraint is violated: I evict s[left], decrement its count, advance left, and re-check — one step at a time — until the count drops back to 1. Then the window is valid and I record max(best, right − left + 1). On 'abcabcbb' this returns 3 for 'abc'. Time O(n), space O(min(n, alphabet)). There's a variant that stores each character's last-seen index and jumps left directly, but the while-loop version is the one I'd write first — it's harder to get wrong and it generalizes to the at-most-K-distinct family."
Follow-up answer
"Because complexity is bounded by total pointer movement, not loop nesting. Right moves forward exactly n times. Left only ever moves forward, so across the entire run it moves at most n times total — the inner while-loop's iterations are shared across all outer iterations, not multiplied by them. Total work ≤ 2n, so it's amortized O(n). The intuition: each element enters the window once and leaves at most once, and we do O(1) work per entry and per exit. The nested-loop O(n²) bound would only apply if left could move backward and re-process elements — and the algorithm's core invariant is that it never does."
Model answer
"First gate: is the answer contiguous — subarray, substring, consecutive, rolling? If it's a subsequence, sliding window is out. Second: is there an explicit k? 'Every window of size k', 'k consecutive elements' → fixed window, no shrink logic, just add-one-evict-one. If instead the size is what I'm solving for — 'longest/shortest such that condition X' — it's a variable window and X becomes my shrink condition. Third gate, the one juniors skip: monotonicity. Expanding must only preserve-or-worsen a violation and shrinking must only preserve-or-fix it. If that doesn't hold, the greedy pointer movement makes unsafe decisions and the pattern breaks. Then one last orientation check: maximizing → shrink to restore validity, harvest after the shrink loop; minimizing → shrink while still valid, harvest before each shrink."
Follow-up answer
"With all-positive numbers, yes — textbook variable window. Extending the window strictly increases the sum and shrinking strictly decreases it, so 'sum ≥ S' is monotonic: expand until valid, then shrink while still valid, recording the length before each eviction. O(n). With negatives allowed, it breaks: a longer window can have a smaller sum, so shrinking might discard a prefix whose negative values were about to be offset — moving left forward is no longer a safe irreversible decision. The correct tool becomes prefix sums with a monotonic deque, which is O(n) but a genuinely different algorithm. Naming that boundary condition unprompted is exactly the check I'd run before writing any code."
Model answer
"The last-seen map is never cleaned up — it remembers occurrences that have already fallen out of the window. So lastSeen[c] can point to a stale index left of the current left pointer. If I assign left = lastSeen[c] + 1 unconditionally, a stale entry can move left backward, and that's fatal twice over: it re-admits characters I already evicted — the window's claimed contents no longer match reality, so I can report a 'longest substring' that actually contains a duplicate — and it breaks the forward-only invariant that the O(n) bound depends on. The fix is one guard: left = Math.max(left, lastSeen[c] + 1). The one-step-at-a-time freq-map version is immune by construction, because eviction physically removes each character from the aggregate as left passes it — the state can't go stale. That's why I default to it under pressure and treat the jump as an optimization to mention, not to lead with."
Follow-up answer
"s = 'abba'. Trace the naive jump: right=0 'a', window 'a'. right=1 'b', window 'ab'. right=2 'b' — duplicate, lastSeen[b]=1, left jumps to 2, window 'b'. Correct so far. right=3 'a' — lastSeen[a]=0, a stale index from before the window moved. Naive jump sets left = 0 + 1 = 1: left moved backward from 2 to 1, window claims to be 'bba', length 3 — but 'bba' contains a repeated b. The algorithm returns 3; the true answer is 2 ('ab' or 'ba'). With the Math.max guard, left stays at 2, the window is 'ba', and the answer is correct. Four characters, minimal counterexample — worth memorizing as the test case for any last-seen-index implementation."
