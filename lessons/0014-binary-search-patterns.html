Binary Search Patterns

Beyond sorted arrays — binary search on the answer, rotated arrays, and search spaces that aren't obviously sorted.

Press F to toggle

The Real Invariant: A Monotonic Predicate

The interview-grade framing is not "binary search finds an element in a sorted array." It's this: binary search halves any search space over which some boolean predicate is monotonic — i.e., there's a function p(x) that is false, false, ..., false, true, true, ..., true as x sweeps the space, and you're finding the boundary where it flips. A sorted array is just the special case where the predicate is arr[i] >= target. Once you see it this way, three things follow:

  • The input doesn't have to be an array. Any ordered range of candidate answers works, as long as feasibility is monotonic across it.
  • The array doesn't have to be globally sorted. It only needs enough structure that, at each step, you can decide which half provably cannot contain the answer (rotated arrays exploit exactly this).
  • If the predicate is not monotonic — feasibility flips true→false→true — binary search is simply wrong for the problem, no matter how sorted the data looks. That's the failure many candidates never check for.

The two loop templates — and why mixing them causes infinite loops

Almost every binary search bug is a mismatch between three coupled choices: the loop condition, the boundary updates, and the direction mid rounds. They come in two consistent templates. Pick one deliberately; never mix parts.

// TEMPLATE A — exact-match search. Search space shrinks to empty.
// Invariant: answer, if it exists, is inside [left, right] (inclusive).
left = 0; right = n - 1
while left <= right:                 // run until the window is EMPTY
    mid = left + (right - left) / 2  // floor division; avoids (left+right) overflow
    if arr[mid] == target: return mid
    elif arr[mid] < target: left  = mid + 1   // mid is ruled out — exclude it
    else:                   right = mid - 1   // mid is ruled out — exclude it
return -1

// TEMPLATE B — boundary search (lower_bound). Search space shrinks to ONE element.
// Invariant: the flip point (first i where p(i) is true) is inside [left, right].
left = 0; right = n                  // note: right = n, "not found" sentinel
while left < right:                  // run until ONE candidate remains
    mid = left + (right - left) / 2  // floor — pairs with right = mid
    if p(mid):  right = mid          // mid might BE the answer — keep it in range
    else:       left  = mid + 1      // mid is definitely not the answer
return left                          // first index where p is true, or n

Three details that separate a clean answer from a buggy one:

  • mid = left + (right - left) / 2, not (left + right) / 2. In fixed-width integer languages (Java, C++, C#), left + right can overflow when both are near INT_MAX. In Python/JS it's habit-signaling; in Java it's correctness. (Historical note: this exact overflow sat in the JDK's Arrays.binarySearch for years before Joshua Bloch flagged it.)
  • right = mid vs right = mid - 1 is not a stylistic choice — it encodes whether mid can still be the answer. In Template B you tested p(mid) is true, so mid may be the boundary itself; excluding it with mid - 1 can skip the answer.
  • Rounding direction must match the update that can stall. With floor rounding, mid can equal left when the window has 2 elements — so the branch that keeps the low side must move it (left = mid + 1). If your update is left = mid (searching for the last true, upper-boundary flavor), you must round up: mid = left + (right - left + 1) / 2. Get this wrong and the loop spins forever on a 2-element window.

Notice the shape of the diagram: it is exactly Template B, but i is a capacity, not an index, and p(x) is a greedy simulation instead of an array read. Feasibility is monotonic by construction — if capacity x ships everything in ≤ D days, capacity x+1 trivially does too — so the false→true flip point exists and is unique. That monotonicity argument is the thing to say out loud in an interview before writing the loop; it's what licenses binary search at all.

Plain English: First Log Entry After a Timestamp

Concrete, production-shaped scenario — the kind you'd hit building observability tooling. You have an append-only event log: entries are written in timestamp order, millions of them, stored as a sorted array (or a memory-mapped file of fixed-width records — same access pattern). An incident started at T = 14:32:07.000 and you want every entry from that moment onward. The query is: find the first index whose timestamp is ≥ T, then stream from there.

This is a pure lower_bound — a boundary search, not an exact match. T itself almost certainly doesn't appear in the log (timestamps are effectively unique at millisecond precision), so Template A's arr[mid] == target would return -1 and tell you nothing. The predicate is p(i) = log[i].ts >= T, which is monotonic precisely because the log is append-ordered: false for every entry before the incident, true for every entry after, one flip point.

// p(i): "is entry i at-or-after the incident start?"
// Monotonic because the log is append-ordered by timestamp.

function firstEntryAtOrAfter(log, T):
    left = 0
    right = log.length            // sentinel: "no entry is >= T"
    while left < right:
        mid = left + (right - left) / 2
        if log[mid].ts >= T:      // p(mid) true: mid could be the first one
            right = mid           //   → keep mid in the window
        else:                     // p(mid) false: entry is pre-incident
            left = mid + 1        //   → mid can never be the answer
    return left                   // first index with ts >= T, or log.length

Walk one iteration to feel the invariant. Say 10M entries, mid = 5,000,000, and log[mid].ts = 14:31:59 — before the incident. Every entry at index ≤ 5,000,000 is also before it (append order), so five million entries are eliminated by one comparison: left = 5,000,001. That's the entire value proposition — ~23 comparisons instead of a 10M-row scan. Two production-grade follow-ons:

  • Range queries compose from two boundary searches. "Entries between T1 and T2" = lowerBound(T1) to lowerBound(T2). Count of entries in the window is just the difference of the two indices — no scan.
  • This is what your database does. A B-tree index seek for WHERE ts >= T ORDER BY ts is the same boundary search generalized to a high-fanout tree (log~200 instead of log2). Lesson 0002's index-seek mechanics and this lesson's lower_bound are one idea at two scales.

Cross-project check — applyGOAT Suppose applyGOAT's pricing has a sorted array of tier price points — [0, 9, 29, 79, 199] — and given a user's stated budget you want the highest tier they qualify for (largest price ≤ budget). That is upperBound(budget) − 1: find the first price strictly greater than the budget, step back one. It's the mirror-image boundary search — you're finding the last true instead of the first, which is exactly the case where left = mid updates appear and mid must round up (or, cleaner: keep the round-down lower-bound template on the predicate price > budget and subtract one from the result). Five tiers doesn't need log-time — the point is that the moment this becomes "find the qualifying entry in a sorted fee schedule / rate table / quota ladder with hundreds of rows," it's the same two lines, and the boundary-vs-exact-match distinction is what keeps a budget of exactly 29 from falling into the wrong tier.

Deeper: The Rotated Sorted Array, Worked Properly

Take nums = [15, 18, 2, 3, 6, 12] — a sorted array rotated left of its minimum. Global sortedness is broken (18 → 2), so plain binary search's comparison no longer tells you which side the target is on. The recovery insight: when you cut a rotated array at any mid, at least one of the two halves contains no rotation point, and is therefore perfectly sorted. There's only one rotation point in the whole array; it can't be in both halves.

So each iteration becomes a two-step decision instead of one comparison:

  1. Which half is sorted? Test nums[left] <= nums[mid]. If true, the left half [left..mid] is sorted (no wrap between them — with distinct values, a wrap inside that span would force nums[left] > nums[mid]). If false, the rotation point is in the left half, so the right half [mid..right] must be sorted.
  2. Is the target inside the sorted half's range? A sorted half has known endpoints, so a simple range check answers definitively. If the target is in range, search that half. If not, it must be in the other (messy) half — which is itself a smaller rotated-or-sorted array, so recursion/iteration is sound.
function searchRotated(nums, target):
    left = 0; right = nums.length - 1
    while left <= right:
        mid = left + (right - left) / 2
        if nums[mid] == target: return mid

        if nums[left] <= nums[mid]:               // LEFT half is sorted
            if nums[left] <= target < nums[mid]:  // target inside sorted-left's range?
                right = mid - 1                    //   yes → binary search left
            else:
                left = mid + 1                     //   no  → must be in messy right
        else:                                      // RIGHT half is sorted
            if nums[mid] < target <= nums[right]: // target inside sorted-right's range?
                left = mid + 1                     //   yes → binary search right
            else:
                right = mid - 1                    //   no  → must be in messy left
    return -1

Trace target = 3 in [15, 18, 2, 3, 6, 12]:

  • left=0, right=5, mid=2 → nums[2]=2. Is left half sorted? nums[0]=15 <= 2? No — so the right half [2, 3, 6, 12] is sorted. Is 2 < 3 <= 12? Yes → left = 3.
  • left=3, right=5, mid=4 → nums[4]=6. nums[3]=3 <= 6 → left half sorted. Is 3 <= 3 < 6? Yes → right = 3.
  • left=3, right=3, mid=3 → nums[3] == 3. Found in 3 probes.

Note the <= in nums[left] <= nums[mid]: when left == mid (2-element window), the left "half" is one element and trivially sorted — using strict < there misroutes that case. And the duplicates caveat is worth volunteering unprompted: with duplicates (e.g. [2, 2, 2, 3, 2]), nums[left] == nums[mid] == nums[right] tells you nothing about which half is sorted; the standard fix is shrinking both ends by one (left++, right--), which is correct but collapses the worst case to O(n). Knowing that binary search's guarantee degrades, and exactly why, reads as senior.

Common failure modes — the ones interviewers actually watch for 1. Infinite loop from mismatched template parts: while left < right with left = mid and floor rounding spins forever on [left, left+1] — mid computes to left, and left = mid makes no progress. 2. Off-by-one on exit: Template A ends with left > right and left = insertion point; Template B ends with left == right = the boundary. Returning the wrong variable after choosing a template is a classic silent bug. 3. Assuming sorted when it's only rotated: plain binary search on a rotated array returns wrong answers silently — it terminates, it's fast, and it's wrong. 4. Binary searching a non-monotonic predicate: "find a value where cost is exactly K" when cost rises then falls has two flip points; binary search converges confidently to garbage. Prove the single-flip property before writing the loop — if you can't articulate why p(x) true ⇒ p(x+1) true, you don't have a binary search problem yet.
Cross-project check — Engineering Execution / observability The log-timestamp lower_bound above is not a toy: it's the read path of every time-series query you'll run during an incident. Loki, ClickHouse, and Prometheus's TSDB all lay out chunks/blocks time-ordered and boundary-search to the start offset before scanning; SQL WHERE created_at >= ? on an indexed column is the B-tree cousin. The transferable habit: whenever you find yourself writing a linear scan over data that has a natural order (timestamps, sequence numbers, monotonic IDs), ask "what's the predicate, and is it monotonic?" — if yes, the scan is a boundary search you haven't written yet.

Interview Playbook

// LIKELY INTERVIEW QUESTIONS — click to reveal answers
Model answer
"Global order is broken, but the key structural fact survives: there's exactly one rotation point, so when I split at any mid, at least one half is fully sorted. Each iteration I do two checks. First, which half is sorted: if nums[left] <= nums[mid], the left half is — a rotation point inside it would force nums[left] > nums[mid] with distinct values. Second, I range-check the target against the sorted half's endpoints, because a sorted range is the only place I can answer 'is it here?' definitively. If it's in range, I recurse into that half; otherwise it must be in the other half, which is itself a smaller rotated-or-sorted array, so the invariant holds. Standard Template-A shell — left <= right, exclude mid on both updates — and I still halve every step, so O(log n). I'd also note the <= in the sorted-half test matters: when left equals mid on a two-element window, that one-element 'half' is trivially sorted, and strict less-than misroutes it."
Follow-up answer
"Duplicates break the half-detection: if nums[left] == nums[mid] == nums[right] — say [2, 2, 2, 3, 2] — I can't tell which side holds the rotation point. The standard fix is to shrink both boundaries by one and continue; it's correct, but in the adversarial all-duplicates case every step shrinks by O(1) instead of halving, so worst case degrades to O(n). That's not a bug in my solution — it's information-theoretic: with duplicates masking the structure, no comparison-based approach can guarantee log time. I'd say that explicitly rather than hand-wave the complexity."
Model answer
"Binary search never needed a sorted array — it needs a monotonic predicate over an ordered search space. Here the space is the range of candidate answers, and the predicate is a feasibility check. Take ship capacities: feasible(c) = 'can I ship everything within D days at capacity c', computed by a greedy O(n) pass. It's monotonic by a one-line argument — any schedule that works at capacity c works unchanged at c+1 — so feasibility is false…false, true…true across the range, and the minimum feasible capacity is the flip point. I binary search for it with the lower-bound template: feasible → right = mid because mid might be the answer; infeasible → left = mid + 1. Total cost O(n log R) where R is the answer range. The tell for this pattern is the phrase 'minimize the maximum' or 'maximize the minimum' — the answer is a threshold, and thresholds over monotone feasibility are binary-searchable. Same skeleton solves Koko Eating Bananas and Split Array Largest Sum; only the feasibility function changes."
Follow-up answer
"Bounds must bracket the flip point: lo = the smallest possibly-valid answer, hi = a value guaranteed feasible. For shipping, lo = max(weights) — anything smaller can't even load the heaviest package, so the predicate isn't just false there, it's ill-defined — and hi = sum(weights), which trivially ships everything in one day. Too-wide bounds only cost extra log-factor iterations — cheap. Too-narrow bounds are a correctness bug: if the true answer sits outside [lo, hi], the loop converges to a boundary value and returns it with full confidence — no error, no signal. So I derive both bounds from the problem's constraints and state the justification out loud; 'hi is guaranteed feasible' is the invariant that makes returning left safe."
Model answer
"Two-element window: right == left + 1. Floor division gives mid = left. If check(mid) is true, the update is left = mid — which is left = left. No progress, infinite loop. The root cause is a rounding/update mismatch: this loop keeps mid in the window on the low side (left = mid, hunting the last true), so mid must round toward the high side to guarantee progress — mid = left + (right - left + 1) / 2. Rule of thumb: whichever boundary can be assigned mid directly, round mid away from that boundary. And separately, (left + right) / 2 is an overflow bug in fixed-width languages — the safe form is left + (right - left) / 2, or with the +1 for round-up. Two distinct bugs in one line; I'd name both."
Follow-up answer
"I default to the lower-bound template — left < right, right = mid / left = mid + 1, floor rounding, return left — because it's the one where floor rounding and the updates can never stall, and it generalizes: exact match is lower_bound plus an equality check, last-occurrence is lowerBound(target + 1) - 1, and search-on-the-answer is the same loop over a value range. One memorized-and-understood template with derived variants beats four half-remembered ones; the failure mode in interviews is stitching the condition from one template to the updates of another under pressure."
