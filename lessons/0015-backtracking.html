The Core Pattern

Backtracking is depth-first search over a decision tree. Each node in the tree is a partial solution; each edge is a decision that extends it. At every node you do exactly three things, in order, for each available option:

function backtrack(state, path):
  if isComplete(path):            # base case — a leaf that's a full solution
    record(copy of path)          # COPY — path is shared mutable state
    return

  for option in candidates(state, path):
    if not promising(option, state, path):   # PRUNE — skip subtrees that
      continue                               # cannot produce a valid solution

    apply(option, state, path)    # 1. CHOOSE  — mutate shared state
    backtrack(state, path)        # 2. EXPLORE — recurse one level deeper
    undo(option, state, path)     # 3. UN-CHOOSE — restore state EXACTLY

The step candidates most often get wrong — or omit entirely — is un-choose. path and any auxiliary structures (a used set, a running sum, a column-occupancy array) are shared mutable state threaded through every recursive call. There is one array, not a copy per branch. If you don't undo a choice before trying the next option at the same node, every sibling branch explores from corrupted state — and the bug is silent: you get plausible-looking but wrong output, not a crash. The symmetric discipline is what makes the pattern work: every mutation before the recursive call has a mirror-image un-mutation after it. If your choose is path.push(x); used.add(x), your un-choose is used.delete(x); path.pop() — same operations, reverse order.

The second defining idea is pruning. Brute-force enumeration is exponential by nature: n elements have 2ⁿ subsets and n! orderings — no algorithm that must output all of them can beat that. Backtracking doesn't change the complexity class. Its value is refusing to descend into subtrees that provably cannot contain a valid solution: if a partial solution already violates a constraint (running sum exceeds target, two queens attack each other), every extension of it also violates the constraint, so the entire subtree below that node — potentially exponentially many leaves — is skipped with one check. Pruning is a constant-factor and typical-case win inside an exponential class, not an escape from it.

Read the greyed nodes as whole subtrees that never execute. The node marked "sum 8" isn't just one skipped call — everything that would have hung below it (choosing 2 again, then 3, then 7…) is gone. And because the candidates are sorted, overshooting with 2 means 3 and 7 must also overshoot, so the loop breaks and those siblings are pruned without even a comparison per branch. That is the visual interviewers want articulated: pruning removes subtrees, and the earlier (higher in the tree) you prune, the more exponentially-sized work you never do.

Plain English: Mini Hermes Planning a Tool-Call Sequence

Your Mini Hermes Agent has a goal — say, "summarize this week's activity for a memory-graph entity and file it" — and a toolbox: search_memory, read_entity, fetch_messages, summarize, write_memory. A plan is a sequence of tool calls, and not every sequence is viable: summarize needs text in the working context, write_memory needs a summary to exist, fetch_messages needs an entity id that read_entity provides. Finding a valid plan is a search over the tree of possible tool-call sequences — exactly the backtracking template:

function plan(goal, worldState, path):
  if satisfies(worldState, goal):
    return copy(path)                        # found a viable plan

  if path.length >= MAX_DEPTH:               # budget prune — a plan this long
    return null                              # isn't worth executing anyway

  for tool in toolbox:
    if not preconditionsMet(tool, worldState):   # PRUNE: summarize with no
      continue                                   # text in context? dead branch.

    effects = simulate(tool, worldState)     # CHOOSE — apply the tool's
    worldState.apply(effects)                # declared effects to sim state
    path.push(tool)

    result = plan(goal, worldState, path)    # EXPLORE
    if result: return result                 # first viable plan wins

    path.pop()                               # UN-CHOOSE — revert BOTH the
    worldState.revert(effects)               # path and the simulated world

Walk one concrete trace. Goal state requires summary_written. Depth budget 4.

  1. Choose write_memory first? Precondition summary_exists is false — pruned before recursing. Same for summarize (nothing in context) and fetch_messages (no entity id). Only search_memory and read_entity survive the precondition check at the root: five children collapse to two.
  2. Choose search_memory → explore. From here, choose search_memory again → again → again — depth budget hit with the goal unsatisfied. Dead end. Each level returns null, pops the call off path, and reverts the simulated effects. This is the backtrack: the agent walks back up the tree undoing decisions, not restarting from scratch.
  3. Back at depth 1 with a clean state, choose read_entity → now entity_id exists → fetch_messages unlocks → summarize unlocks → write_memory unlocks → goal satisfied. Return [read_entity, fetch_messages, summarize, write_memory].

Two things to notice, because they're the same two things that matter in interview problems. First, the precondition check is pruning — with 5 tools and depth 4, blind enumeration is 5⁴ = 625 sequences; precondition pruning explores a handful. The search stays worst-case exponential in depth (an adversarial toolbox forces it), but real constraint structure makes it cheap in practice. Second, the un-choose must revert everything choose touched — both path and worldState. Revert only the path and the next branch simulates from a world where messages were already fetched: the planner emits plans that skip required steps. Stale-state corruption, the classic backtracking bug, in production clothing.

The i vs i + 1 distinction in the two Combination Sum rows is where off-by-one bugs live. Passing i means "the current element may be chosen again" (unbounded reuse, LC 39). Passing i + 1 means "move past it" (each element once, LC 40). Passing start unchanged, or restarting from 0, means "regenerate every ordering" — you'll emit [2,3] and [3,2] as separate results and fail on duplicates. The start-index isn't bookkeeping; it's the mechanism that makes combinations combinations instead of permutations.

Cross-project check — Mini Hermes Agent The planner sketch above is a real architecture, not an analogy: classical AI planning (STRIPS-style) is backtracking search over action sequences with precondition pruning, and an agent that "tries a tool path, hits a dead end, and revises the plan" is doing exactly this with an LLM proposing the candidate ordering. The design lesson transfers directly: make tool effects explicitly declared and reversible in simulation, so the un-choose step is trivial. If a tool's effects on world state are implicit or entangled, you can't cleanly revert a simulated branch — which is the agent-architecture version of forgetting to backtrack shared state. It's also why real agent frameworks separate planning (simulated, cheap, revertible) from execution (real side effects, not revertible) — you backtrack in the simulator, never in production.
Cross-project check — applyGOAT Interview scheduling under constraints — interviewer availability, candidate availability, no double-booking — is N-Queens wearing a suit: assign a slot per interview (row), reject assignments conflicting with already-booked slots (column/diagonal check), backtrack when a partial schedule strands a later interview with zero feasible slots. The senior insight is knowing when not to use it: with a handful of interviews, backtracking with constraint checks is fine; at scale, scheduling is NP-hard and production systems switch to constraint solvers or greedy heuristics with repair. "I'd backtrack here, and here's the input size where I'd stop" is a stronger answer than either extreme.

Full Worked Trace: Combination Sum

Candidates [2, 3, 7] (sorted), target 7, reuse allowed. Real pseudocode with the three phases and the prune marked:

function combinationSum(candidates, target):
  sort(candidates)                       # prerequisite for the break-prune
  results = []
  path = []                              # ONE shared array for all branches

  function backtrack(start, remaining):
    if remaining == 0:
      results.push(copy(path))           # copy! path keeps mutating after this
      return

    for i in start .. candidates.length - 1:
      if candidates[i] > remaining:      # PRUNE: sorted ⇒ every later
        break                            # candidate overshoots too

      path.push(candidates[i])           # CHOOSE
      backtrack(i, remaining - candidates[i])   # EXPLORE — 'i' not 'i+1':
                                                # reuse of same element allowed
      path.pop()                         # UN-CHOOSE

  backtrack(0, target)
  return results

The execution, step by step. Indentation = recursion depth; every push has its mirrored pop:

results = [[2,2,3], [7]]

Three details in this trace that are the difference between a working solution and a subtle disaster:

  • copy(path) at the record step. Push a reference to path itself and every "result" is the same array — which is empty by the time the function returns, because every push got popped. You return [[], []]. This is the single most common backtracking bug in JavaScript and Python, and it produces no error, just garbage.
  • The break vs continue prune. With unsorted candidates you can only continue (skip this one candidate). Sorting upgrades it to break — one overshoot proves all remaining siblings overshoot. At backtrack(0, 1) that one break eliminated three subtrees in a single comparison. Prune as high in the tree and as cheaply as you can; a prune that fires at the leaves saves almost nothing because you've already paid for the path down.
  • backtrack(i, …) not backtrack(i+1, …), and never backtrack(0, …). Passing i allows [2,2,3]. Passing i+1 would forbid reuse (that's Combination Sum II). Restarting at 0 would also find [3,2,2] and [2,3,2] — duplicates by reordering. Notice in the trace: after popping into path=[2,3], the search does not retry 2, because start is 1. That asymmetry is deliberate and load-bearing.
// LIKELY INTERVIEW QUESTIONS — click to reveal answers
Model answer
"It's backtracking over a decision tree: each level fixes the next position, each branch is which unused element fills it. I keep a shared path and a used set. At each node, for each element not in used: choose — push it and mark used; explore — recurse; un-choose — unmark and pop, so the next sibling sees clean state. When path length equals n, I record a copy — the path array itself keeps mutating, so pushing the reference would give me n! views of the same, ultimately empty, array. Complexity is O(n · n!): n! leaves, and the copy at each leaf costs O(n). Space is O(n) for the recursion depth and path, excluding the output. The un-choose step is the part I'd flag explicitly: choose and un-choose are mirror images around the recursive call, and if I mutate anything else along the way, it gets the same treatment."
Follow-up answer
"With duplicates, distinct orderings of identical values collapse — [1,1,2] has 3 unique permutations, not 6. Sort first, then add one pruning line: skip element i if nums[i] == nums[i−1] and i−1 is currently unused — meaning an identical element at the same tree level was already tried, so this branch would regenerate an identical subtree. That's dedup by pruning at generation time, which beats generating all n! and filtering through a set — the whole point is never materializing the duplicate subtrees."
Model answer
"Worst case O(n!): the first row has n column choices, the second at most n−1 that don't conflict, then at most n−2, and so on — the product is bounded by n!. Each placement's conflict check is O(1) with three hash sets — columns, r−c diagonals, r+c diagonals. And no, pruning does not change the Big-O. Big-O is a worst-case bound, and I can't prove pruning eliminates a fraction of the tree that grows with n in the worst case. The honest statement is: still O(n!), but pruning makes typical instances feasible. Hand-waving 'it's fast because of pruning' is exactly the answer a senior interviewer is fishing to catch — I'd rather state the true bound and then explain why practice beats it."
Follow-up answer
"Subtree elimination. A conflict detected at row 2 kills every completion of that prefix — for n=8 that's on the order of 6! ≈ 720 board completions never examined, from one O(1) check. Because pruning fires near the root where subtrees are largest, the explored tree is a sliver of the full n! tree — that's why 8-Queens finds all 92 solutions instantly despite the factorial bound. Pruning changes the constant and the typical case, sometimes by orders of magnitude; it just can't move the worst case, because an adversarial instance can defeat every prune."
Model answer
"Missing-and-malformed with no exception is the signature of shared mutable state not being restored. I check three places, in order. One: does every mutation before the recursive call have its exact mirror after it — path.push/path.pop, used.add/used.delete, sum increment/decrement? A missed undo means the next sibling branch explores with a stale element still marked, so it skips valid options — that's the missing output. Two: is there an early return or continue between choose and un-choose that jumps past the undo? That leaks state only on the code path that triggers it — which is exactly why only some inputs fail. Three: is the record step storing a copy, or a live reference that later branches keep mutating — that's the malformed output. The discipline that prevents all three: choose and un-choose adjacent to the recursive call, nothing between explore and undo."
Follow-up answer
"Copying does eliminate the bug class — each call owns its state, nothing to restore. The cost: O(n) copy per node instead of O(1) push/pop, multiplied across an exponential number of nodes, plus O(n) memory per stack frame instead of one shared structure. For subsets that's an extra factor of n over 2ⁿ nodes. It's a legitimate trade — in concurrent search or persistent-data-structure languages it's the default — but in an interview I mutate-and-undo and say why: the undo discipline is cheap once you're rigorous about it, and the copy cost is real. Mentioning the trade-off, then choosing deliberately, is the senior move."
