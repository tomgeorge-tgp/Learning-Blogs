Dynamic Programming
Patterns

Turning exponential recursion into polynomial time by remembering what you've already computed.

Press F to toggle

The Core Definition

Dynamic programming applies when a problem has both of these properties — and only then:

  • Optimal substructure. The optimal solution to the whole problem can be assembled from optimal solutions to its subproblems. "Best way to reach step n" is built from "best way to reach step n−1" and "best way to reach step n−2" — you never need a suboptimal answer to a subproblem to build the global optimum. If a locally optimal subproblem answer can poison the global answer (longest simple path in a graph is the classic case), DP doesn't apply.
  • Overlapping subproblems. A naive recursive solution recomputes the same subproblems many times. Naive fib(n) spawns O(2ⁿ) calls, but there are only n distinct subproblems — the exponential blowup is pure recomputation. This is exactly what distinguishes DP from divide-and-conquer: merge sort's subproblems (left half, right half) are disjoint, so caching them buys nothing. DP is precisely "divide and conquer, plus a cache, on a problem where the cache actually hits."

Once both hold, the entire technique is: identify the distinct subproblems, compute each exactly once, and store the result. Two mechanical strategies exist — same recurrence, opposite execution order:

Top-down (memoization)Bottom-up (tabulation)
Execution order Start at the full problem, recurse down; cache each result on the way back up. Only subproblems actually reachable from the top get computed. Start at the base cases, fill a table upward in dependency order until you reach the answer. Computes every table cell, needed or not.
How you derive it Directly from the recursive brute force — write the plain recursion, then add a cache check at the top of the function. Lowest-friction path from "correct but exponential" to "correct and polynomial". Requires you to know the dependency order up front (which cells does dp[i][j] read?). Slightly more design work, but the loop structure makes cost and dependencies explicit.
Space & practicality Cache = O(#states), plus O(recursion depth) call stack — deep inputs can blow the stack. Recursion/dictionary overhead adds constant-factor cost. No recursion, no stack risk, tight loops. Because you control fill order, collapsing the table (keep only the last row / last two values) is visible and easy — this is where space optimization lives.
Prefer when State space is huge but sparsely visited; recurrence is irregular; you're deriving under interview pressure and want correctness first. All (or most) states are needed anyway; input is large enough that stack depth matters; you want O(1)/O(n) space.

The interview-safe framing: "Memoization and tabulation execute the same recurrence — one is demand-driven with a cache, the other is schedule-driven with a table. I'll derive top-down because it falls out of the brute force, then convert to bottom-up if we want to optimize space."

Plain English: Diffing a Resume Against a Job Description

In applyGOAT you compare a candidate's resume against a job description and quantify how much needs to change. Strip that to its essence: given two sequences, what is the minimum number of single-element edits — insert, delete, substitute — to turn one into the other? That is exactly the edit distance (Levenshtein) problem, and it's the canonical 2D DP. Take a tiny real case: the resume says node, the JD says nest. How far apart are those tokens?

Walk the five construction steps — this checklist is the whole skill; the code is a transcription of it:

Step 1 — Define the state, in words, before anything else. dp[i][j] = the minimum number of edits to convert the first i characters of the resume string into the first j characters of the JD string. Prefixes, not suffixes; counts, not booleans. Say this sentence out loud in an interview — every later index decision follows from it, and both properties hold: optimal substructure (the cheapest way to convert prefix i→j extends a cheapest smaller conversion) and overlap (many alignment paths pass through the same (i, j) prefix pair).

Step 2 — The recurrence. Look at the last characters, a[i-1] and b[j-1]:

if a[i-1] == b[j-1]:
    dp[i][j] = dp[i-1][j-1]                  // free — nothing to fix
else:
    dp[i][j] = 1 + min(
        dp[i-1][j],      // delete a[i-1] from the resume string
        dp[i][j-1],      // insert b[j-1] into the resume string
        dp[i-1][j-1]     // substitute a[i-1] → b[j-1]
    )

Three options because there are exactly three ways an optimal edit script can handle the final character pair. Optimal substructure is what licenses the min: whichever option wins, the remaining prefix conversion inside it must itself be optimal.

Step 3 — Base cases. dp[i][0] = i (converting i characters to an empty string = i deletions) and dp[0][j] = j (empty to j characters = j insertions). Note the base cases fall out of the state definition — if step 1 is precise, step 3 is free.

Step 4 — Fill order. dp[i][j] reads up, left, and up-left, so row by row, left to right, works.

Step 5 — Where's the answer? dp[m][n] — full resume string into full JD string. Bottom-right cell.

Now execute by hand for node → nest:

""nest
""01234
n10123
o21123
d32223
e43233

Answer: dp[4][4] = 3. Sanity check: node → substitute o→e, d→s, e→t → nest. Three edits, matching the table. Every cell was computed once — O(m·n) time on two strings whose naive recursive comparison would branch three ways per character.

Reconstructing the actual edit script (the follow-up interviewers love): the table stores costs, not choices — but the choices are recoverable. Start at dp[m][n] and walk backwards: at each cell, ask which neighbor produced it. If characters matched and the value equals the diagonal, move diagonally with no edit; otherwise the neighbor achieving value − 1 tells you the operation (diagonal = substitute, left = insert, up = delete). Record operations along the walk, reverse, done — O(m + n) extra work. In applyGOAT terms, that backtrace is the diff view: not just "3 changes needed" but which changes. One caveat that matters below: reconstruction requires keeping the full table, which conflicts with space optimization — say that trade-off out loud before an interviewer raises it.

Your Patterns, Mapped

ProblemState definition (in words)Recurrence shapeSpace-optimized
Climbing Stairs dp[i] = number of distinct ways to reach step i dp[i] = dp[i−1] + dp[i−2] (count: sum the choices) O(n) time, O(1) space — two variables
House Robber dp[i] = max loot obtainable considering houses 0…i dp[i] = max(dp[i−1], dp[i−2] + v[i]) (optimize: max over choices) O(n) time, O(1) space — two variables
Longest Common Subsequence dp[i][j] = LCS length of first i chars of A and first j chars of B Match → dp[i−1][j−1] + 1; else max(dp[i−1][j], dp[i][j−1]) O(m·n) time, O(min(m, n)) space — keep two rows
Edit Distance dp[i][j] = min edits converting first i chars of A into first j chars of B Match → dp[i−1][j−1]; else 1 + min(up, left, diagonal) O(m·n) time, O(min(m, n)) space — keep two rows

Notice the split: counting problems sum over choices, optimization problems take min/max over choices — the state-definition discipline is identical. And every 1D entry collapses to O(1) space, every 2D entry to one-row space, for the same structural reason: the recurrence only ever reaches back a fixed distance.

Cross-project check — applyGOAT The toy above scales to the real feature. Token-level edit distance over resume bullet points vs JD requirements (sequences of words, not characters) gives you a "how much must change" score per section — and the backtrace gives the actual rewrite suggestions. Two production notes: (1) O(m·n) per pair is fine for documents but not for scoring one resume against thousands of JDs — that's why lesson 10's embedding search does the coarse retrieval and edit distance only runs on the shortlist; coarse-then-exact is the same two-tier move as an index scan followed by a heap filter. (2) If you only need "is the distance ≤ k", you can restrict computation to a diagonal band of width ~2k+1 in the table — O(k·min(m,n)) instead of O(m·n) — because any path straying further off-diagonal already costs more than k.
Cross-project check — AI Sales Agent Aligning a live conversation's turn sequence against a playbook's expected intent sequence ("greeting → discovery → objection → close") is the same alignment problem: insertions are off-script detours, deletions are skipped steps, substitutions are mismatched intents. Sequence alignment via DP is how you'd score "how far off-script did this call go" without brittle exact matching.

The 5-Step Checklist Applied — House Robber, End to End

Houses in a row, each with a value; rob any subset but never two adjacent; maximize loot. Run the checklist cold, in order:

1. State. dp[i] = the maximum loot obtainable from houses 0…i (whether or not house i itself is robbed — the state covers the prefix, not a commitment about house i). This wording matters: define it as "max loot ending with a robbery at i" instead and you get a different (also valid) recurrence with a different final-answer location — pick one meaning and hold it.

2. Recurrence. For house i, two choices: skip it (keep dp[i−1]) or rob it (its value plus the best over houses 0…i−2, since i−1 is now forbidden):

dp[i] = max(dp[i-1], dp[i-2] + value[i])

Optimal substructure check: if we rob house i, the loot from 0…i−2 must itself be optimal — swapping in a better prefix answer can never break the adjacency constraint at i. Overlap check: the naive recursion branches two ways per house over the same shrinking prefixes — fib-shaped, O(2ⁿ) without a cache, n distinct states with one.

3. Base cases. dp[0] = value[0]; dp[1] = max(value[0], value[1]) (adjacent, pick one).

4. Fill order. dp[i] reads i−1 and i−2 → left to right.

5. Answer location. dp[n−1] — the state definition says "prefix 0…i", so the full prefix is the answer.

function rob(values) {
  const n = values.length;
  if (n === 1) return values[0];
  const dp = new Array(n);
  dp[0] = values[0];
  dp[1] = Math.max(values[0], values[1]);
  for (let i = 2; i < n; i++) {
    dp[i] = Math.max(dp[i - 1], dp[i - 2] + values[i]);
  }
  return dp[n - 1];   // O(n) time, O(n) space
}

Space optimization — the standard senior follow-up

The loop only ever reads dp[i−1] and dp[i−2]. The rest of the array is dead weight kept alive by habit. Replace it with two rolling variables:

function rob(values) {
  let prev2 = 0, prev1 = 0;          // best up to i-2, best up to i-1
  for (const v of values) {
    const cur = Math.max(prev1, prev2 + v);
    prev2 = prev1;
    prev1 = cur;
  }
  return prev1;   // O(n) time, O(1) space
}

The general principle: look at what the recurrence reads. If dp[i] reads a fixed window of previous values → O(1) space. If dp[i][j] reads only row i−1 (LCS, edit distance, most grid problems) → keep two rows, O(min(m, n)) by iterating over the shorter string. This is why tabulation earns its keep: the fill order makes the dependency window visible, and the optimization is a mechanical rewrite. Two caveats to volunteer: collapsing the table destroys the backtrace (no reconstruction of the actual path/edit script), and memoized top-down can't do this trick at all — the cache must retain every state because recursion revisits them in no predictable order.

Common failure modes (1) Recurrence before state. Writing transitions before you can say what dp[i] means in one sentence is where every off-by-one lives — "first i characters" vs "index i" vs "ending at i" are three different tables. (2) DP-shaped code with no cache. Recursing on the right subproblems but forgetting memoization is still O(2ⁿ) — the recurrence gets zero credit without the store. (3) Forcing DP where substructure is absent. Longest simple path, or any problem where a subproblem's optimum can conflict with global constraints (the visited-set makes subproblems non-independent) — that's lesson 15's territory: backtracking, not DP. The two preconditions are a real admission test, not a formality.

Model answer
"This is edit distance — a 2D DP. First the state: dp[i][j] is the minimum edits to convert the first i characters of A into the first j characters of B. Recurrence by cases on the last characters: if A[i−1] equals B[j−1], dp[i][j] = dp[i−1][j−1], the pair is free; otherwise it's 1 + min of three options — dp[i−1][j] for deleting from A, dp[i][j−1] for inserting into A, dp[i−1][j−1] for substituting. Base cases fall out of the state definition: dp[i][0] = i, all deletions; dp[0][j] = j, all insertions. Each cell reads up, left, and up-left, so I fill row by row, and the answer is dp[m][n]. O(m·n) time and space. Before coding I'd verify both DP preconditions: optimal substructure — an optimal edit script's prefix handling must itself be optimal — and overlapping subproblems — the naive three-way recursion revisits the same (i, j) prefix pairs exponentially often."
Follow-up answer
"Row i only reads row i−1, so keep two rows — O(min(m, n)) space by iterating over the shorter string. But that's a trade-off, not a free win: reconstructing the edit sequence needs a backtrace from dp[m][n], walking to whichever neighbor produced each cell — diagonal means match or substitute, left means insert, up means delete — and that walk needs the full table, O(m·n) space, though the backtrace itself is only O(m + n) time. So: distance only → two rows; actual diff → full table. There's a divide-and-conquer refinement (Hirschberg's algorithm) that recovers the alignment in linear space if the interviewer pushes, but I'd name it rather than derive it."
Model answer
"Two tests. First, optimal substructure: can I build the optimal answer from optimal answers to smaller subproblems? Greedy needs this too — the difference is greedy additionally requires that one locally best choice is provably safe without examining alternatives, while DP hedges by evaluating every choice and taking min/max. If I can't prove the greedy exchange argument, DP is the safe default. Second, overlapping subproblems: does the naive recursion recompute the same states? If subproblems are disjoint — merge sort's halves — it's divide and conquer, and a cache never hits; if they overlap, caching converts exponential to (number of states) × (transition cost). Signal words: 'minimum/maximum cost', 'number of ways', 'longest/shortest subsequence', decisions over prefixes with a small state footprint. And a negative test: if a subproblem's answer depends on the whole path taken to reach it — longest simple path with its visited set — substructure fails, and it's backtracking territory instead."
Follow-up answer
"Same recurrence either way, so it's an engineering call. I derive top-down first — brute-force recursion plus a cache check is the shortest path to correct, and it only computes reachable states. I convert to bottom-up when the input is deep enough to threaten stack limits, when essentially all states get visited anyway so recursion overhead is pure loss, or when I want the space optimization — tabulation's explicit fill order exposes that each row reads only the previous one, which the memoized version's unpredictable visit order can never exploit."
Model answer
"Concrete counterexample: coins {1, 3, 4}, amount 6. Greedy takes 4, leaving 2, then 1 + 1 — three coins. Optimal is 3 + 3 — two coins. Taking the biggest coin committed us to a remainder (2) whose best decomposition is bad, and greedy has no mechanism to reconsider. DP fixes exactly this: define dp[a] = fewest coins to make amount a, with dp[a] = 1 + min over coins c of dp[a−c], base dp[0] = 0, filled from 0 upward, answer at dp[amount]. It doesn't guess which coin is best — it evaluates all three continuations of every subamount and keeps the min. O(amount × #coins). The structural lesson: this problem has optimal substructure, but it does not have the greedy-choice property — the locally largest coin isn't provably part of an optimal solution. Those are separate requirements, and coin change is the cleanest demonstration that the second doesn't follow from the first."
Follow-up answer
"Because {1, 5, 10, 25} is a canonical coin system — each denomination relates to the others such that the greedy exchange argument goes through: any solution using smaller coins where a larger one fits can be exchanged into one that takes the larger coin without getting worse. That's a property of the specific denomination set, not of coin change in general — add a 21-cent coin and greedy breaks again (63 = 21×3, but greedy says 25 + 10×3 + 5 + 1×3, eight coins). The senior takeaway: greedy correctness is always a theorem about the input structure, never a default assumption. If I can't sketch the exchange argument in a minute, I write the DP."
