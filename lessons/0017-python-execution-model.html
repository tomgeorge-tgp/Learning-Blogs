Python Runtime Internals

Python Execution Model:
Bytecode & the CPython VM

What actually happens between "def" and a running program — tokenizer, AST, compiler, bytecode, frames, and the object model underneath.

Press F to toggle

The Core Definition

"Python is interpreted" is imprecise. CPython (the reference implementation, and the one running your FastAPI services) compiles source to an intermediate bytecode, then executes that bytecode on a stack-based virtual machine. Six stages, every time you run a .py file:

StageInput → OutputWhat it does
Tokenizer (lexer)Source text → token streamSplits def score(a, b): into tokens: NAME def, NAME score, OP (, NAME a, … No meaning attached yet, just symbols.
ParserTokens → ASTApplies Python's grammar to build an Abstract Syntax Tree — a nested structure of FunctionDef, BinOp, Return nodes. This is the same AST ast.parse() gives you.
CompilerAST → bytecodeWalks the AST and emits bytecode instructions (LOAD_FAST, BINARY_ADD, RETURN_VALUE, …) plus a code object holding constants, names, and metadata.
Cache (.pyc)Bytecode → diskCompiled bytecode is cached in __pycache__/*.pyc, keyed by source hash/mtime, so re-imports skip tokenizing/parsing/compiling.
Python Virtual Machine (PVM)Bytecode → executionA big switch loop (ceval.c) that reads one bytecode instruction at a time and mutates a per-call frame: an operand stack, local variables, and an instruction pointer.
Object modelRuntime valuesEvery value — int, string, your Pydantic model — is a PyObject*: a heap allocation with a type pointer and a reference count. This is what memory management (refcounting + GC) operates on.
See it yourself import dis; dis.dis(score) prints the exact bytecode the compiler emitted for a function. This is not a metaphor for what Python does — it's a direct dump of the instructions the VM executes. Any time you're unsure what an expression "actually does" (is a list comprehension really faster? does += mutate in place?), dis answers definitively.

Plain English — Tracing applyGOAT's Resume Scorer

applyGOAT has a plain function that scores a resume against a job description:

def score(matched: int, total: int) -> float:
    return matched / total if total else 0.0

Nothing here is magic — trace every stage. Tokenizing turns this into a flat stream: NAME(def), NAME(score), OP((), NAME(matched), OP(:), NAME(int)… — pure lexical symbols, no understanding of "function" yet. Parsing imposes Python's grammar on that stream and builds a tree: a FunctionDef node named score, containing an If node testing total, whose branches are a BinOp(Div) and a constant 0.0. Compiling walks that tree depth-first and emits stack-machine instructions:

  2           0 LOAD_FAST                1 (total)
              2 POP_JUMP_IF_FALSE        8
              4 LOAD_FAST                0 (matched)
              6 LOAD_FAST                1 (total)
              8 BINARY_TRUE_DIVIDE
             10 RETURN_VALUE
        >>    8 LOAD_CONST               1 (0.0)
             10 RETURN_VALUE

Now the PVM runs this, one instruction at a time, against a fresh frame created for this call. LOAD_FAST 1 pushes the local variable total's value onto the frame's operand stack. POP_JUMP_IF_FALSE pops it, tests truthiness, and either falls through or jumps to offset 8. BINARY_TRUE_DIVIDE pops the top two stack values and pushes their quotient — a brand-new PyObject* float, heap-allocated with refcount 1. RETURN_VALUE pops that object and hands it back to the caller's frame, and this function's frame is discarded. Every call to score() — thousands of times per resume-batch job — repeats exactly this: new frame, same bytecode, same object-allocation pattern. This is why hot paths in applyGOAT's scoring loop are sensitive to how many objects get allocated per call — every temporary float, every intermediate list, is a refcounted heap object the VM must create and eventually free.

Your Stack, Mapped to the Execution Model

Five places this lesson's pipeline shows up in code you actually run. Each row below first says the idea in plain words, then connects it to your stack.

ConceptWhere you actually touch itWhy it matters
.pyc caching
In plain words: Python saves its compiled bytecode to disk so it doesn't have to redo the tokenize→parse→compile work next time it imports the same file.		 Docker image builds. __pycache__ is just a folder Python creates next to your source files to hold these cached .pyc files. If your Dockerfile copies source code in a way that doesn't refresh this folder, a container can end up running old compiled bytecode even though the .py source on disk looks updated. "Cold-start latency" means: how long a freshly started container takes before it's ready to serve its first request. Skipping tokenize/parse/compile on every import after the first run is exactly what makes that startup faster — but only if the cache is trustworthy.
Frame objects
In plain words: a frame is the VM's record of "which function is running, on which line, with which local variables" for one function call.		 Every error log entry FastAPI prints when something crashes — that block of text showing file names, line numbers, and a chain of function calls — is called a traceback. It's literally a list of frame objects, one per function call, from where the crash happened all the way back to where the request started. pdb (Python's built-in interactive debugger) and the traceback module both work by walking this same chain of frames. Once you know a frame is just "one function call's snapshot," a scary-looking traceback becomes a readable list: function A called function B called function C, and C is where it broke.
Bytecode inspection
In plain words: printing out the exact low-level instructions the compiler generated for a function, so you can see what it really does instead of guessing.		 dis is a built-in module; dis.dis(my_function) prints that function's bytecode. Run it on a function you're about to optimize, before you change anything. It turns a guess into a fact. "This rewrite feels faster" is a guess; "this rewrite emits 3 fewer instructions per loop iteration" is something you can actually check in seconds with dis, instead of trusting a hunch about readability.
Object allocation cost
In plain words: creating any Python value — even a small one — costs real time and memory, because it means allocating a new object on the heap (see Lesson 17's object-model section above).		 Pydantic is the validation library FastAPI uses to check that incoming request data (and outgoing response data) matches the shape you declared. Every time a request comes in, Pydantic builds a full model instance — one object per field, plus validation logic running for each one. This is why "just add request validation" is not a free abstraction — a profiler (a tool that measures where a program actually spends its time) will show real CPU time spent inside Pydantic on every single request, because every field access and every validator is genuine object creation and function-call work, not something that happens "for free" at compile time.
Import system
In plain words: every time your code says import something, Python runs that module's file through the full tokenize→parse→compile→execute pipeline — but only the very first time; after that it reuses the result.		 When a FastAPI app starts up, every import statement across your whole codebase runs once. The result of each import gets stored in sys.modules — a dictionary, built into every running Python program, that acts as a cache of "modules already imported." The next time anything asks for that same module, Python just hands back the cached version instead of re-running the whole pipeline. A codebase with a lot of heavy dependencies means a lot of modules to compile and run before the app can accept its first request — that's startup time you pay once, up front. "Lazy imports" means moving an import statement from the top of a file into the specific function that needs it, so that dependency is only loaded (and only costs time) if that function actually gets called — sometimes worth doing for a rarely-used, heavy dependency.
Cross-project check — AI Sales Agent

In plain words first: LangGraph is a library for building an AI agent as a graph of steps — each step is called a node, and a graph executor is the code that calls each node function in the right order, threading a shared "state" object through them. Strip away the framework naming, and a node is nothing more than a Python function that gets called — so it goes through the exact same tokenize→AST→bytecode→frame pipeline as any other function call in this lesson.

The part worth getting precisely right — including where a common mental model overshoots — is how a node's return value actually updates that state. The idiomatic pattern is: a node does not receive the state, mutate it, and hand the same object back. It receives the current state and returns a small dict containing only the keys it's changing — a partial update, not the whole state. The executor then merges that partial dict into the shared state, one key at a time, using whatever reducer is declared for each key:

  • No reducer declared (the default) — the value a node returns for that key replaces whatever was there. This is a plain overwrite, not an append.
  • A reducer declared (e.g. a message-history field annotated Annotated[list, add_messages]) — the returned value is merged/appended according to that reducer's logic instead of replacing the field outright.

So "the node returns a dict and it gets appended to state" is correct for the classic example — conversation history is almost always a reducer-annotated field, specifically so each node's new message accumulates instead of erasing the transcript so far. But it's a property of that field's declared reducer, not a universal rule: a plain field with no reducer gets overwritten by whatever a node returns for it, every time.

Where the object-model connection from this lesson still matters: the state object a node receives genuinely is a live Python object — the same PyObject, same heap allocation, same refcount bookkeeping as anything else in this lesson — not a fresh deep copy manufactured for each node call. But the idiomatic contract is that a node reads from it and returns updates, rather than mutating it directly and handing the same object back:

def node_one(state: dict) -> dict:
    return {"intent": "objection"}   # a partial update, not the whole state

# The executor merges this into shared state:
#   - "intent" has no reducer declared  -> overwritten with "objection"
#   - "messages" has add_messages       -> new message appended, not replaced

Mutating the input dict in place instead (state["intent"] = "objection"; return state) is technically possible, purely because Python hands every node a reference to the same shared object rather than a copy — but it's an anti-pattern in LangGraph specifically, because it bypasses the reducer/merge logic entirely, which is also what LangGraph's checkpointing (persisting state between steps, for replay/resumption) and parallel-branch execution rely on to track what changed. The bug this actually causes in practice isn't "node 2 sees a surprise mutation" — it's simpler and more common: forgetting to declare a reducer on a field you meant to accumulate (like history or collected evidence), so each node's return value silently overwrites the previous one instead of building on it.

The Complete Mental Model: Frames, Exceptions & the Traceback Walk

The "Frame objects" row above says a traceback is "a chain of frame objects." Here's that idea made concrete, end to end, using applyGOAT's own login path: a request calls login(), which calls authenticate(), which calls load_user() — and load_user() fails.

Every arrow in that diagram is a real object reference, not a metaphor. Each Frame(...) box is an actual frame object, created the instant its function is called and holding that function's locals, its position in the bytecode, and — critically — a pointer back to the frame that called it. When load_user() raises, the exception object Python constructs carries a __traceback__ attribute pointing at the frame where the failure happened. From there, walking "backward" is just following each frame's caller-pointer: load_user → authenticate → login → the request handler that started it all. The traceback printer does nothing more than walk that chain once and print each frame's file, line number, and function name, innermost or outermost first depending on Python version.

Once you see a traceback this way, it stops being a wall of text and becomes a narrative you already know how to read: "the request handler called login, which called authenticate, which called load_user, and load_user is where it broke." Every extra frame in a long traceback is just one more "called by" step in that same sentence — nothing about it is special-cased or hidden from you; it's the exact same frame-chaining this lesson's execution-model pipeline builds on every single call, just captured and printed at the one moment things went wrong.

Why this matters in practice In a FastAPI error log, the traceback for a failed /login request in applyGOAT is this exact chain, just with real file paths and line numbers instead of function names alone. Reading from the bottom of the traceback up tells you where it failed (load_user, last frame); reading from the top down tells you how execution got there (the request handler → login → authenticate). Debugging is faster once you stop reading a traceback top-to-bottom as undifferentiated text and instead ask "which frame is the failure, and what's the calling chain that reached it?"

The Object Model: refcounting, small-int caching, string interning

In plain words first: every single value in Python — a number, a string, a list, your own class instances — is stored the same way underneath: as a PyObject, a chunk of memory on the heap (the pool of memory the program can allocate from as needed, as opposed to the fixed, function-call-scoped stack). Every PyObject carries two things at minimum: a type pointer (which says "I am an int" / "I am a string" / etc.) and a reference count — a running total of how many places in the program currently have a name pointing at this exact object.

That reference count (refcount for short) is the whole mechanism. a = b = "hello" does not create two separate strings — both names a and b point at the one string object in memory, and its refcount becomes 2. When a name is reassigned to something else, or the function it lived in returns, that refcount goes down by one. The moment a refcount hits zero — nobody, anywhere, still has a name pointing at it — Python frees that memory immediately, on the spot. This is different from languages that only use a "tracing garbage collector" (a background process that periodically scans all memory looking for anything nobody references anymore) — Python's refcounting means most memory is freed the instant it's no longer needed, not on some later sweep.

Two real optimizations are built on top of this basic mechanism:

  • Small integer caching. CPython pre-creates one shared object for every integer from −5 to 256 and reuses it everywhere, instead of creating a fresh object each time. So a = 100; b = 100; a is b is True — a and b really are the same object in memory. But a = 1000; b = 1000; a is b is usually False — outside that small cached range, each 1000 becomes its own separate heap object, even though they're equal in value. (is asks "are these the exact same object in memory?" — a much stricter question than ==, which asks "do these have the same value?" This gap is exactly why relying on is to compare numbers is a bug waiting to happen — always use == for values, and save is for checks like is None.)
  • String interning. Any string literal that looks like it could be an identifier (a variable or function name — no spaces, no punctuation) gets deduplicated into one shared object at compile time, the same way small integers are shared. The payoff: comparing two interned strings can often be answered with a quick "are these the same memory address?" check (a pointer comparison — comparing two addresses, which is nearly free) before ever falling back to comparing the strings character by character.
a = 100
b = 100
print(a is b)          # True — guaranteed. Both land in the -5..256 cache,
                        # so a and b are provably the same object, every time.

a = 1000
b = 1000
print(a is b)           # Usually False at the top level of a script/module —
                        # 1000 is outside the cache, so each literal typically
                        # becomes its own separate heap object.

def same_block():
    a = 1000
    b = 1000
    return a is b

print(same_block())     # Often True! Both 1000 literals appear together inside
                         # one function body, and CPython's compiler can fold
                         # identical constants within a single code object into
                         # one shared value — a compiler optimization, not a
                         # language guarantee.

This is the actual, honest answer to "what does a = 1000; b = 1000; print(a is b) print?" — it depends, and you should never bank on either outcome. Whether the two 1000 literals get folded into one shared object depends on implementation details like where in the code they appear (same function vs. separate top-level statements vs. the interactive REPL), which CPython version you're running, and whether the compiler's constant-folding pass happens to notice they're identical. None of that is part of the language specification — it's an optimization detail that's free to change. The only line that's ever safe to write is the one that asks the actual question you mean: a == b for "do these have the same value," never a is b for numbers. Save is for singleton checks like x is None, where there is genuinely only one such object in the entire running program by design.

a = "hello"
b = "hello"
a is b            # True  — both identifier-shaped literals, interned to the same object

c = "hello world"
d = "hello world"
c is d            # Usually False — contains a space, not identifier-shaped, so it isn't
                   # auto-interned; two separate heap objects that happen to be equal

e = "".join(["hel", "lo"])   # built at runtime, not a literal the compiler ever saw
e is a             # False — even though e == "hello", it was never deduplicated
                   # against the interned "hello" object created for `a`

import sys
f = sys.intern("hel" + "lo")   # force interning by hand at runtime
f is a             # True — sys.intern() looks up (or registers) the shared object,
                   # so a runtime-built string can be made to match the cache too

Notice the shape of the trap: a is b being True for two identical short literals is a genuine memory optimization, not a language guarantee — it works because both literals happened to be compiled together and both look like identifiers. The moment a string is built at runtime (concatenation, formatting, reading from a file or a request body), it isn't interned automatically, and is silently starts returning False for values that are still equal. This is exactly the small-integer trap from the bullet above, wearing a string costume: == is the only correct tool for comparing string values; is only tells you whether two names happen to point at the interpreter's one cached copy.

Refcounting has one real blind spot: cyclic garbage — object A holds a reference to object B, and B holds a reference back to A. Even if nothing else in the whole program points at either one, each object's refcount is still 1 (from the other), so neither ever reaches zero, and refcounting alone would leak that memory forever. This is exactly why CPython also ships a second, separate mechanism: the generational garbage collector (the gc module) — a background process that periodically scans specifically for these reference cycles and breaks them. It's a backstop for the one case plain refcounting can't handle, not a replacement for it. Concretely:

class Node:
    def __init__(self):
        self.parent = None
        self.children = []

root = Node()
child = Node()
root.children.append(child)   # root --> child
child.parent = root           # child --> root : a reference cycle

del root
del child
# Neither object's refcount ever hits zero — each still holds one
# reference from the other. Only the generational GC's cycle
# detector will eventually notice both are unreachable and free them.

Circular references between ORM (Object-Relational Mapping — the layer, like SQLAlchemy, that represents database rows as Python objects) model instances and their own cached relationships are the most common place this bites production backend code: a parent row object referencing its children, and each child referencing its parent right back.

Preview — forward reference Refcounting has a second, sharper consequence beyond memory management: it's the reason CPython needs the Global Interpreter Lock (GIL) at all. Incrementing or decrementing a refcount has to happen atomically — meaning it can never be half-done and interrupted, even when multiple threads are running — because two threads changing the same object's refcount at the same moment could corrupt it. Rather than adding a separate lock to every single object in the entire program, CPython uses one single global lock around bytecode execution to guarantee this safety. That trade-off — one lock, simple and fast, but no true multi-core parallelism for Python code — is the subject of the next lesson.

Model answer
"Both, depending on what you mean by 'compiled.' CPython compiles source to bytecode — an intermediate, platform-independent instruction set — via tokenizing, parsing to an AST, then a compiler pass. That bytecode is then interpreted by the Python Virtual Machine, a stack-based bytecode interpreter written in C. So there's no interpretation of raw source text at runtime, but there's also no compilation to native machine code — the VM decodes and dispatches each bytecode instruction on every execution. That's the meaningful distinction from a fully interpreted language (re-parsing text each run) and from an ahead-of-time compiled language like C (producing native instructions)."
Follow-up answer
".pyc files hold the compiled bytecode plus a header (magic number, source hash/mtime) so the import system can skip tokenizing/parsing/compiling on subsequent imports if the source hasn't changed. It's not compiled in the C sense because it's not native machine code — it's still an abstract instruction set that the PVM interprets at every run. Deleting __pycache__ costs you re-compilation time on next import, not correctness — the .py source is still the source of truth."
Model answer
"I wouldn't take it on faith — I'd run dis.dis() on both versions and compare the emitted bytecode, then benchmark with timeit on representative data. List comprehensions typically compile to a specialized LIST_APPEND loop with less per-iteration overhead than an explicit for loop calling .append() as a method lookup each time — that's a real, measurable difference in instruction count per iteration, not folklore. But 'always faster' is the part I'd push back on: it depends on what's inside the loop."
Follow-up answer
"If the loop body does something with side effects beyond building a list — writing to a database, logging, mutating shared state — you can't express it as a comprehension without contorting the code, and forcing it often makes it both slower and unreadable. Also, if the loop body dominates the cost (an API call, a regex match), the fixed per-iteration bytecode savings of a comprehension are noise next to the actual work — dis and timeit both, never just one, because instruction count and wall-clock time can diverge once I/O or C-extension calls dominate."
Model answer
"A frame is the PVM's per-call execution context — an operand stack for intermediate values, a locals array/dict, a reference to the code object being executed, and an instruction pointer tracking which bytecode instruction runs next. Every function call, comprehension, and generator resumption creates or reuses a frame. A traceback is literally a linked chain of frames from the exception site back to the entry point — each line in a traceback is one frame's file, line number, and function name at the moment the exception propagated through it."
Follow-up answer
"Usually yes, immediately — when a function returns, its frame's reference to each local goes away, and if that was the only reference (the common case for locals that don't escape via closure or return value), the object's refcount hits zero and it's freed right there, deterministically, no GC pause required. The exception is anything that outlives the frame: values returned to the caller, values captured by a closure, or values referenced by a generator that's paused mid-execution — a generator's frame is kept alive between yields specifically so its locals survive across resumptions."
