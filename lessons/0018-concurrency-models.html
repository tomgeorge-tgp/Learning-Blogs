Concurrency Models

Concurrency Models:
Threads, Processes, the GIL & AsyncIO

Why "just use threads" is the wrong answer more often than not — matching the concurrency model to whether work is CPU-bound or I/O-bound.

Press F to toggle

The Core Definition

In plain words first: a process is a running program with its own private chunk of memory — when you start python app.py, that's one process. A thread is a separate "line of execution" that lives inside a process and shares that process's memory with any other threads in it — cheaper to create than a whole new process, but nothing stops two threads from stepping on the same piece of shared memory at the same time unless something prevents it. A lock (a mutex, short for "mutual exclusion") is exactly that something: a flag one thread can hold so that only it, and nothing else, is allowed to run the code the lock guards, until it lets go.

CPython — the actual Python you run — has exactly one lock that governs its whole concurrency story: the Global Interpreter Lock (GIL). It's "global" because there's a single one for the entire process, not one per object or per piece of code. The rule it enforces is blunt: at most one thread executes Python bytecode at a time, per process, ever — regardless of how many CPU cores the machine has sitting idle. Two threads in the same Python process can never both be running Python code at the exact same instant; they take turns holding the GIL.

TermDefinitionWhat it actually buys you
GIL
In plain words: the one lock, shared by the whole process, that only lets one thread run Python code at a time.		 A mutex around the bytecode interpreter loop (the same loop from Lesson 17 that reads and executes bytecode instructions one at a time) — only the thread currently holding this lock is allowed to execute Python bytecode; every other thread in that process has to wait its turn. Makes refcount increments/decrements (Lesson 17's reference-counting mechanism) safe without needing a separate lock on every single object in the program. The cost of that safety: bytecode execution is serialized — done one thread at a time, never truly simultaneously, no matter how many CPU cores exist.
Threading (threading)
In plain words: several threads inside one process, all sharing the same memory, passing the one GIL back and forth between them.		 Multiple threads in one process, sharing memory, taking turns holding the GIL — Python's built-in threading module is how you create and manage them. Real concurrency, but only for I/O-bound work — work that spends most of its time waiting (on a network response, a disk read), not computing. The GIL is released while a thread is blocked waiting on something external, so other threads get to run Python code during that wait. It does not help CPU-bound work — work where the thread is doing continuous computation with nothing to wait on (adding up numbers in a loop, hashing a file, resizing an image). A thread like that never hits a point where it lets go of the GIL on its own, so it holds the lock for its whole run, and a second thread doing the same kind of computation just has to sit and wait its turn instead of running alongside it.
Multiprocessing (multiprocessing)
In plain words: several entirely separate Python programs running side by side, each with its own memory and its own GIL, talking to each other by passing messages.		 Multiple OS processes (each a fully independent running copy of the Python interpreter, with its own memory — not shared like threads), communicating by pickling (converting a Python object into bytes so it can be sent somewhere else, then rebuilding it on the other end) data through pipes, queues, or shared memory. True parallelism — actual simultaneous execution — for CPU-bound work — work where the bottleneck really is computation (parsing, hashing, image processing). Because each process has its own separate GIL, four processes can use four CPU cores at the same literal instant; four threads in one process never can.
AsyncIO (asyncio)
In plain words: one single thread that juggles many waiting tasks at once, switching between them only at points the code explicitly marks as "I might be waiting here."		 A single thread running an event loop (a scheduler that decides which piece of code runs next) that switches between coroutines (functions written to pause and resume, defined with async def) at explicit await points — the exact spots in the code where it's safe to say "pause me here and go run something else while I wait." Massive I/O-bound concurrency — potentially thousands of pending tasks — with far less overhead than threads: no OS-level thread switching (which costs real time and memory), no fighting over the GIL between threads, and switching is cooperative (code chooses exactly when to yield control, at an await) rather than preemptive (the OS forcibly interrupting a thread mid-instruction, the way OS-level thread scheduling works).
The one distinction that decides everything CPU-bound = the bottleneck is computation (parsing, hashing, image processing, ML inference on CPU). Only multiprocessing gives you real parallelism here. I/O-bound = the bottleneck is waiting (network calls, disk reads, database queries). Threading and asyncio both work here because the GIL is released while waiting — but asyncio wins on overhead at scale. Get this classification wrong and you'll add threads to a CPU-bound problem and watch throughput barely move, because the GIL was never the thing letting you down — the CPU was always fully used by one core.

Plain English — applyGOAT's Resume Upload Pipeline

A user uploads a resume PDF. The pipeline has two very different phases, and applyGOAT uses a different concurrency model for each — this contrast is the whole lesson in miniature.

Phase 1 — Extract text from the PDF. This is CPU-bound: parsing PDF structure, decoding fonts, running OCR if it's a scanned image. If you fan this out across threading.Thread workers, you get almost no speedup — every thread still fights for the same GIL to execute Python bytecode, and PDF parsing spends its time computing, not waiting. The fix is multiprocessing (or a process pool via concurrent.futures.ProcessPoolExecutor): each worker process gets its own interpreter, its own GIL, and genuinely runs on a separate core. Four processes on four cores parses four resumes in parallel, for real.

Phase 2 — Call the embedding API and write to Postgres/Pinecone. This is I/O-bound: the CPU is idle while waiting for OpenAI's API and the database round-trip. Here, asyncio is the right tool — a single event loop can have hundreds of these requests in flight simultaneously, because "waiting for a network response" doesn't require CPU attention. Threading would technically also work (the GIL releases during the socket wait), but each OS thread costs real memory (default ~8MB stack) and the OS scheduler overhead of switching between hundreds of threads dwarfs asyncio's near-zero-cost coroutine switches. This is exactly why FastAPI's async def route handlers scale to thousands of concurrent connections that a thread-per-request model never could.

Naming the pattern precisely "I used asyncio to make it faster" is vague and will get probed. The precise version: "Phase 1 is CPU-bound, so I parallelized it with a process pool to actually use multiple cores past the GIL. Phase 2 is I/O-bound, so I used asyncio to hold hundreds of pending API/DB calls in one thread without the memory and scheduling cost of one OS thread per request." That sentence is the entire lesson, compressed — and it's what a senior interviewer is listening for.

Why Does the GIL Actually Exist? (Refcounting Is Only Part of It)

The Core Definition table above gives the most commonly cited reason for the GIL — refcount safety — but that's one of at least four real reasons it exists and has survived largely unchanged for three decades:

  1. Reference counting safety. As covered above: refcount increments/decrements must be atomic, and one global lock is the simplest way to guarantee that without a lock on every object.
  2. Interpreter implementation simplicity. It's not only object refcounts — every mutable internal structure inside the interpreter (how a dict resizes, how a list grows) would need its own careful locking without a GIL. Getting fine-grained locking right everywhere, with no deadlocks and no subtle races, is a genuinely hard engineering problem that a single global lock sidesteps entirely.
  3. C-extension ecosystem compatibility. Arguably the biggest reason the GIL has persisted this long. Decades' worth of C extensions — NumPy, and much of the ML/data stack built on it — were written assuming the GIL already protects them from concurrent access to Python objects. Removing the GIL breaks that assumption unless every extension author adds their own locking. That's a slow, ecosystem-wide coordination problem, not a fix CPython's own maintainers can make unilaterally.
  4. Single-threaded performance. A single lock is cheap for the overwhelmingly common case: a program using only one thread. Fine-grained per-object locking adds real, constant overhead — acquiring and releasing tiny locks — even when nothing is actually contended. An experimental 1990s patch that removed the GIL made ordinary single-threaded code roughly twice as slow, which is exactly the trade-off CPython's maintainers weren't willing to make for decades.

One more precision worth having ready in an interview: the GIL is a CPython implementation detail, not a requirement of the Python language itself. Jython (Python compiled to run on the JVM) and IronPython (on .NET's CLR) don't use reference counting at all — they lean on their host platform's own garbage collector for memory management — so reason #1 above simply doesn't apply to them, and neither has a GIL of CPython's kind. "Python has a GIL" really means "CPython — the implementation almost everyone actually runs — has a GIL," which is a meaningfully more precise thing to say.

GIL Release Points & the Free-Threaded Future

The GIL is released — allowing another thread to run — at specific, predictable points: blocking I/O calls (socket reads, file reads), and periodically during long-running pure-Python bytecode loops (a "check interval" forces a release so other threads aren't starved). Some C extensions (NumPy's array math, zlib compression, hashlib) explicitly release the GIL around their internal loop because that loop never touches Python objects — this is why "NumPy releases the GIL" is a real, specific, correct thing to say, not a general exception to the CPU-bound rule.

CPython 3.13 introduced an experimental free-threaded build (PEP 703) that removes the GIL entirely, using per-object fine-grained locking instead — replacing refcounting itself with a different, thread-safe scheme (reason #1 above), which is exactly why this took roughly 30 years: the hard part was never just "protect the refcount," it was reason #3 — getting the vast C-extension ecosystem to a place where removing the GIL wouldn't silently break everything built on top of it. It's not yet the default build, and most C-extension libraries (including many used in ML/data stacks) aren't yet compatible — worth naming as "aware of the direction, not yet relying on it in production" rather than claiming it's solved.

Model answer
"The GIL is a single mutex around the bytecode-execution loop — at most one thread executes Python bytecode at a time, no matter how many CPU cores are available. It exists primarily because CPython's memory management relies on reference counting, and refcount increments/decrements on shared objects need to be atomic. Rather than adding a lock to every single object (expensive, and easy to deadlock), CPython uses one global lock around interpretation. The trade-off: simpler, faster single-threaded execution and trivial C-extension integration, at the cost of no true multi-core parallelism for pure-Python CPU-bound code."
Follow-up answer
"Because the GIL is released during blocking I/O — file reads, socket calls, time.sleep. If one thread is waiting on a network response, it releases the GIL so another thread can run Python bytecode in the meantime. So threading gives real concurrency for I/O-bound workloads even though it gives zero parallelism for CPU-bound ones. That's precisely the distinction: threading overlaps waiting, it doesn't parallelize computing."
Model answer
"No — image resizing is CPU-bound, and asyncio doesn't add parallelism, it adds concurrency for waiting. Wrapping a synchronous CPU-heavy call in `async def` without ever awaiting anything inside it just runs it on the event loop thread, blocking every other coroutine in that process until it finishes — actually worse than doing nothing, because it stalls unrelated I/O-bound requests too. The right tool is a process pool — `concurrent.futures.ProcessPoolExecutor` — to spread the resize work across cores, each in its own process with its own GIL."
Follow-up answer
"It changes the threading answer, not the asyncio one. If the resize call is implemented in C and explicitly releases the GIL during its internal loop (many image libraries do, similar to NumPy), then `threading` — not asyncio — becomes viable for real parallelism, because multiple threads can be inside that C code simultaneously on separate cores while the GIL sits released. I'd verify this against the specific library's documentation rather than assume it, since 'GIL-releasing C extension' is a property of that specific function, not a general guarantee."
Model answer
"No — this is a common misconception. The GIL guarantees that individual bytecode instructions execute atomically, but most Python statements compile to multiple bytecode instructions, and the GIL can switch threads between any two of them. `counter += 1` is a read, an add, and a write — three separate opportunities for a thread switch in between. Two threads can both read the same old value before either writes back the increment, losing an update. The GIL prevents low-level memory corruption of CPython's internals; it does not prevent race conditions in your application logic. You still need `threading.Lock` (or an atomic higher-level primitive) around any shared mutable state that multiple threads modify."
Follow-up answer
"Classic case: a shared dict used as an in-memory counter or cache, incremented by multiple threads handling concurrent requests — `cache[key] = cache.get(key, 0) + 1` run by two threads near-simultaneously can lose an increment, because the get, add, and set are three separate bytecode-level steps the GIL can interleave between. Run this ten thousand times across threads without a lock and the final count will be measurably less than ten thousand. The fix is a `Lock` around the read-modify-write, or using `collections.Counter` with explicit locking, or moving the counter out of process-local memory into something like Redis with atomic `INCR`."
