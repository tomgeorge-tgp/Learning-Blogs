The Core Definition

Every HTTP request is built on layers, each solving a distinct problem, each adding latency:

LayerJobFailure mode if absent
DNSResolves a hostname (api.applygoat.com) to an IP addressThe client has no idea which machine to even connect to
TCPEstablishes a reliable, ordered, bidirectional byte stream between two IP:port pairs via a 3-way handshakePackets could arrive out of order, get duplicated, or silently drop with no retransmission
TLSEncrypts the TCP stream and authenticates the server's identity via certificateAnyone on the network path can read or tamper with every byte (credentials, tokens, payloads)
HTTPA request/response text protocol layered on top of the encrypted TCP stream — method, path, headers, bodyNo shared convention for "what am I asking for and what am I getting back"
A socket is the API for all of this A socket is the OS-level handle a program uses to send/receive bytes over the network — the abstraction that sits underneath every HTTP client and server. When Python's requests, httpx, or Uvicorn accepts a connection, at the bottom of that call stack is a socket file descriptor the OS manages. Everything above this lesson's diagram — FastAPI routing, Pydantic validation — happens only after a socket has already been read into a buffer.

Plain English — A Request Hits applyGOAT

A candidate opens their dashboard. Trace every step before their data appears:

1. DNS resolution. The browser asks a resolver "what's the IP for api.applygoat.com?" This may hit a local cache, the OS resolver, the ISP's resolver, or walk the full hierarchy (root → .com TLD → applygoat.com's authoritative nameserver) if nothing is cached. Result: an IP address, cached for the record's TTL.

2. TCP handshake. The browser's OS opens a socket and sends a SYN packet to that IP on port 443. The server replies SYN-ACK; the client replies ACK. Three packets, and now both sides have a reliable, ordered byte-stream connection — this is pure overhead paid before a single byte of your actual request goes anywhere, which is precisely why HTTP keep-alive (reusing one TCP connection for many requests) matters so much for latency.

3. TLS handshake. Layered on top of the now-open TCP connection: the client and server negotiate a cipher suite, the server presents its certificate (proving it's actually applyGOAT and not an attacker), and both sides derive a shared encryption key. Everything from here on is encrypted — this is what makes HTTP into HTTPS.

4. The actual HTTP request. Only now does the browser send bytes that look like GET /api/applications HTTP/1.1\r\nHost: api.applygoat.com\r\n.... In applyGOAT's deployment, this hits Caddy first (reverse proxy, TLS termination), which forwards the request over the internal Docker network to Uvicorn, which hands it to FastAPI/Starlette for routing and, eventually, your async def handler.

5. The response. FastAPI returns JSON; Uvicorn writes it back over the same TCP connection; Caddy relays it back to the client, still inside the TLS session. If the connection is kept alive (the HTTP default since 1.1), the next request from this same client skips steps 2 and 3 entirely — this is why connection reuse is one of the single biggest latency wins available, and why closing/reopening connections per-request is a real performance bug.

Your Stack, Mapped

ConceptWhere it lives in your stackWhy it matters
Reverse proxy / TLS terminationCaddy sits in front of Uvicorn, terminates TLS, forwards plain HTTP internally over the Docker bridge networkUvicorn never has to manage certificates; Caddy also gives you automatic Let's Encrypt renewal — a config problem, not a code problem
Connection keep-alivehttpx.AsyncClient reusing a connection pool for outbound calls (e.g. to OpenAI's API)Skipping repeated TCP+TLS handshakes on every outbound API call is one of the highest-leverage latency fixes in an AI pipeline that calls external APIs constantly
DNS cachingDocker's internal DNS resolves service names (postgres, redis) to container IPs on the compose networkContainer restarts get new internal IPs; service-name resolution is what makes that transparent instead of a hardcoded-IP outage
HTTP/2 multiplexingCaddy negotiates HTTP/2 by default when the client supports itMultiple requests share one TCP connection without head-of-line blocking at the HTTP layer — fewer handshakes under load, especially for a page loading many small API calls
Cross-project check — AI Sales Agent Streaming an LLM response token-by-token (SSE, Server-Sent Events) is a single long-lived HTTP response where the server writes chunks incrementally instead of one final payload — the TCP connection and TLS session stay open for the whole generation, and the client renders each chunk as it arrives. This is why streaming responses are sensitive to any reverse proxy sitting in between (Caddy, nginx) buffering the response instead of flushing chunks immediately — a proxy misconfigured to buffer will turn a "typing" effect into one big delayed blob, which is a networking-layer bug, not an application bug, even though it looks like one from the frontend.
Cross-project check — Mini Hermes Agent MCP (Model Context Protocol) can run over different transports — stdio (a local subprocess pipe, no networking stack involved at all) or HTTP/SSE for a remote MCP server. Recognizing which transport a given MCP server uses tells you immediately whether DNS/TCP/TLS considerations apply at all: a stdio-based local tool server has none of this lesson's latency profile, while an HTTP-based remote one inherits every layer discussed here, including the same keep-alive and streaming concerns as the AI Sales Agent callout above.

HTTP/1.1 vs HTTP/2 vs HTTP/3

VersionTransportKey change
HTTP/1.1TCPKeep-alive by default, but one request per connection must complete before the next starts (head-of-line blocking at the HTTP layer) unless the client opens multiple TCP connections
HTTP/2TCPMultiplexes many requests over a single TCP connection using binary framing — no more opening 6 parallel connections per host just to load a page faster
HTTP/3QUIC (over UDP)Moves multiplexing below HTTP entirely, onto QUIC, so one dropped packet only stalls the one stream it belongs to — HTTP/2 over TCP still stalls every multiplexed stream on a single lost TCP segment, since TCP itself guarantees strict ordering
Common mistake Treating "HTTPS is just HTTP with encryption" as the whole story undersells the cost: TLS adds a full extra round-trip of handshake latency on a fresh connection (mitigated by TLS 1.3's 1-RTT handshake and session resumption) — which is exactly why connection reuse and HTTP/2's single-connection multiplexing matter so much for perceived performance, not just for polish.

// LIKELY INTERVIEW QUESTIONS — click to reveal answers
Model answer
"First, DNS resolution turns the hostname into an IP address, walking cache layers up to the authoritative nameserver if needed. Then the client opens a TCP connection to that IP via the three-way handshake — SYN, SYN-ACK, ACK — establishing a reliable byte stream. On top of that, for HTTPS, a TLS handshake negotiates encryption and the server proves its identity via certificate. Only then does the actual HTTP request go out as plaintext-looking text inside the encrypted channel — method, path, headers, body. The server's response comes back over the same connection, which then usually stays open (keep-alive) for subsequent requests instead of repeating the handshakes."
Follow-up answer
"TLS sits between TCP and HTTP — it wraps the already-established TCP connection in an encrypted, authenticated channel before any HTTP bytes are sent. It protects against two distinct things: confidentiality, so anyone intercepting the traffic on the network path sees ciphertext, not your session token or request body; and authenticity, via the certificate chain, so the client can verify it's actually talking to the real server and not a machine-in-the-middle impersonating it. Without TLS, both are absent — HTTP alone offers no protection for either."
Model answer
"First I'd check whether clients are actually reusing TCP connections — keep-alive — or opening a fresh connection (and paying a full TCP + TLS handshake) per request. That's the single biggest lever: eliminating repeated handshakes entirely for a client making many sequential calls. I'd check the HTTP client configuration (connection pool size, keep-alive timeout) on both the outbound side, if this service calls other APIs, and confirm the reverse proxy in front (Caddy/nginx) isn't closing connections aggressively. I'd also check TLS session resumption is enabled, since that turns a full handshake into an abbreviated one for repeat clients."
Follow-up answer
"HTTP/2 helps if the client is making many requests to the same host and previously opened separate TCP connections per request to work around HTTP/1.1's one-request-in-flight-per-connection limitation — HTTP/2 multiplexes them over one connection, cutting handshake count dramatically. HTTP/3 helps further only if packet loss on the network path is itself a factor, since QUIC's per-stream independence avoids one dropped packet stalling every multiplexed stream the way TCP's strict ordering does under HTTP/2. If the bottleneck is purely 'too many fresh handshakes,' HTTP/2 alone captures most of the win; HTTP/3's advantage is specifically about lossy networks, not handshake count."
Model answer
"My first hypothesis is response buffering at the reverse proxy layer — Caddy or nginx sitting in front of Uvicorn may be configured to buffer the full response body before forwarding it to the client, which defeats the entire point of a streaming response even though the application is correctly flushing chunks as they're generated. This is a networking/infra-layer issue, not an application bug, even though the symptom shows up in the UI as 'streaming doesn't work.'"
Follow-up answer
"I'd check response headers for buffering-disable directives the proxy respects (e.g. `X-Accel-Buffering: no` for nginx), and check the proxy's own streaming/buffering config directives directly. To isolate it, I'd curl the Uvicorn instance directly, bypassing the proxy entirely, and watch whether chunks arrive incrementally — if they do, the application is fine and the proxy is the culprit; if they don't, the bug is upstream in FastAPI's response generation itself, e.g. accidentally buffering the full generator into a list before returning it."
