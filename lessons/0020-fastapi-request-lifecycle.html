The Core Definition

ASGI (Asynchronous Server Gateway Interface) is the contract between a Python web server and a Python web application, successor to WSGI, designed around async/await instead of one-thread-per-request. FastAPI is built on Starlette (ASGI toolkit) plus Pydantic (validation); Uvicorn is the ASGI server that actually owns the socket and event loop.

ASGI, precisely An ASGI application is just a callable: async def app(scope, receive, send). scope is a dict describing the connection (method, path, headers). receive is an awaitable that yields incoming message chunks (the request body, for instance). send is an awaitable you call to emit response chunks. FastAPI/Starlette build an enormous amount of convenience — routing, validation, dependency injection — entirely on top of this three-argument contract; there is no additional magic underneath it.

Plain English — Tracing applyGOAT's GET /applications

A logged-in user's dashboard requests their application list. Follow the request through every layer that's been built across this module.

Socket to scope (Lesson 19 meets Lesson 17). Uvicorn's event loop has an already-established, kept-alive TCP connection (or accepts a new one) and reads bytes into an HTTP parser, producing an ASGI scope dict — method GET, path /applications, headers including the auth token. This is the first point where "networking" becomes "Python data" — the scope is an ordinary dict, a PyObject, subject to everything Lesson 17 covered.

Routing and dependency injection. Starlette's router matches the path to your @app.get("/applications") handler. FastAPI resolves every Depends(get_current_user) and Depends(get_db_session) in the signature — each dependency is itself potentially an async generator, run before your handler body executes, and torn down (session closed) after it returns, even on exception.

The database call — where concurrency becomes real. Inside your handler, await session.execute(query) asks asyncpg (or psycopg in async mode) for a connection from a connection pool — a fixed set of already-open TCP connections to PostgreSQL, reused across requests instead of opening a fresh one per call. If a connection is available, it's checked out immediately; if the pool is exhausted, this await blocks until one frees up. Critically: while this coroutine is suspended waiting on the database's network round-trip, the event loop is free to run other requests' handlers — this is Lesson 18's asyncio model, made concrete. One worker process can have hundreds of requests "in flight," each parked at an await, because waiting costs nothing but a suspended coroutine's memory.

Response. FastAPI serializes the return value against the declared response_model (a Pydantic model — more validation, more object allocation, per Lesson 17), Starlette's middleware chain runs in reverse (logging, CORS headers), and Uvicorn writes the HTTP response back over the same socket, ready for keep-alive reuse by the next request.

The Production Questions, Answered Directly

QuestionAnswer
Can one FastAPI server support only 100 PostgreSQL sessions? No — conflating "concurrent HTTP requests" with "concurrent DB connections" is the core misunderstanding this lesson exists to fix. A connection pool of, say, 20 connections can serve far more than 20 concurrent requests, because each request only holds a connection for the brief duration of its actual query — most of a request's wall-clock time (auth checks, serialization, waiting on other awaits) holds no connection at all. Thousands of requests can be "in flight" against a pool of 20.
What happens when the pool is exhausted? New requests needing a connection await at the checkout call until one is released or a configured timeout elapses, at which point most drivers raise a timeout error. This shows up as a latency cliff under load, not a crash: p50 stays fine while p99 spikes as requests queue for a connection — the classic symptom that the pool, not the CPU, is the bottleneck.
Should every request open a new DB connection? No — opening a PostgreSQL connection is expensive (TCP handshake, TLS if required, PostgreSQL's own auth handshake, backend process fork on the server side). A pool amortizes that cost across the application's lifetime instead of paying it per-request; this is the database-layer analogue of Lesson 19's HTTP keep-alive.
How should pool size be configured? Roughly: enough to cover peak concurrent in-database-call requests, not peak concurrent HTTP requests — those are very different numbers under an async model. Oversizing wastes PostgreSQL's own connection-handling resources (each backend connection costs the database server memory); undersizing causes queuing. Start from PostgreSQL's own max_connections divided across all worker processes and services sharing that database, then tune from measured queue-wait time, not guesswork.
How many Uvicorn/Gunicorn workers? One worker process per CPU core is the standard starting point (Lesson 18: each worker is a separate process with its own GIL, so this is how you actually use multiple cores) — each worker then runs its own asyncio event loop handling many concurrent connections. More workers than cores doesn't add parallelism, only context-switching overhead and — critically — multiplies your effective connection pool count, since each worker process holds its own pool.
What happens with 10,000 concurrent users? If they're mostly idle (viewing a page, not hammering the DB), asyncio's cheap-per-connection model handles this comfortably on modest hardware — this is exactly the workload asyncio was built for. If all 10,000 are actively issuing DB-bound requests simultaneously, the actual bottleneck shifts to the database connection pool and PostgreSQL's own capacity, not to Python's concurrency model — at that point the fix is read replicas, caching (Redis), or horizontal scaling of the database tier, not more Uvicorn workers.
Cross-project check — eBider eBider's competing-bid storm is the connection-pool-exhaustion scenario made adversarial: an auction's final seconds produce a burst of concurrent bid submissions, all wanting a database connection at the same instant, and each holding it slightly longer than usual because of row-level locking contention (Lesson 18's cross-project check) on the same auction row. This is precisely where pool-size tuning and row-lock duration compound each other — a lock held a few extra milliseconds under contention can be the difference between a pool that keeps up and one that visibly queues, because every blocked request is also a connection held open longer than the happy path.
Cross-project check — AI Sales Agent A LangGraph-driven conversation turn typically does two things sequentially: an LLM call (seconds of pure waiting, I/O-bound, no DB connection needed) and a database write to persist conversation state (milliseconds, DB connection needed only for that instant). Structuring the handler to hold a DB connection for the entire request — including the multi-second LLM wait — instead of only around the actual database call is a real and common performance bug: it inflates a pool's effective load by however long the LLM call takes, for a resource the request didn't need during that wait at all.
Common mistake — the sync-call-in-async-handler trap Calling a blocking, synchronous library (a non-async DB driver, a synchronous requests.get()) inside an async def handler without await blocks the entire event loop for that duration — every other in-flight request on that worker stalls, not just this one. This is the single most common way to accidentally destroy FastAPI's concurrency benefits: one blocking call inside an async handler silently serializes the whole worker. The fix is either a true async driver (asyncpg, httpx) or explicitly running the blocking call in a thread pool via run_in_executor/anyio.to_thread.run_sync.

Model answer
"Potentially yes — 'concurrent users' and 'concurrent database connections' are different numbers under an async model. A user's request holds a pooled connection only for the actual query duration, not for the full request lifecycle; most of the request's time is spent on things that need no database connection at all — auth checks, serialization, waiting on external API calls. With short queries and an async driver, a pool of 20 can service thousands of concurrent requests as long as no single request holds a connection for long. The real question isn't the pool size in isolation, it's average connection-hold time times concurrent DB-bound requests versus pool size."
Follow-up answer
"Connection acquisition wait time — the time a request spends awaiting a checkout from the pool before it gets one, separate from query execution time itself. If that's consistently near zero, the pool is sized fine regardless of total user count. If it climbs under load, requests are queuing for connections and either the pool needs to grow, PostgreSQL needs more capacity, or queries need to get faster/shorter so connections free up sooner. I'd also watch PostgreSQL's own `pg_stat_activity` for connections stuck in a long-running or idle-in-transaction state — that's usually a code bug (holding a connection across an await it didn't need) rather than a genuine capacity problem."
Model answer
"A synchronous call inside an async handler blocks the single event-loop thread for its entire duration — not just this request, but every other in-flight request on that worker process, since there's only one thread cooperatively switching between coroutines. One slow synchronous call turns the whole worker's concurrency into serialized, one-at-a-time execution for as long as that call takes. The fix is either swapping in a genuinely async client (httpx instead of requests) or explicitly offloading the blocking call to a thread pool via `run_in_executor` so it doesn't occupy the event loop thread."
Follow-up answer
"The signature is latency correlation across unrelated requests — if request A's latency spikes correlate suspiciously with the presence of unrelated request B in the same time window, that points at shared event-loop blocking rather than each request's own independent cost. Concretely: p99 latency on fast, simple endpoints climbing in step with traffic to a totally different, slow endpoint on the same worker is the tell. Event loop lag monitoring (measuring how late a trivial scheduled callback actually runs) is the more direct instrument, if the framework/APM supports it."
Model answer
"More workers only helps if the current bottleneck is CPU or event-loop contention within existing workers — each new worker is a new process with its own GIL and its own connection pool. If the actual bottleneck is downstream — PostgreSQL itself is saturated, or an external API is slow — adding workers just means more processes queuing on the same downstream constraint, and can even make it worse: each new worker adds its own connection pool, multiplying total connections against a database that was already the limiting factor. Workers scale compute and I/O-multiplexing capacity; they don't scale a shared downstream resource."
Follow-up answer
"I'd profile where time is actually spent per request before touching worker count: is it CPU-bound work inside the handler (more workers/processes genuinely help), time waiting on the database (check pool exhaustion and query latency first — more workers can worsen this), or time waiting on an external API (that's a rate limit or third-party latency problem no amount of local scaling fixes). Only after confirming the bottleneck is 'this worker process itself is saturated with CPU or event-loop-bound work' would I add workers with confidence it'll help."
